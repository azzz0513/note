### 进程
进程的两个基本属性：
- 资源分配的基本单位
- 调度的基本单位
#### 进程模型
在进程模型中，计算机上所有可运行的软件，通常也包括操作系统，被组织成若干**顺序进程**（sequential process），简称**进程**（process）。一个进程就是一个正在执行程序的实例，包括程序计数器、寄存器和变量的当前值。从概念上讲，每个进程拥有它自己的虚拟CPU。但是实际上，真正的CPU是在各进程之间来回切换。

A program can be the execution program of multiple process; also, a process can call multiple programs.

#### 进程创建
4种主要事件会导致进程的创建：
- 系统初始化
	- 前台进程：同用户交互并且替它们完成工作的那些进程
	- 后台进程：这些进程与特定的用户无关，具有某些专门的功能
	- 守护进程（daemon）：停留在后台处理诸如电子邮件、Web页面、新闻、打印之类活动的进程
- 正在运行的程序执行了创建进程的系统调用
- 用户请求创建一个新进程
- 一个批处理作业的初始化
![[images/Pasted image 20250611165627.png]]
In UNIX, a fork() system call is used to create a new process.
- Initially, the parent and the child have the same memory image, the same environment strings, and the same open files.
- The execve() system call can be used to load a new program.
- But the parent and child have their own distinct address space.

#### 进程终止
进程在创建之后，它开始运行，完成其工作。但是迟早这个进程会终止，通常由下列条件引起：
- 正常退出（自愿的（voluntary））
- 出错退出（自愿的）
- 严重错误（非自愿（involuntary））
- 被其他进程杀死（非自愿）


#### 进程的层次（process hierarchies）
当进程创建了另一个子进程后，父进程和子进程就以某种形式继续保持关联。子进程自己可以创建更多的进程，组成一个进程的层次。

在UNIX中，进程和它的所有子进程以及后裔组成一个**进程组**（process group）

在Windows中没有进程层次这一概念，所有的进程都是地位相同的。唯一类似于进程层次的是在创建进程的时候，父进程得到一个特别的令牌（句柄），该句柄可以用来控制子进程。但是，它有权讲这个句柄传送给其他进程，这样就不存在进程层次了。

#### 进程的状态
进程有三种基础的状态：
- 运行态（running）：该时刻进程实际占用CPU
- 就绪态（ready）：可运行，但因为其他进程正在运行而暂时停止
- 阻塞态（blocked）：除非某种外部事件发生，否则进程不来了运行

### 线程（thread）
将进程的两个基本属性分开，由操作系统分开处理：
- 对于作为调度和分派的基本单位，不同时作为拥有资源的单位，以做到“轻装上阵”
- 对于拥有资源的基本单位，又不对之进行频繁的切换

thread：轻量化进程（lightweight process（LWP））
- Thread：CPU利用的基本单位
- Process：资源分配的基本单位

Thread is a sequential execution stream within a process.
Threads separate the notion of execution from the process abstraction.
![[images/Pasted image 20250611174850.png]]

原进程PCB的内容分成两部分：
- 描述进程资源和空间的部分
- 描述执行现场、状态及调度的部分（将这部分内容作为线程控制块TCB的内容，且一个进程内允许多个线程存在）

新进程描述为：
- 一个独立的进程空间，可装入进程映像
- 一个独立的进程相关联的执行文件
- 线程所用的系统资源
- 一个或多个线程

线程不拥有系统资源，只有其运行所必须的一些数据结构：TCB、程序计数器、寄存器组、堆栈。它与该进程中其他线程共享该进程所拥有的全部资源。
- 堆栈和寄存器用来存储线程内的局部变量
- TCB用来说明线程存在的标识、记录线程属性和调度信息

If the process has multiple threads of control, it can do more than one task at a time. This situation is called **multithreading**.

- 进程是资源分配的基本单位， 所有与该进程有关的资源分配情况， 如打印机、 I/O缓冲队列等， 均记录在进程控制块PCB中， 进程也是分配主存的基本单位， 它拥有一个完整的虚拟地址空间。 而线程与资源分配无关， 它属于某一个进程， 并与该进程内的其它线程一起共享进程的资源。
- 不同的进程拥有不同的虚拟地址空间， 而同一进程中的多个线程共享同一地址空间。
- 进程调度的切换将涉及到有关资源指针的保存及进程地址空间的转换等问题。 而线程的切换将不涉及资源指针的保存和地址空间的变化。 所以， 线程切换的开销要比进程切换的开销小得多。
- 进程的调度与切换都是由操作系统内核完成， 而线程则既可由操作系统内核完成， 也可由用户程序进行。
- 进程可以动态创建进程。 被进程创建的线程也可以创建其它线程。
- 进程有创建、 执行、 消亡的生命周期。 线程也有类似的生命周期。

#### 线程的实现
三种形式：
- 在用户空间中实现线程
- 在内核中实现线程
- 混合实现

##### 用户级线程（User Threads（ULT））
The threads package entirely in user space, kernel doesn’t know thread.
Threads management done by user-level threads library.
Thread switch don’t need kernel privilege, user-level threads are fast to create and manage.
Problem: If the kernel is single-threaded, then any userlevel thread performing a blocking system call will cause the entire process to block.

![[images/Pasted image 20250611181429.png]]
在用户空间管理线程时，每个进程需要有其专用的线程表（thread table），用来跟踪该进程中的线程。这些表和进程表类似，不过它仅仅记录各个线程的属性，如各个线程的PC、堆栈指针、寄存器和状态等。当一个线程转换到就绪状态或阻塞状态时，在该线程表中存放重新启动该线程所需的信息，与内核在进程表中存放进程的信息一样。

用户级线程允许每个进程有自己定制的调度算法。

用户级线程的问题：如果一个线程开始运行，那么在该进程中的其他进程就不能运行，除非第一个线程自动放弃CPU。在一个单独的进程内部，没有时钟中断，所以不可能用轮换调度的方式调度线程。
##### 内核级线程（Kernel Threads（KLT））
Supported by the Kernel: The kernel performs thread creation, scheduling, and management in kernel space.
No thread library, kernel provide API.
Kernel maintain context of process and thread.
Thread switch need kernel.
Thread is the basic unit of Scheduler .
Disadvantage: high cost

![[images/Pasted image 20250611182103.png]]
内核中的线程表保存了每个线程的寄存器、状态和其他信息。这些信息和在用户空间中的线程是一样的，但是现在保存在内核中。

所有能够阻塞线程的调用都以系统调用的形式实现，这与用户级线程相比，代价是非常可观的。当一个线程阻塞时，内核根据其选择，可以运行同一个进程中的另一个线程或者运行另一个进程中的线程。而在用户级线程中，运行时系统时钟运行自己进程中的线程，直到内核剥夺它的CPU（或没有可运行的线程存在）为止。

##### 混合实现
![[images/Pasted image 20250611182902.png]]
内核只识别内核级线程，并且对其进行调度。其中一些内核级线程会被多个用户级线程多路复用。如同在没有多线程能力操作系统中某个进程中的用户级线程一样，可以创建、撤销和调度这些用户级线程。在这种模型中，每个内核级线程有一个可以轮流使用的用户级线程集合。

##### 弹出式线程（Pop-Up Threads）
![[images/Pasted image 20250611183613.png]]
在消息到达时创建一个新的线程：（a）消息到达之前（b）消息到达之后

弹出式线程的好处是：由于这种线程相当新，没有历史——没有必须存储的寄存器、堆栈诸如此类的内容，每个线程从全新开始，每一个线程彼此之间都完全一样。这样，就有可能快速创建这类线程。对该线程指定要处理的消息。使用弹出式线程的结果是：消息到达与处理开始之间的时间非常短。


### 进程间通信（Inter-process Communication）
进程间通信（IPC）涉及三个问题：
- 一个进程如何将信息传递给另一个进程
- 共享资源
	- 如何确保在参与关键活动时，两个或多个进程之间不会出现交叉
- 过程协同
	- 当存在依赖关系时，进行适当的排序

**进程同步**：对多个相关进程在执行次序上的协调，用于保证这种关系的相应机制称为进程同步。 （或相互合作的一组并发进程在一些关键点上可能需要互相等待与互通消息，这种相互制约的等待与互通消息称为进程同步。）
例：进程A产生数据而进程B打印数据，那么B在打印之前必须等待，直到A已经产生一些数据。

#### 竞争条件（Race condition）
**竞争条件**：即两个或多个进程读写某些共享数据，而最后的结果取决于进程运行的顺序的情况。
避免竞争条件的关键是禁止多个进程同时读写共享数据。

#### 临界区
**互斥**（mutual exclusion）：即以某种手段确保当一个进程在使用一个共享变量或文件时，其他进程不能做同样的操作。
**临界区域**（critical region）或**临界区**（critical section）：对共享资源进行访问的程序片段。
**临界资源**（critical resource）：一次仅允许一个进程访问的资源

我们可以通过适当地安排，使得两个进程不可能同时处于临界区中，从而避免竞争条件。

对于一个好的提供互斥的方案，应该包括下面4各条件：
- 任何两个进程不能同时处于其临界区
- 不应对CPU的速度和数量做任何假设
- 临界区外运行的进程不得阻塞其他进程
- 不得使进程无限期等待进入临界区

#### 互斥
##### 屏蔽中断（Disabling Interrupt）
使每个进程在刚刚进入临界区后立即屏蔽所有中断，并在就要离开之前再打开中断。屏蔽中断后，时钟中断也会被屏蔽。CPU只有发生时钟中断或其他中断才会进行进程切换，这样，在屏蔽中断之后CPU将不会被切换到其他进程。从而实现互斥。

问题：
- 一个进程屏蔽中断后不再打开：导致整个系统因此终止
- 如果系统是多处理器：屏蔽中断仅对执行disable指令的那个CPU有效。其他CPU仍将继续运行，并可以访问共享内存。

##### 锁变量（Lock Variable）
设想有一个共享（锁）变量，其初始值为0。当一个进程想进入临界区时，它首先检查这把锁，如果锁的值为0，这该进程将其设置为1并进入临界区；如果锁的值为1，则该进程将等待直到其值变为0。

问题：
假设一个进程读出锁变量的值并发现它的值为0，而恰好在它将其值设置为1之前，另一个进程被调度运行，将该锁设置为1。当第一个进程再次运行时，它同样会把该锁设置为1，则此时同时又两个进程进入临界区中。

##### 睡眠与唤醒（Sleep and Wakeup）

###### 生产者-消费者问题（Producer-Consumer Problem）
也称为**有界缓冲区**（bounded-buffer）问题
两个进程 共享一个公共的固定大小的缓冲区。其中一个是生产者，将信息放入缓冲区；另一个是消费者，从缓冲区中取出信息。
问题在于当缓冲区已满，而此时生产者还想往其中放入一个新的数据项的情况。其解决方法是让生产者睡眠，待消费者从缓冲区中取出一个或多个数据项时再唤醒它。同样地，当消费者试图从缓冲区中取数据而发现缓冲区为空时，消费者就会睡眠，直到生产者向其中放入一些数据时再将其唤醒。
![[images/Pasted image 20250611212750.png]]
这里可能出现竞争条件，其原因是对count的访问未加限制。有可能出现下面这些情况：
缓冲区为空，消费者刚刚读取count的值发现它为0。此时调度程序决定暂停消费者并启动运行生产者。生产者向缓冲区中加入一个数据项，count加1。现在count的值为1了。它推断认为由于刚才count为0，所以消费者此时一定在睡眠，所以生产者调用wakeup来唤醒消费者。但是消费者此时并未睡眠，所以wakeup信号丢失。当消费者下次运行时，它将测试先前读到的count值，发现其值为0，然后进入睡眠。生产者迟早会填满整个缓冲区，然后睡眠。这样一来，两个进程都会永远睡眠下去。
解决方案：可以加上一个**唤醒等待位**。当一个wakeup信号发送给一个清醒的进程信号时，将该位置设为1。随后，当该进程要睡眠时，如果唤醒等待位为1，则将该位清除，而该进程仍然保持清醒。

##### 信号量（Semaphores）
引入一个新的变量类型，称为**信号量**。一个信号量的取值可以为0（表示没有保存下来的唤醒操作）或者为正值（表示有一个或多个唤醒操作）。
设立两种操作：down和up
- 对一个信号量执行down操作，则是检查其值是否大于0。若该值大于0，则将其值减1（用掉一个保存的唤醒信号）并继续；若该值为0，则进程将睡眠，而且此时down操作并未结束。检查数值、修改变量值以及可能发生的睡眠操作均作为一个单一的、不可分割的**原子操作**完成。
- up操作对信号量的值增1。如果一个或多个进程在该信号量上睡眠，无法完成一个先前的down操作，则由系统选择其中的一个并允许该进程完成它的down操作。于是，对一个有进程在其上睡眠的信号量执行一次up操作后，该信号量的值仍旧为0，但在其上睡眠的进程却少了一个。

- down操作用于申请资源并且减少count
- up操作用于释放资源并增加count
![[images/Pasted image 20250612142541.png]]
执行一次P操作后，若S.count<0，则|S.count|等于Q队列中等待S资源的进程数。

![[images/Pasted image 20250612142612.png]]
执行一次V操作后，若S.count≤0，则表明Q队列中仍有因等待S资源而被阻塞的进程，所以需要唤醒(wakeup)Q队列中的进程。
![[images/Pasted image 20250612143509.png]]
当上面两步调换之后，会出现死锁问题。假设现在缓冲区已满，即empty=0，full=N，mutex=1。现在生产者先执行down(&mutex)，然后执行down(&empty)，因为empty=0，所以生产者进程被阻塞。然后系统调度消费者执行，消费者先执行down(&mutex)，发现mutex=0，也被阻塞。两个进程都在等待对方释放资源并唤醒自己，所以导致死锁问题。
但是假如消费者中的down(&full)和down(&mutex)调换则不会导致死锁问题。

- 信号量mutex用于互斥，它用于保证任一时刻至于哦一个进程读写缓冲区和相关的变量。
- 信号量empty和full用于实现**同步**（synchronization）。信号量full和empty用来保证某种事件的顺序发生和不发生。
确保down(&mutex)在临界资源访问之前调用，up(&mutex)在临界资源访问之后调用。

##### PV操作题目的解题思路
1. 关系分析：找出题目中描述的过程，分析他们之间的同步、互斥关系
2. 设置信号量：设置需要的信号量，并根据题目条件确定信号量初值（互斥信号量初值一般为1，同步信号量初值要看对应资源的初始值为多少）

#### 管程（monitor）
A **monitor** is a collection of procedures, variables, and data structures that are all grouped together in a special kind of module or package.
![[images/Pasted image 20250612152626.png]]

- A process/thread enters the monitor by invoking one of its procedures.
- Only one process/thread can be active within the monitor at a time.(mutual exclusion).
	- Any other process that has invoked the monitor is suspended, waiting for the monitor to become available.
- No process can directly access a monitor’s local variables.
- A monitor may only access its local variables.

任一时刻管程中只能有一个活跃进程，这一特性使管程能有效地完成互斥。
进入管程时的互斥由编译器负责，但通常的做法是用一个互斥量或二元信号量。

**条件变量**（condition variables）以及相关的两个操作：wait和signal。当一个管程发现它无法继续运行时，它会在某个条件变量上执行wait操作。该操作导致调用进程自身阻塞，并且还将另一个以前等在管程之外的进程调入管程。
- The operation wait(x) means that the process invoking this operation is suspended until another process invokes signal(x).
- The signal(x) operation resumes exactly one suspended process. If no process is suspended, then the signal operation has no effect.

### 调度（Scheduling）
当计算机系统是多道程序设计系统时，通常就会有多个进程或线程同时竞争CPU。只要有两个或更多的进程处于就绪状态，这种情形就会发生。如果只有一个CPU可用，那么就必须选择下一个要运行的进程。在操作系统中，完成选择工作的这一部分称为**调度程序**（scheduler），该程序使用的算法称为**调度算法**（scheduling algorithm）。

#### 进程行为
进程通常由下面两部分组成：
- CPU bursts：进程需要CPU的时间周期称为 CPU bursts
- I/O bursts：进程需要I/O的时间周期称为I/O bursts
![[images/Pasted image 20250612163723.png]]

- CPU-bound Process（CPU密集型进程）
	- 长CPU突发，I/O等待不频繁
- I/O-bound Process（I/O密集型进程）
	- 短CPU突发，I/O等待频繁
随着CPU的发展，进程往往更多受I/O限制。
![[images/Pasted image 20250612164117.png]]
（a）CPU密集型进程
（b）I/O密集型进程

#### 何时调度
- 一个新进程被创建。需要决定是运行父进程还是子进程。
- 在一个进程退出时。一个进程不再运行，所以必须从就绪进程中选择另外的进程。
- 当一个进程由于某些原因阻塞时。
- 在一个I/O中断发生时。如果中断来自I/O设备，而该设备完成了工作，某些被阻塞的等待该I/O的进程就成为了可运行的就绪进程了。
- 时钟中断。可以在每个时钟中断或者在每k个时钟中断时做出调度决策。
	- **非抢占式**调度算法挑选一个进程，然后让该进程运行直至被阻塞，或者直到该进程主动释放CPU。这样的结果是，在时钟中断发生时，不会进行调度。
	- **抢占式**调度算法挑选一个进程，并且让该进程运行某个固定时段的最大值。如果在该时段结束时，该进程仍在运行，则它被挂起，而调度程序挑选另一个进程运行（如果有就绪进程）。进行抢占式调度，需要在时间间隔的末端发生时钟中断，以便把CPU控制返回给调度程序。

#### 调度算法分类
- 批处理（Batch System）
	- 非抢占式算法
	- 每个进程都有长时间周期的抢占式算法
- 交互式（Interactive System）
	- 抢占式是必需的
- 实时（Real-Time System）
	- 抢占有时是不需要的

#### 调度算法的目标
- Fair (nobody cries)
- Priority (lady first)
- Efficiency (make best use of equipment)
- Encourage good behavior (good boy/girl)
- Support heavy loads (degrade gracefully)
- Adapt to different environments (interactive, realtime…)

对于所有系统：
- Fairness: No process should suffer starvation
- Efficiency: Keep resources as busy as possible
- Policy Enforcement: Seeing that stated policy is carried out

对于批处理系统：
- Throughput: Number of processes that completes in unit time
- Turnaround Time (also called elapse time): Amount of time to execute a particular process from the time its entered
- Waiting Time: amount of time process has been waiting in ready queue
- Processor Utilization: Percent of time CPU is busy

对于交互式系统：
- Response Time: amount of time from when a request was first submitted until first response is produced.
- Proportionality: meet users' expectation

对于实时系统：
- Meeting Deadlines: avoid losing data
- Predictability: Same time/cost regardless of load on the system

#### 批处理系统
##### 先来先服务（FCFS）
也称为FIFO。使用该算法，进程按照它们请求CPU的顺序使用CPU。基本上，有一个就绪进程的单一队列。

该算法是非抢占式的：进程不会因为运行太长时间而被中断。当其他作业进入时，它们就排到就绪队列的尾部。当正在运行的进程被阻塞时，就绪队列中的第一个进程接着运行。当在被阻塞的进程变为就绪时，就像一个新来的作业一样，排到就绪队列的尾部。

性能指标：平均等待时间
![[images/Pasted image 20250612171334.png]]
![[images/Pasted image 20250612171423.png]]

问题：
- 无法并行使用资源
	- 假设有1个CPU密集型进程和了n-1个 I/O密集型进程：I/O密集型进程可以快速地通过就绪队列并暂停自己以等待I/O。CPU密集型进程到达队列头部并完成自己的作业。I/O密集型进程再次加入就绪队列，必须等待CPU密集型进程完成作业。在这段时间中，I/O设备处于空闲状态，直到CPU密集型进程完成作业。

##### 最短剩余时间优先（Shortest Remaining Time First）
是最短作业优先的抢占式版本。
总是选择剩余运行时间最短的进程运行。
要求：
- 同样要求提前知道所有进程的运行时间

当一个新的作业到达时，其整个时间同当前进程的剩余时间做比较。如果新的进程比当前运行进程的需要更少时间，当前进程被挂起，而运行新的进程。

#### 交互式系统
交互式系统的调度算法通常是抢占式的，将时间划分为**时间片**（quantum）
调度算法通常在每个时间片开始时进行选择。
每个进程被分配一个时间片，允许进程在该时间段内运行。如果在时间片结束时，该进程还在运行，则将剥夺CPU并分配给另一个进程。如果该进程在时间片结束之前阻塞或结束，则CPU立即进行切换。
##### 轮转调度（round-robin（RR））
以轮转的方式从就绪队列中选择进程。
轮转调度很容易实现，调度程序要做的只是维护一张可运行进程列表，当一个进程用完其时间片后，就被移到队列的末尾。
![[images/Pasted image 20250612174214.png]]

问题：
- 时间片太大：
	- 变为FCFS
	- 响应时间慢
- 时间片太小：
	- 太多上下文切换
	- 浪费CPU资源

##### 优先级调度（Priority Scheduling）
每个进程被赋予一个优先级，允许优先级最高的可运行进程先运行。
在每个优先级中使用FCFS
可能无法提供最佳的AWT，并且可能无限期地阻塞或饥饿

设置优先级：
- 静态：适用于常规的系统
- 动态：优先级可能基于：
	- 用户成本
	- 用户的重要性
	- 进程类型
	- 资源需求

![[images/Pasted image 20250612174920.png]]

##### 多级队列（Multi-Queue Scheduling）
轮转调度与优先级调度的混合。
根据进程类型，CPU使用时间，IO访问，内存大小等属性将进程划分为组。
将就绪队列也拆分为几个队列，将进程分配给其中一个组。
每个队列有自己的调度算法：
- 交互式进程：RR
- 后台进程：FCFS/SRTF

队列之间的调度：
- 固定的优先级（可能导致饥饿）
- 队列的CPU使用率

![[images/Pasted image 20250612175410.png]]

##### 多级反馈（Multi-level Feedback Algorithm（MLFQ））
具有优先级的多级队列。
进程可以在队列之间移动
- 每个进程都在高优先级的队列中开始；当其结束CPU突发后，将其转移到低优先级队列

每个队列代表CPU使用率相近的作业。
给定队列中的作业以给定的时间片运行。

**CTSS**：Queuei拥有时间片$t2^i$
- 如果一个Queuei中的一个作业使用完分配给它的时间片后，将其转移到Queuei+1
- 最低优先级的队列使用FCFS

##### 保证调度（Guaranteed Scheduling（QoS））
向用户作出明确的性能保证，然后去实现它。
例：
在n个进程运行的情况下，调度器确保每个进程获得1/n的CPU周期。
为了实现所做的保证，系统必须跟踪各个进程自创建依赖已使用了多少CPU时间。然后计算各个进程获得的时间，即自创建以来的时间除以n。即计算实际消耗的CPU时间与应获得的CPU时间之比。然后选择比率最低的进程，知道该进程的比率超过它的最近接近者为止。

##### 彩票调度
- 是基于概念的：给进程彩票。在调度时间，随机选择一个彩票，持有该彩票的进程获得资源。
- 为该优先级的进程分配更多彩票（类似于优先级调度），或给短作业分配更多彩票（类似于SJF）
