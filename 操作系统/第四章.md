### 文件
长期存储信息有三个基本要求：
- 能够存储大量信息
- 使用信息的进程终止时，信息仍然存在
- 必须能使多个进程并发访问有关信息

解决方法：将信息存储在磁盘上或存储在一个叫做**文件**的内部单元中。

文件系统是操作系统管理文件的一部分。

#### 文件系统的基本功能
- 呈现文件和目录的逻辑（抽象）视图
	- 隐藏硬件设备的复杂性
- 促进存储设备的高效使用
	- 优化访问
- 支持共享
	- 提供保护

#### 文件命名
在进程创建文件时，它给文件命名。在进程终止后，该文件仍然存在，并且其他进程可以通过这个文件名对它进行访问。

文件名：`文件名.文件拓展名`
有效名称：
- 字符数：
	- 字母、数字、特殊字符的字符串
	- 最多255个字符
- 小写或大写
文件拓展名：
- 绑定到文件类型
- 由应用程序使用

#### 文件结构
- 文件是一种无结构的字节序列，事实上操作系统也不知道不关心文件内容是什么，操作系统所见到的是字节，其文件内容的任何含义只在用户程序中解释。UNIX和Windows使用这种方法。
- 文件是具有固定长度记录的序列，每个记录都有其内部结构。把文件作为记录序列的中心思想是：读操作返回一个记录，而写操作重写或追加一个记录。
- 文件由一棵记录树构成，每个记录不必具有相同的长度，记录的固定位置上有一个**键**字段。这棵树按“键”字段进行排序，从而可以对特定“键”进行快速查找。

#### 文件访问
- **顺序访问**（sequential access）：
	- 从头读取所有字节/记录
	- 不能跳过某一些内容，也不能不按顺序读取
	- 顺序访问文件是可以返回到起点的，需要时可以多次读取该文件。
	- 在存储介质是磁带而不是磁盘时，顺序访问文件是很方便的
- **随机访问**（random access）：
	- 以任何顺序读取的字节/记录
	- 对数据库系统至关重要
	- 有两种方法可以指示从何处开始读取文件：
		- 一种是每次read操作都给出开始读文件的位置。
		- 另一种是使用一个特殊的seek操作设置当前位置，在seek操作后，从这个当前位置顺序地开始读文件。

文件都有文件名和数据。另外，所有的操作系统还会保存其他与文件相关的信息，如文件创建的日期和时间、文件大小等。这些附加信息称为文件属性。

#### 文件操作
“字节序列”文件的操作
- create：创建不包含任何数据的文件。
- delete：当不再需要某个文件时，必须删除该文件以释放磁盘空间。
- open：在使用文件之前，必须打开文件。open调用的目的是：把文件属性和磁盘地址表装入内存，便于后续调用的快速访问。
- close：访问结束后，不再需要文件属性和磁盘地址，这时应该关闭文件以释放内部表空间。
- read：在文件中读取数据块。一般的，读取的数据来自文件的当前位置。调用者必须指明需要读取多少数据，并且提供存放这些数据的缓冲区。
- write：向文件中读取数据。一般的，写操作也是从文件当前位置开始的。如果当前位置是文件末尾，文件长度增加。如果当前位置为文件中部，则现有数据被覆盖，并且永远丢失。
- append：是write的限制形式。它只能在文件末尾添加数据。
- seek：对于随机访问文件，要指定从何处开始获取数据，通常的方法是用seek系统调用把当前位置指针指向文件中指定位置。seek调用结束后，就可以从该位置开始读写数据了。
- get attributes：进程运行常需要读取文件属性。
- set attributes：某些属性是可由用户设置的，甚至是在文件创建之后。
- rename：修改文件名。


### 目录（directory）
文件系统通常提供目录或文件夹用于记录文件的位置，在很多系统中目录本身也是文件。

- 单层目录（a single-level directory）：有一个包含所有文件的目录（根目录）
- 二级目录（a two-level directory）：包含根目录和用户目录
- 分层目录（a hierarchical directory）：有一个根目录很任意数量的子目录

### 文件系统的实现
#### 文件系统布局
文件系统存放在磁盘上。多数磁盘划分为一个或多个分区，每个分区中有一个独立的文件系统。磁盘的0号扇区称为**主引导记录**（Master Boot Record，MBR），用来引导计算机。
在MBR的结尾是分区表。该表给出了每个分区的起始和结束地址。表中的一个分区被标记未活动分区。
在计算机被引导时，BIOS读入并执行MBR。MBR做的第一件事是确定活动分区，读入它的第一个块，称为**引导块**（boot block），并执行之。引导块中的程序将装载该分区中的操作系统。为了统一起见，每个分区都从一个引导块开始，即使它不含有一个可启动的操作系统。
![[images/Pasted image 20250615141201.png]]

#### 文件字节与磁盘扇区
- 文件是字节序列
	- 文件I/O的单位是字节
- 磁盘是扇区阵列（512字节）
	- 磁盘I/O的单位是扇区
	- 文件数据必须存储在扇区中
- 文件系统定义块大小
	- $块大小 = 2^n*扇区大小$
	- 连续的扇区被分配到一个块中
- 文件系统将磁盘视为块阵列
	- 必须将块分配给文件
	- 必须管理磁盘上的可用空间

#### 文件实现
文件存储实现的关键问题是记录各个文件分别用到哪些磁盘块。

##### 连续分配（Contiguous Allocation）
将每个文件存储为连续的数据块
![[images/Pasted image 20250615141832.png]]
（a）为7个文件连续分配空间
（b）删除文件D和F后的磁盘状态
![[images/Pasted image 20250615141950.png]]

优势：
- 易于实现（只需要起始扇区和文件长度）
- 读取性能极佳（适用于顺序访问和直接访问）
劣势：
- 外部碎片
- 创建文件时必须知道最大文件大小，否则文件无法增长

适用于CD-ROM，DVD和其他一次写入光学介质
- 所有文件大小都是预先知道的
- 文件永远不会被删除

##### 链表分配（Linked List Allocation）
将文件存储为磁盘块的链表，块可能分散在磁盘上的任何地方。
每个块的第一个字用作指向下一个字的指针。
![[images/Pasted image 20250615142456.png]]
![[images/Pasted image 20250615142506.png]]
优势：
- 无外部碎片
- 目录条目简单（只需要起始地址）
- 只要有空闲块可用，文件就可以继续增长
- 适合顺序访问
劣势：
- 随机访问速度慢
- 块中的数据不是2的幂

##### 采用内存中的表进行链表分配（Linked List Allocation Using FAT）
从每个块中提取表指针字，并将其换入一个索引表——**文件分配表**（FAT）中。
在每个分区的开头留出一段磁盘来包含FAT。
FAT应该保存在内存中，以减少磁盘查找。
- 空间开销可能很大

**FAT**（File Allocation Table）
- 磁盘上每个块一个表项，按块号排序
- 每个表项都包含下一个块的地址
- 文件结束标记（-1）
- 特殊值（-2）表示该块是空闲的
- 用于OS/2和MS-DOS
![[images/Pasted image 20250615143519.png]]
![[images/Pasted image 20250615143527.png]]
优势：
- 每个块都可用于数据
- 随机访问
	- 搜索链表（但全部在内存中）
- 表项只需要一个数字
劣势：
- 整个表必须位于内存中
	- 例子：磁盘大小=200GB，块大小=1KB，FAT表项大小=4字节，需要800MB的内存大小用于存储FAT


##### i节点（Indexed Allocation（i-Node））
将所有指针放在一个位置（索引块或索引节点）。
每个文件都有一个称为**i节点**（index-node）的数据结构，其中列出了文件的属性和文件块的磁盘地址。
![[images/Pasted image 20250615144207.png]]
仅当打开相应的文件时，才需要将i-node加载到内存中。
每个目录表项仅包含i-number
![[images/Pasted image 20250615144309.png]]

i节点的一个问题是，如果每个i节点只能存储固定数量的磁盘地址，那么当一个文件所含的磁盘块数目超出了i节点所能容纳的数目怎么办？
一个解决方法是：最后一个“磁盘地址”不指向数据块，而是指向一个包含额外磁盘块地址的块的地址。如果有两个或更多个包含磁盘地址的块，或者指向其他存放地址的磁盘块的磁盘块。

**多级指数化分配**（Multi-level Indexed Allocation）：
- 单个间接块包含指向数据块的指针
- 双间接块包含指向单间接块的指针
- 三重间接块包含指向双间接块的指针
![[images/Pasted image 20250615145210.png]]

优势：
- 快速查找和随机访问
- 不受外部碎片的影响
- 只有当相应的文件打开时，i节点才需要在内存中
缺点：
- 索引的空间开销

目录的存储方式和普通文件相同

- 目录文件是目录项的列表
打开文件时，文件系统使用路径名来定位目录项
目录项提供查找磁盘块所需的信息。
- 整个文件的磁盘地址（连续块）
- 第一块的编号（链表）
- i节点的编号（i节点）

文件属性的位置
- 在目录项中
	- 固定大小的目录项
	- 在目录项中有磁盘地址和属性
![[images/Pasted image 20250615145753.png]]

文件属性的位置
- 在单独的数据结构中（例如i节点）
	- 目录项包含文件名和i节点号
	- i节点包含文件属性
![[images/Pasted image 20250615145933.png]]

处理目录中的长文件名：
解决方案1：固定长度名称
- 通常为255个字符
- 简单的
- 浪费空间

解决方案2：每个目录项有一个固定部分，这个固定部分通常以目录项的长度开始，后面是固定格式的数据，通常包括所有者、创建时间、保护信息以及其他属性。这个固定长度的头后面是一个任意长度的实际文件名。
- 节省空间
- 删除文件时，会引入可变大小的间隙，但是下一个进来的文件不一定正好适合这个空隙
- 单个目录项可能跨越多个页面，因此在读取文件名时可能会出现页面错误

解决方案3：使目录项的长度固定，而将文件名放置在目录后面的堆中。
- 易于移走文件目录项，不需要压实
- 文件名后不需要填充字符
- 必须管理堆
- 处理文件名时仍可能出现缺页中断
![[images/Pasted image 20250615150930.png]]

在每个目录搜索文件：
- 线性查找
- 哈希表查找
- 缓存查找结果

#### 共享文件（shared file）
共享文件用于允许文件出现在多个目录中
- 目录和共享文件之间的连接称为链接。文件系统本身是一个**有向无环图**（Directed Acyclic Graph，DAG）而不是树。
- 问题：
	- 如果目录包含磁盘地址，则必须将磁盘地址复制到B目录中
	- 如果B或C续写文件，新的数据块只列入进行添加工作的用户的目录中。其他的用户对此是不知道的。

![[images/Pasted image 20250615151551.png]]

解决方案1：磁盘块不列入目录，而是列入一个与文件本身关联的小型数据结构中。目录将指向这个小型数据结构。i节点→硬链接。
- 两个目录都指向同一个i节点
![[images/Pasted image 20250615152356.png]]

解决方案2：创建一个类型为LINK的新文件，并把该文件放在B的目录下，使得B与C的一个文件存在链接。新的文件中只包含了它所链接的文件的路径名。当B读该链接文件时，操作系统查看到要读的文件是LINK类型，则找到该文件所链接的文件的名字，并且去读该文件。这一方法称为**符号链接**（Symbolic Link，软链接）。
- 一个目录指向文件文件的i节点
- 其他目录包含“路径”
![[images/Pasted image 20250615152416.png]]

删除文件：
- 目录项已从目录中删除
- 文件中的所有块都返回到空闲列表
- 多个链接指向同一个文件

硬链接：
- 每个i节点放置一个“引用计数”字段
- 统计指向该文件的目录数量
- 从目录中删除文件时，递减计数
- 当计数为0，回收文件中的所有块
![[images/Pasted image 20250615153244.png]]

软链接（只有所有者目录有指向i节点的指针）：
- 删除真实文件时，软链接变为“断开”
- 删除软链接不会影响文件

硬链接的局限性
- 无法跨文件系统建立链接
- 如果其中一个文件被移动到其他文件系统，则会将其复制，并相应地调整这两个文件的链接计数
- 只有超级用户可以创建指向目录的硬链接

软链接的问题：
- 磁盘上有额外的空间和额外的i节点来存储链接文件
- 穿越路径中的开销
- 如果原始文件被移动到其他位置，则无法再通过符号链接访问它
- 在将文件转储到磁带上时，可以将文件的多个副本设置为复制


### 磁盘系统管理
存储n字节文件的策略：
- 分配n个连续字节的磁盘空间——段
- 分配n/k个大小为k字节的块——分页

连续分配问题：如果文件增长，则必须在磁盘上移动，这是一项昂贵的操作，并会导致外部碎片。
几乎所有文件系统都将文件分割为固定大小的块，这些块不需要相邻。

#### 块大小（block size）
文件系统定义块大小：$块大小 = 2^n*扇区大小$
- 连续的扇区被分配到一个块

文件系统将磁盘看作块阵列
- 必须将块分配给文件
- 必须管理磁盘上的可用空间

大的块尺寸：
- 内部碎片：最后一个块（平均）浪费1/2得到空间
- 多小文件：浪费更多
- 磁盘利用率较低，但是性能更好

小的块尺寸：
- 更多的查找：文件访问会变慢
- 磁盘利用率较高，但是性能较差

通常的大小为1k、2k、4k


#### 记录空闲块
使用磁盘块链表：链表的每个块中尽可能多的空闲磁盘块号。
示例：1KB大小的块和32位的磁盘块号，空闲表中每个块包含有255个空闲块的块号（需要有一个位置存放指向下一个块的指针）。
![[images/Pasted image 20250615155924.png]]

另一种空闲磁盘空间管理的方法是采用位图：n个块的磁盘需要n位位图。在位图中，空闲块用1表示，已分配块用0表示。
- 位图所需空间较少，因为每个块只用一个二进制位标识，而在链表中，每一块要用到32位。


counting：
- 与其保留n个空闲磁盘空间地址的列表，我们可以保留第一个空闲块的地址和第一个块后的连续空闲块的数量n。
- 可用空间列表中的每个表项都由一个磁盘地址和一个计数组成。
- 虽然每个表项比简单的磁盘地址需要更多的空间，但只要计数通常大于1，整个列表会比较短。


#### 文件系统可靠性
- 文件系统的丢失可能是毁灭性的
- **备份**(backup)是为了处理
	- 从意外的灾难中恢复
	- 从错误的操作中恢复
- 在哪里备份：三级存储器（Tertiary storage）
- 备份需要时间和空间

备份问题：
- 是否要备份整个文件系统
	- 通常不备份特殊I/O文件，二进制文件
- 不要备份自上次备份以来未备份的文件
	- **增量转储**（Incremental dumps）：周期性地做全面的备份，而每天只对从上一次全面转储起发生变化的数据做备份。
- 在写入磁带之前压缩数据
- 如果备份活动文件系统
	- 在备份时间内使系统脱机是不可接受的
- 备份介质的安全性

将磁盘转到磁带的两种策略：
- 物理转储（physical dump）：从块0开始到最后一个，按顺序写入每个块
	- 优点：简单
	- 缺点：
		- 无法跳过所选目录
		- 无法进行增量转储
		- 无法根据要求还原单个文件
- 逻辑转储（logical dump）：从一个或几个指定的日期开始，递归地转储其自给定基准日期后有所更改的全部文件和目录。
	- 基准日期可以是上次增量转储、上次全面转储等
	- 还要转储位于已修改文件或目录路径上的所有目录（即使未修改）

#### 文件系统的一致性（file system consistency）
大多数系统都有一个检查文件系统一致性的应用程序。
- Windows：scandisk
- UNIX：fsck

可以进行两种类型的一致性检查：
- 块的一致性检查
- 文件的一致性检查

fsck：块的一致性检查：
- 基本事实：每个磁盘块都在文件（或目录）中，或在空闲列表中
- 程序构造两张表，每张表中为每个块设立一个计数器，都初始化为0。
	- 第一个表中的计数器跟踪该块在文件中的出现次数
	- 第二个表中的计数器跟踪该块在空闲表或空闲位图中的出现次数
- 接着检验程序使用原始设备读取所有的i节点，忽略文件的结构，只返回从零开始的所有磁盘块。由i节点开始，可以建立相应文件中用到的全部块的块号表。每当读到一个块号时，该块在第一个表中的计数器加1。然后，该程序检查空闲表或位图，查找全部未使用的块。每当在空闲表中找到一个块时，就会使它在第二个表中的计数器加1。
- 如果文件系统一致，则每一块或者在第一个表计数器中为1，或者在第二个表计数器中为1。

![[images/Pasted image 20250615172657.png]]
（a）文件系统一致（consistent）
（b）块丢失（miss block）：磁盘块2没有出现在任何一张表中。可以通过文件系统检验程序把它们加到空闲表中即可。
（c）空闲表中有重复块：重新建立空闲表即可。
（d）重复数据块：在两个或多个文件中出现同一个数据块。先分配一空闲块，把重复块中的内容复制到空闲块中，然后把它插到其中一个文件中。这样文件的内容未改变，但至少保持了文件系统的一致性。


fsck：检查目录：
- 使用每个文件计数器而不是每个块
- 从根目录递归下降，检查文件系统中的每个目录。
- 对每个目录中的每个文件，将文件使用计数器加1。硬链接也参与计数，但是软链接不计数。
- 将计数器值与存储在文件i节点中的链接数目相比较。当文件创建时，这些计数器为1，随着每次对文件的一个硬链接的产生，对应计数器加1。如果文件系统一致，这两个计数应相等。
- 如果引用计数太大或太小。解决方案：强制i节点中的链接计数为目录表项的实际数量。

#### 文件系统性能
- 高速缓存：避免磁盘I/O开销
- 块提前读：提高缓冲区缓存命中率
- 减少磁盘臂运动：避免搜索开销
