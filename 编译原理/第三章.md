## 语法分析（syntax analysis）
在词法分析后，我们有一系列词法单元。
在语法分析中，我们希望
- 恢复这一系列词法单元描述的结构
- 如果这些词法单元未正确编码结构，就报告错误

![[images/Pasted image 20250619231402.png]]![[images/Pasted image 20250620101140.png]]

输入：解析器调用扫描程序来获取下一个词法单元
输出：需要构建显式或隐式语法树
![[images/Pasted image 20250620102530.png]]

用于语法分析的形式：
- 上下文无关文法（context-free grammer）
	- CFG和Grammars
	- Derivations（推导）
	- Concrete and Abstract Syntax Trees
	- Ambiguity（二义性）
- 解析算法：
	- 自顶向下（Top-Down Parsing）
	- 自底向上（Bottom-Up Parsing）

一个字母表是一组由字母组成的符号集∑。
∑上的*语言*是由∑中的符号组成的一组字符串。
在扫描时，我们的字母表是ASCII字符。我们生成词法单元。
当解析时，我们的字母表是扫描程序生成的词法单元集。

![[images/Pasted image 20250620105507.png]]
在扫描时，我们使用正则表达式来定义每个词法单元。
但是正则表达式不能定义编程语言。
- 无法定义匹配所有表达式的正则表达式，并带有适当平衡的括号。
- 无法定义与具有正确嵌套块结构的所有函数匹配的正则表达式。

### 文法
文化的形式化定义：
G=(Vt, Vn, P, S)


一个**上下文无关文法**（CFG）是一个由终结符号、非终止符号、一个开始符号和一组产生式组成。
- **终结符号**（terminals）是组成串的基本符号。术语“词法单元名字”是“终结符号”的同义词。我们经常使用“词法单元”来指称终结符号。
- **非终结符号**（nonterminals）是表示串的集合的语法变量。非终结符号表示的串集合用于定义由文法生成的语言。非终结符号给出了语言的层次结构，而这种层次结构是语法分析和翻译的关键。
- 在一个文法中，某个非终结符号被指定为**开始符号**（start symbol），表示的式该文法中最大词法单元。这个符号表示的串集合就是这个文法生成的语言。按照惯例首先列出开始符号的产生式。
- 一个文法的产生式描述了将终结符号和非终结符号组合成串的方法。每个**产生式**（production）由下列元素组成：
	- 一个被称为产生式**头**或**左部**的非终结符号。这个产生式定义了这个头所代表 的串集合的一部分
	- 符号→。有时也用`::=`来替代箭头。
	- 一个由零个或多个终结符号与非终结符号组成的产生式**体**或**右部**。产生式体中的成分描述了产生式头上的非终结符号所对应的串的某种构造方式。

- 下述符号是终结符号：
	- 在字母表里排在前面的小写字母，比如a，b，c
	- 运算符号，比如`+`、`*`等
	- 标点符号，比如括号、逗号等
	- 数字0、1、...、9
	- 黑体字符串，比如**id**或**if**。每个这样的字符串表示一个终结符号
- 下述符号是非终结符号：
	-  在字母表中排在前面的大写字母，比如A、B、C
	- 字母S。它出现时通常表示开始符号
	- 小写、斜体的名字。比如*expr*或*stmt*
- 当讨论程序设计语言的构造时，大写字母可以用于表示代表程序构造的非终结符号。比如，表达式、项和因子的非终结符号通常分别用E、T和F表示
- 在字母表中排在后面的大写字母（比如X、Y、Z）表示文法符号。也就是说，表示非终结符号或终结符号
- 在字母表中排在后面的小写字母（比如x、y、z）表示（可能为空的）终结符号串
- 小写的希腊字母，比如α、β，表示（可能为空的）文法符号串
- 除非特别说明，第一个产生式的头就是开始符号

With the following conventions, we can only write productions of a grammar
- Unless otherwise stated, the left side of the first production is the start symbol
- Using lower-case letters to represent terminals
- Using upper-case letters or name with <…> to represent nonterminals
- If  A→α1,A→α2,…,A→αn are all productions with A on the left, we may write A→α1|α2|…|αn

正则表达式的语法不适用于CFG。
- 不能使用`*`、`|`或括号。

#### 推导（derivation and language）
从开始符号出发，每个重写步骤把一个非终结符号替换为它的某个产生式的右部。
- 从开始符号S出发，反复用产生式规则（α→β）替换非终结符，最终生成终结符串的过程。

![[images/Pasted image 20250620115404.png]]

- `=>*`被定义为一个推导步骤的传递闭包
	- `α=>*β`当且仅当存在0个或多个推导步骤：`α0=>α1=>α2=>...=>αn-1=>αn`
	- 使得α=α0和β=αn。如果n=0，则α=β。

设G的起始符号为S的上下文无关文法，那么S的语言是：
`L(G) = {s∈VT* | there exists S =>* s of G}`
S是G的起始符号，s表示任意一串词法单元符号。（有时称为句子（sentence））

G的句子形式：
S是G的起始符号，如果`S =>* α, α∈(VN U VT)*`，α是G的句子形式（sentential form）。
G的一个句子：
w是G的句子形式，如果w仅包含终结符号，则w是G的一个句子。

### 语法分析树（Parse Trees）
- 语法分析树是一个词法单元字符串的结构的有效表示
- 语法分析数直观地表示推导

一个文法G上的语法分析树G是具有以下属性的有根标记树：
- 根节点（root）标有起始符号S
- 每个叶节点（leaf）都表示终结符号或ε
- 每个非叶节点都表示非终结符号
- 如果一个节点标记为A∈VN有n个标签为X1，X2，...，Xn的子节点（可能是非终结符号也可能是终结符号），则：A→X1X2...Xn∈P

一个推导可以被表示为一个语法分析树
- 开始符号为根节点
- 对于产生式X→Y1...Yn，添加子节点Y1，...，Yn到节点X

### 二义性（ambiguity）
如果一个文法可以为某个句子生成多棵语法分析树，那么它就是二义性的。换句话说，二义性文法就是对同一个句子有多个最左推导或多个最右推导的文法。

![[images/Pasted image 20250620142224.png]]

如果存在字符串w∈L(G)，使得w具有多个不同的语法分析树（或最左/最右推导），则G是具有二义性的。
二义性在编程语言中是非常常见的。
- 优先级和关联
- 悬空else问题（IF-THEN-ELSE）

有两种方法解决二义性问题：
- 消二义性规则（disambiguating rule）
- 重写文法

#### 消二义性规则
声明一个规则，该规则指定在每个模棱两可的情况下哪个语法分析树是正确的。
![[images/Pasted image 20250620142747.png]]
- 没有改变文法
- 语法结构不再经由文法提供

#### 重写文法
将文法更改为强制构建正确语法解析树的形式。

### 自顶向下分析
开始于开始符号，尝试寻找输入串的最左推导的过程。
可以看作为输入串构造语法分析树的问题，它从语法分析树的根节点出发，按照从根到叶先序创建这棵语法分析树的各个节点。
![[images/Pasted image 20250620144144.png]]
![[images/Pasted image 20250620144423.png]]

将分析视为图搜索
- **每个节点都是一种句子形式**（一个可以由起始符号推导到的终结符号和非终结符号组成的串）
- 如果α=>β，则从α到β有一条边
![[images/Pasted image 20250620144741.png]]

自顶向下分析的关键问题为推导步骤中选择的产生式。
- 如果最左端要被替代的非终结符号是B，这里总共有n个B的产生式：`B→A1|A2|...|An`，要怎么知道选择哪一个
- **回溯分析**（Backtracking Parsing）（深度优先（Depth-First））：
	- 如果非终结节点有多个产生式，则根据当前输入符号，分析器无法确定选择哪个产生式
	- 如果一种可能性失败，则必须尝试不同的可能性
	- 但是回溯会使其变慢
- **预测分析**（Predictive Parsing）：
	- 分析器尝试使用一个或多个提前词法单元来预测输入字符串中的下一个结构
	- 两种预测分析：
		- **递归下降分析**（Recursive-descent parsing）
		- LL(1) parsing

### Predictive Parsing
#### 条件
- 对输入字符串的分析从文法的开始符号开始
- 如果它可以根据输入中的提前词法单元唯一地确定在推导中下一步使用哪个产生式，则分析是可预测的。
- 预测分析器接受LL(k)文法
	- L表示“从左到右”扫描输入
	- L表示“最左推导”
	- k表示“需要  $\leq k$个提前词法单元来预测”
	- 在实践中使用LL(1)文法

#### LL(1)文法
一个文法G是LL(1)的，当且仅当G的任意两个不同的产生式`A→α|β`满足以下的条件：
- 不存在终结符号a使得α和β都能够推导出以a开头的串
- α和β中最多只有一个可以推导出空串
- 如果`β=>*ε`，那么α不能推导出任何以FOLLOW(A)中某个终结符号开头的串。
- 也就是FIRST(A)和FIRST(B)是不相交的集合。如果ε在FIRST(β)中，那么FIRST(α)和FOLLOW(A)是不相交的集合。

![[images/Pasted image 20250620160756.png]]

识别LL(1)文法：
- Compute the set of nullable nontermianls
- Compute FIRST(α) for the right-hand side string α of  each production
- Compute FOLLOW(A) for each nonterminal A
- Recognize based on the definition of LL(1)

**Nullable**
- 如果一个非终结符号A存在推导：`A=>*ε`，则A是nullable的
- 算法：让U表示一个可控的非终结符号集
	- 首先，U={Ai | Ai→ε是一个生成式}
	- 对于每个生成式p：Ap→X1...Xn，如果X1...Xn∈U，则U:=UU{Ap}
	- 重复第二步直到U没有变化

##### 非LL(1)文法
- 如果一个文法有左因子（**left factor**）或左递归（**left recursion**），则其不是LL(1)文法
- 一个文法既没有左因子也没有左递归也不一定是LL(1)文法

##### 左因子
左因子是两个或多个文法规则选择共享一个共同的前缀字符串，如A→αβ|αr
因为FIRST(αβ)∩FIRST(αr)≠Ø，所以这不是LL(1)文法

##### 左递归
如果文法的生成具有以下形式，则文法是递归的：
- A→Aβ：立即左递归，其中左递归仅发生在单个非终结符号的生成式中
- A→Bβ       B→Aα：间接左递归，其中A=>Bβ=>Aαβ，有`A=>*A`
- 其中α和β为任意字符串
- 所以这个不是LL(1)文法

#### 将非LL(1)文法改写为LL(1)的方法
- 左递归删除（Left recursion removal）
- 左因式分解（Left factoring）

注意：这个方法不能保证这些计数的应用会将文法转换为LL(1)文法

	- 选项对应于代码中的替代项（case或if语句）

#### LL(1)分析

![[images/Pasted image 20250624111323.png]]
一个成功的LL(1)解析原理图是：
- 解析首先将开始符号推入堆栈中
- 将堆栈顶部的词法单元与下一个输入字符串匹配，匹配后，通过弹出堆栈并推进输入来丢弃堆栈中的词法单元和输入
- 使用A→α来替换堆栈顶部的非终结符号A（α反向推到堆栈上）
- 堆栈和输入再成功解析结束时为空

![[images/Pasted image 20250624111911.png]]
LL(1)分析表：
功能：表示非终结符号在适当的解析步骤中使用的产生式选择
- 有非终结符号和终结符号索引的二维数组
- `M[N, t]`是根据当前输入t（终结符号或$）在适当步骤中使用的N的产生式选择
- 如果`M[N, t]`在构造后保持为空，则表示在解析过程中可能发生的潜在错误
对每个非终结符号A和产生式A→α：
- 对于First(α)中的每个词法单元a，添加A→α到`M[A, a]`
- 如果ε在First(α)中，则对于Follow(A)中的每个元素a（词法单元或$），添加A→α到`M[A, a]`

如果关联的LL(1)分析表在每个表项中最多只有一个生成式，则该文法是LL(1)文法

#### 错误恢复
解析器必须确保程序在语法上是否正确
解析器对语法错误的响应通常是编译器有用性的关键因素

- 正常的语法错误：
	- 开始和后续符号错误
		- 程序、表达式和语句的开始和后续符号
![[images/Pasted image 20250624113755.png]]
- 标识符和常量错误
	- 例如，“const”、“var”、“procedure”后面没有标识符
- 括号匹配错误
	- 例如，(-)、do-while不匹配
- 符号错误
	- 例如相等关系中的符号不是`==`

##### 错误处理
- 提供有意义的错误消息。
- 选择一个可能的位置来恢复解析。解析器应该始终尝试解析尽可能多的代码，以便在单个翻译期间找到尽可能多的实际错误。

##### 递归下降解析器中的错误恢复
恐慌模式（panic mode）：语法分析器忽略输入中的一些符号，直到输入中出现由设计者选定的同步词法单元集合中的某个词法单元
- 为每个递归过程提供一个额外的参数，该参数由一组同步词法单元组成
- 如果遇到错误，解析器会向前扫描，丢弃词法单元，直到在输入中看到同步词法单元集中的之一


### 自底向上分析（Bottom-Up Parsing）
定义：
解析从输入字符串开始，通过缩减步骤（**归约**（reduction）），尝试将输入字符串缩减为语法的开始符号。
解析树的构造：
输入字符串是解析树的叶子，解析工作到根，即开始符号

使用`|-`表示归约步骤，即最右推导的反向步骤

- 自底向上的关键是如何确定要归约的子字符串

- Top-down parsing stack contains tokens and nonterminals, while bottom-up parsing stack contains tokens, nonterminals and states
- Top-down parsing stack stores symbols waiting for being matched in the parsing, while bottom-up parsing stack stores the symbols having been matched

根据堆栈内容，并使用输入中的下一个词法单元作为前瞻来确定要执行的下一个动作
自底向上的分析器有两种可能的动作：
- Shift：将终端从输入的前面移动到堆栈的顶部
- Reduce：将堆栈顶部的α减少给一个非终结符号A，给出生产式A→α
所以自底向上的分析也称为shift-reduce分析

![[images/Pasted image 20250624142608.png]]

#### 特征（character）
自底向上分析实现的关键：
- 确定合适转移和何时减少
- 不同的确定方法会导致具有不同功效和复杂性的分析器

自底向上分析的特征：
- 右句型（Right Sentential Form）
- 可行前缀（Viable Prefix）
- 句柄（Handle）

##### Right Sentential Form
shift-reduce分析器以相反的顺序跟踪输入字符串的最右推导：
- 例：S→aAcBe    A→b     A→Ab     B→d
- “abbcde”的最右推导为：`abbcde |- aAbcde |- aAcde |- aAcBe |- S`
最右推导中的每个中间判断形式都称为**右句型**（right sentential form）

![[images/Pasted image 20250624143934.png]]
shift-reduce分析器会将终结符号从输入移动到堆栈直到可以执行归约以获得下一个正确的右句型


##### Viable Prefix
分析堆栈上的符号序列称为右句型的**可行前缀**
![[images/Pasted image 20250624144154.png]]
只要分析堆栈中的内容是右句型的可行前缀，则shift-reduce分析是正确的。


##### Handle
定义：
右句型γ=αβw的句柄是一个子字符串β，它与产生式A→β的右侧匹配，并且其可以替换A来产生一个γ的最右推导的前右句型

也就是说，右句型αβw的句柄β有三种条件：
- αβw是一个右句型（`S=>*αβw`）
- A→β是一个产生式
- `S=>*αAw=>αβw`（句柄右侧的w只包含终结符号）
![[images/Pasted image 20250624144931.png]]
然而，在很多情况下，与某些 产生式A→β的最右侧匹配的最左侧子字符串被β可能不是句柄，因为产生式A→β的归约会产生一个不是右句型的字符串
![[images/Pasted image 20250624145332.png]]

- 句柄和可行前缀：
	- 可行前缀是右句型的前缀，它从左到右不超过该句型的句柄的右侧
	- 示例：右句型：aAbcde（Ab为句柄）
	- 可行前缀为a、aA、aAb
- 句柄和shift-reduce分析：
	- 确定分析器中的下一个句柄是shift-reduce分析器的主要任务
	- 当下一个句柄位于堆栈顶部，将执行动作“reduce”
	- 当下一个句柄还未在堆栈顶部形成，将执行动作“shift”
![[images/Pasted image 20250624150049.png]]

#### LR分析方法
LR(k)分析：基于分析堆栈顶部的字符串（表示为状态），并使用输入中的下一个k（$0 \leq k$）词法单元作为前瞻来确定要归约的句柄
- L表示从左到右处理输入
- R表示生成最右推导
- k表示做出分析决策的前瞻输入符号数

LR(k)分析能力的结果：
- LR(0)分析，其中在做出分析决策时不参考前瞻符号
- SLR(1)分析（简单的LR(1)）是对LR(0)的改进
- LR(1)分析是最强大和最复杂的（只有一个前瞻符号）
![[images/Pasted image 20250624150818.png]]

分析表：每一行代表一个状态，每一列是一个文法符号
![[images/Pasted image 20250624150935.png]]
- ACTION：获取一个状态和一个终结符号，确定要发生的下一个动作
- GOTO：获取一个状态和一个文法符号，确定下一个状态

![[images/Pasted image 20250624151211.png]]
- ACTION表
- 状态Sm和输入ai的对应表项`action[Sm,ai]`有四个值：
	- Shift(sk)：将符号ai和状态K放入堆栈
	- Reduction(rk)：归约k个产生式（A→γ），这个动作包括：
		- 从堆栈中弹出字符串γ及其所有对应的状态。假设当前栈顶是状态Si
		- 将A推入栈中
		- 将状态`Sj=GOTO[Si, A]`推入栈中
	- Accept：指示分析已成功完成
	- Error（Empty）：指示分析已发现错误
![[images/Pasted image 20250624151803.png]]


#### LR(0)项和LR(0)分析的有限自动机
LR(0)分析：
- 使用LR(0)分析表的LR分析器的LR(0)分析器；可以为其构建LR(0)分析表的文法称为LR(0)文法
- LR(0)分析器仅使用堆栈的内容来确定句柄，它不需要输入词法单元来作为前瞻
- 几乎所有“真正的”文法都不是LR(0)

##### LR(0)项（item）
- 文法G的LR(0)项是G的产生式，在其右侧具有显著的位置
- 例：产生式U→XYZ有四个项：
	- U→·XYZ
	- U→X·YZ
	- U→XY·Z
	- U→XYZ·
- 产生式A→ε只有一个项A→·
- 这些项称为LR(0)项，因为它们不包含对前瞻的显式引用

###### 项的含义
一个项记录了产生式右侧识别的中间步骤：
- A→·βγ意味着我们可能即将通过产生式A→βγ来识别A
- A→β·γ意味着已经看到β（β必须出现在栈顶）
- A→βγ·，表示βγ现在位于栈顶，可能是句柄，而A→βγ将用于下一个归约


- 初始项（Initial Item）：形式为A→·α的项，表示识别α的首字母
- 完成项（Complete Item）：形式为A→α·的项，表示识别α的完整性

##### 项的有限自动机
LR(0)项可以作为有限自动机的状态，该自动机维护有关分析堆栈和shift-reduce分析进度的信息
![[images/Pasted image 20250624161016.png]]
![[images/Pasted image 20250624162005.png]]

##### 直接构造项的DFA
**增广文法**（augmented grammar）：在构建DFA之前，首先通过单个产生式S‘→S，来扩充文法，其中S’为新的非终结符号，它称为增广文法的开始符号
- S‘→·S称为DFA开始状态中的第一个项
![[images/Pasted image 20250624162326.png]]
- DFA中每个状态都是一组LR(0)项
	- 如何构造一个状态：闭包操作（closure operation）
- 符号X上从状态i到状态j的转换
	- 如果X是一个词法单元，这个转换对应于在分析期间X从输入到堆栈顶部的shift
	- 如果X∈VN，则此转换对应于将X推送到堆栈顶部。它只能在产生式X→r的归约期间发生
	- 如何构造从一种状态到另一种状态的过渡：goto operation

###### The Closure Operation
如果I是一组LR(0)项，那么closure(I)是按照一下规则从I构造的项集：
- 最初，closure(I)=I，将I中的各项加入到closure(I)中
- 如果A→α·Bβ在closure(I)中且B∈VN，则对于每个产生式B→r，将项B→·r添加到closure(I)中（如果不存在）
- 重复上一步，直到没有更多新项被添加
- 对于‘·’位于游荡或者跟着终结符号的每个项，此项的闭包为其本身
![[images/Pasted image 20250624163255.png]]

###### The goto Operation
I是一组项，X∈VN∪VT
goto(I, X) = closure(J)
其中J是I中所有形如`[A→α·Xβ]`的项所对应的项`[A→αX·β]`的集合
![[images/Pasted image 20250624163859.png]]

###### DFA的构建
- closure({S‘→·S})是DFA的开始状态，且未标记
- 从DFA获取未标记的状态ISi
	- 标记ISi
	- 对于ISi的每个项目U→x·Ry（R∈VN∪VT，x和y是字符串），计算goto(ISi, R)=ISj
		- 如果ISj不存在，则将ISj作为未标记添加到DFA
		- 在R上添加从ISi到ISj的转换
- 重复上面第二步，直到DFA中没有未标记的状态
![[images/Pasted image 20250624165826.png]]


##### 构建LR(0)分析表
![[images/Pasted image 20250624165924.png]]
给定文法G，我们增广G以产生G‘
- 为G’构建LR(0)项集的DFA
- 状态K的ACTION部分确定如下（将每一个状态映射到一个动作）：
	- 如果A→α·β∈K，则`ACTION[K]=Shift`
	- 如果A→α·∈K（归约项），并且A→α的数量为j，则设置`ACTION[K]=Rj`
- 状态K的GOTO部分（将状态/符号对映射到下一个状态），使用以下规则为所有符号构建：
	- 如果goto(K, X)=J，X∈VN∪VT∪{$}，则设置`GOTO[K, X]=j`
![[images/Pasted image 20250624170658.png]]


#### SLR(1)分析
- 项集中的冲突
	- Shift-Reduce Conflict：如果一个集合包含shift项A→α·β和完成项B→r·，则对于是将a移动还是将r归约到B会产生歧义
	- Reduce-Reduce Conflict：如果集合包含完整项A→β·和B→r·，则对于使用哪个产生式进行归约会产生歧义
当且仅当项集中没有上述冲突时，文法为LR(0)
![[images/Pasted image 20250624180246.png]]

##### 消除SLR(1)中的冲突
SLR(1)（Simple LR(1)）的主要思想为
- SLR(1)分析时LR(0)的一个简单而有效的扩展
- 它使用LR(0)项集的DFA
- 它通过使用输入字符串中的下一个词法单元来指导其操作，从而提高LR(0)的解析的能力
- 前瞻的简单使用注意解析几乎所有实用语言

使用前瞻词法单元的两个方法：
- 它会在shift之前查询输入词法单元，以确保存在适当的DFA过渡
- 它使用非终结符的Follow集来决定是否进行归约
对于项A→r·，仅当下一个词法单元a∈FOLLOW(A)时，才会进行归约
