# kafka的优势
- 高吞吐量：Kafka的高吞吐量表现非常惊人，单机每秒处理几十上百万的消息量，即使存储了TB级别的消息，它依然能够保持稳定的性能
- 高性能：Kafka的高性能不仅体现在高吞吐量上，还体现在它可以支持大量的客户端连接，单节点可以支持上千个客户端，并保证零停机和零数据丢失
# 创建主题Topic
使用Kafka之前，第一件事是必须创建一个主题（Topic）
- Topic类似于文件系统中的文件夹
- Topic用于存储事件（events）
	- events也称为记录或消息，比如支付交易、手机地理位置更新、运输订单、物联网设备或医疗设备的传感器测量数据等等都是events
	- events被组织和存储在topic中
	- 简单来说，topic类似于文件系统中的文件夹，events是该文件夹中的文件

创建主题使用：`kafka-topics.sh`脚本（docker中在/opt/kafka/bin目录下）
- 不带任何参数会告知该脚本如何使用：`./kafka-topics.sh`
- 创建topic：`./kafka-topics.sh --create --topic <topic_name> --bootstrap-server localhost:9092`
	- `--bootstrap-server localhost:9092`这个参数是必须的，用于指明kafka服务器
- 查看所有可用的topic：`./kafka-topics.sh --list --bootstrap-server localhost:9092`
- 删除指定的topic：`./kafka-topics.sh --delete --topic <topic_name> --bootstrap-server localhost:9092`
- 显示topic详细信息：`./kafka-topics.sh --describe --topic <topic_name> --bootstrap-server localhost:9092`
- 修改主题信息：`./kafka-topics.sh --alter --topic hello --partitions 5 --bootstrap-server localhost:9092`

# 概念
![[images/Pasted image 20251024101251.png]]
- Kafka中每个topic可以有一个或多个分区（Partition）
- 当创建topic时，如果不指定该topic的patition数量，那么默认就是1个partition
- 偏移量（offset）是标识每个partition中消息的唯一位置，从0开始
- 默认情况下，当启动一个新的消费组时，它会从每个分区的最新偏移量（即该分区中最后一条消息的下一个位置）开始消费。


# 生产者，消费者
一个生产者，一个消费者的模式：
```Go
package main  
  
import (  
    "fmt"  
    "log"  
    "github.com/IBM/sarama")  
  
func main() {  
    // 配置生产者  
    config := sarama.NewConfig()  
    config.Producer.RequiredAcks = sarama.WaitForAll // 等待所有副本确认  
    config.Producer.Retry.Max = 5                    // 重试次数  
    config.Producer.Return.Successes = true          // 返回成功的消息  
  
    // 创建同步生产者  
    producer, err := sarama.NewSyncProducer([]string{"localhost:9092"}, config)  
    if err != nil {  
       log.Fatalf("Failed to start Sarama producer: %v", err)  
    }  
    defer producer.Close()  
  
    // 发送消息  
    topic := "test-topic"  
    for i := 0; i < 5; i++ {  
       msg := &sarama.ProducerMessage{  
          Topic: topic,  
          Value: sarama.StringEncoder(fmt.Sprintf("Hello World %d", i)),  
       }  
  
       // 同步发送，返回分区和偏移量  
       partition, offset, err := producer.SendMessage(msg)  
       if err != nil {  
          log.Printf("Failed to send message: %v", err)  
       } else {  
          log.Printf("Message sent to partition %d at offset %d", partition, offset)  
       }  
    }  
}
```

```Go
package main  
  
import (  
    "log"  
  
    "github.com/IBM/sarama")  
  
func main() {  
    // 配置消费者  
    config := sarama.NewConfig()  
    config.Consumer.Return.Errors = true  
    config.Consumer.Offsets.Initial = sarama.OffsetOldest // 从最新的偏移量开始消费  
  
    // 创建消费者（非消费组模式）  
    consumer, err := sarama.NewConsumer([]string{"localhost:9092"}, config)  
    if err != nil {  
       log.Fatalf("Error creating the consumer: %v", err)  
    }  
    defer consumer.Close()  
  
    // 指定要消费的主题和分区  
    topic := "test-topic"  
    partition := int32(0) // 分区ID（从0开始）  
  
    // 获取分区消费者（从指定偏移量开始消费）  
    // 第三个参数为起始偏移量，这里使用 sarama.OffsetNewest 表示从最新的消息开始消费，使用 sarama.OffsetOldest 则表示从最早的消息开始消费  
    pc, err := consumer.ConsumePartition(topic, partition, sarama.OffsetOldest)  
    if err != nil {  
       log.Fatalf("Error creating partition consumer: %v", err)  
    }  
    defer pc.Close()  
  
    // 消费消息  
    log.Println("Starting to consume messages...")  
    for msg := range pc.Messages() {  
       log.Printf("Received message: %s, Partition: %d, Offset: %d", string(msg.Value), msg.Partition, msg.Offset)  
    }  
}
```

# 消费者组
默认消费者，只能从最开始的位置消费
要么从最新的位置开始消费，要么就自己指定偏移量去消费
消费者自己不知道自己消费到哪里了

使用消费者组（consumer group）可以自动给实现 “记录消费位置” 的功能，无需手动管理偏移量，这个是消费者组最核心的优势之一

消费者组会通过Kafka集群自动跟踪和存储每个分区的消费偏移量（默认存储在Kafka内部的`__consumer_offsets`主题中），并在消费者重启或重平衡（rebalance）时自动恢复消费位置，无需手动持久化偏移量
```GO
package main  
  
import (  
    "context"  
    "fmt"    "log"    "os"    "os/signal"    "syscall"  
    "github.com/IBM/sarama")  
  
// GroupConsumer 自定义消费者，实现 sarama.ConsumerGroupHandler 接口  
type GroupConsumer struct{}  
  
// Setup 在消费者组启动时调用  
// 在一个新的消费会话开始时，sarama 会调用此方法  
func (gc *GroupConsumer) Setup(sarama.ConsumerGroupSession) error {  
    return nil  
}  
  
// Cleanup 在分区被重新分配强调用（清理）  
// 在消费会话结束时，sarama 会调用此方法  
func (gc *GroupConsumer) Cleanup(sarama.ConsumerGroupSession) error {  
    return nil  
}  
  
// ConsumeClaim 处理每个分区的消息  
// 这是最核心的方法。一旦消费者组成功连接并分配了分区，sarama会为每个分配到的分区启动一个goroutine，并在这个goroutine中调用这个方法。  
func (gc *GroupConsumer) ConsumeClaim(session sarama.ConsumerGroupSession, claim sarama.ConsumerGroupClaim) error {  
    // 遍历消息  
    for msg := range claim.Messages() {  
       // 处理消息  
       fmt.Printf("Received message: %s, Partition: %d, Offset: %d\n", string(msg.Value), msg.Partition, msg.Offset)  
  
       // 手动提交偏移量  
       // 在消息处理成功后再提交，避免消息丢失  
       session.MarkMessage(msg, "") // 标记消息为已处理  
    }  
    return nil  
}  
  
func main() {  
    // 配置消费者组  
    config := sarama.NewConfig()  
    config.Version = sarama.V2_5_0_0  
  
    // 偏移量配置，默认自动提交  
    config.Consumer.Offsets.AutoCommit.Enable = true      // 启用自动提交  
    config.Consumer.Offsets.AutoCommit.Interval = 5000    // 自动提交间隔5毫秒  
    config.Consumer.Offsets.Initial = sarama.OffsetOldest // 从最早的偏移量开始消费  
  
    // 创建消费者组  
    groupId := "auto-offset-group" // 消费者组唯一标识  
    consumerGroup, err := sarama.NewConsumerGroup([]string{"localhost:9092"}, groupId, config)  
    if err != nil {  
       log.Fatalf("Error creating consumer group %v", err)  
    }  
    defer consumerGroup.Close()  
  
    // 要消费的主题  
    topics := []string{"test-topic"}  
    consumer := &GroupConsumer{}  
  
    // 启动消费循环  
    go func() {  
       for {  
          err := consumerGroup.Consume(context.Background(), topics, consumer)  
          if err != nil {  
             log.Fatalf("Error during consumption: %v", err)  
             return  
          }  
       }  
    }()  
  
    // 等待中断信号  
    sigChan := make(chan os.Signal, 1)  
    signal.Notify(sigChan, syscall.SIGINT, syscall.SIGTERM)  
    <-sigChan  
    log.Println("Shutting down consumer group...")  
}
```
消费者组核心功能：
- 能直到自己消费到哪里了
- 同一个组，不同的实例可以故障转移，其中的实例停止后，会有另一个实例继续消费
## 负载均衡模式
如果跑相同的消费者组的消费者
则默认只有其中一个消费者可以消费消息（因为topic只有一个分区）
然后把那个能消费消息的消费者停掉，另一个就可以消费到消息，这个就是负载均衡模式
- 假设一个topic有2个partitions
	- 消费者组中只有一个实例，该实例会消费2个patrition中所有消息
	- 消费者组中有两个实例，每个实例都会消费其中一个patrition的消息，假如其中一个实例中断，另一个实例会继续消费整个topic的消息
	- 消费者组中有三个实例，有两个实例会消费其中一个patrition的消息，还有一个无法消费消息，假如有一个消费消息的实例中断，则那个无法消费消息的实例会顶上

广义上的负载均衡模式，应该是一个生产者生产消息，消息被轮询到不同的消费者上
这个需要再创建topic的时候，指定分区数量
```Go
package main  
  
import (  
    "fmt"  
    "log"    
    "time"  
    "github.com/IBM/sarama")  
  
func createTopicIfNotExists(brokers []string, topic string, partitionCnt int32, replicationFactor int16) error {  
    // 配置  
    config := sarama.NewConfig()  
    config.Version = sarama.V2_5_0_0 // 指定 Sarama 版本，建议与 Kafka broker 版本匹配  
    config.Net.DialTimeout = time.Second * 10  
  
    // 创建一个集群管理员  
    admin, err := sarama.NewClusterAdmin(brokers, config)  
    if err != nil {  
       return fmt.Errorf("创建集群管理员失败: %v", err)  
    }  
    defer admin.Close()  
  
    // 定义主题详情  
    topicDetail := &sarama.TopicDetail{  
       NumPartitions:     partitionCnt,  
       ReplicationFactor: replicationFactor,  
    }  
  
    // 创建主题  
    err = admin.CreateTopic(topic, topicDetail, false)  
    if err != nil {  
       // 如果主题已经存在，我们不认为这是一个错误  
       if e, ok := err.(*sarama.TopicError); ok && e.Err == sarama.ErrTopicAlreadyExists {  
          log.Printf("主题 %s 已存在，跳过创建", topic)  
          return nil  
       }  
       return fmt.Errorf("创建主题失败: %v", err)  
    }  
  
    log.Printf("主题 %s 创建成功，分区数：%d，副本因子：%d", topic, partitionCnt, replicationFactor)  
    return nil  
}  
  
func main() {  
    brokers := []string{"localhost:9092"}  
    topicName := "topic02"  
    partitionCnt := int32(2)  
    replicationFactor := int16(1)  
  
    err := createTopicIfNotExists(brokers, topicName, partitionCnt, replicationFactor)  
    if err != nil {  
       log.Fatalf("操作失败：%v", err)  
    }  
    log.Println("操作成功")  
}
```

# 部分概念
## ISR副本
在同步中的副本（In-Sync Replicas)，包含了Leader副本和所有与Leader副本保持同步的Follower副本；
写请求首先由Leader副本处理，之后Follower副本会从Leader上拉取写入的消息，这个过程会有一定的延迟，导致Follower副本中保存的消息略少于Leader副本，但是只要没有超过阈值都可以容忍，但是如果一个Follower副本出现异常，比如宕机、网络断开等原因长时间没有同步到消息，那这个时候，Leader就会把它踢出去，Kafka通过ISR集合来维护一个“可用且消息量与Leader相差不多的副本集合，它是整个副本集合的一个子集”；

在kafka中，一个副本要称为ISR副本，需要满足一定条件：
- Leader副本本身就是一个ISR副本
- Follower副本最后一条消息的offset与Leader副本的最后一条消息的offset之间的差值不能超过指定的阈值，超过阈值则该Follower副本将会从ISR列表中剔除
	- `replica.lag.time.max.ms`：默认是30秒，如果该follower在此时间间隔内一直没有追上过Leader副本的所有消息，则该Follower副本就会被剔除ISR列表
	- `replica.lag.max.messages`：落后了多少条消息时，该Follower副本就会被剔除ISR列表，该配置参数现在在新版本的Kafka已经过时

## LEO
LEO：日志末端偏移量（Log End Offset），记录该副本消息日志（log）中下一条消息的偏移量，注意是下一条消息，也就是说，如果LEO=10，那么表示该副本只保存了偏移量值是0~9的10条消息。

## HW
HW（Hight Watermark）：即高水位值，它代表一个偏移量offset信息，表示消息的复制进度，也就是消息已经成功复制到了哪个位置了？即在HW之前的所有消息都已经成功写入副本中并且可以在所有的副本中找到，因此，消费者可以安全地消费这些恶已经被成功复制的消息。

对于同一个副本而言，小于等于HW值得所有消息都被认为是”已备份“的，消费者只能拉取到这个offset之前的消息，确保了数据的可靠性。
