# 集群搭建
- 基于Zookeeper的集群搭建方式：
	- kafka是一个压缩包，直接解压即可使用，这里假设解压三个
	- 配置kafka集群：`server.properties`
		- 三台分别配置为：`broker.id=1、broker.id=2、broker.id=3`
		- 三台分别配置`listener=PAINTEXT:IP:PORT`
		- 三台分别配置`advertised.listeners=PAINTEXT:IP:PORT`
		- 配置日志目录：
			- `log.dirs=/tmp/kafka-logs-9091`
			- `log.dirs=/tmp/kafka-logs-9092`
			- `log.dirs=/tmp/kafka-logs-9093`
		- 配置zookeeper连接地址：`zookeeper.connect=localhost:2181`
	- 集群启动并测试
![[images/Pasted image 20251026160048.png]]
集群有三个节点都是Broker角色，其中一个Broker（褐色）是Controller节点，控制器节点将集群元数据信息（比如主题分类、消息进度等）保存到zookeeper，用于集群各节点之间分布式交互


- 基于KRaft的集群搭建方式：
	- 准备三个kafka
	- 配置kafka集群：
		- 配置`config/kraft/server.properties`文件
			- 三台分别配置为：`broker.id=1`、`broker.id=2`、`broker.id=3`
			- 三台分别配置节点角色：`process.roles=broker,controller`
			- 三台分别配置参与投票的节点：`controller.quorum.voters=1@节点1,2@节点2,3@节点3`
			- 三台配置各自监听本机的ip和端口：例：`listener=PLAINTEXT://0.0.0.0:9091,CONTROLLER://0.0.0.0:9081`
			- 三台配置对外开放访问的ip和端口：例：`advertised.listeners=PLAINTEXT://节点ip:端口`
			- 三台分别配置日志目录：例：`log.dirs=/tmp/kraft-combined-logs-9091`
	- 集群启动并测试
		- 生成Cluster UUID（集群UUID）：`./kafka-storage.sh random-uuid`
		- 格式化日志目录：`./kafka-storage.sh format -t 生成的uuid -c ../config/kraft/server.properties`
		- 启动Kafka
		- 关闭Kafka

![[images/Pasted image 20251026160226.png]]
- Controller节点在Kafka集群中扮演着管理和协调的角色，管理整个集群中所有分区和副本的状态，当某个分区的leader副本出现故障时，Controller负责为该分区选举新的leader副本
- Broker节点在Kafka集群中主要承担消息存储和转发等任务
一个集群有四个broker节点，人为指定其中三个作为Controller角色（蓝色），从三个Controller中选举出一个作为主控制器（褐色），其他两个备用，Zookeeper不再被需要，相关的元数据信息以kafka日志的形式存在（即以消息队列消息的形式存在）
