## 同步通讯与异步通讯
- **同步通讯**：就像**打电话**。你（发送方）拨通电话后，必须等待对方（接收方）接听，你们才能开始对话。在对话过程中，你说一句，必须等待对方回应，然后才能说下一句。这个“等待”期间，你不能去做别的事情（比如同时和另一个人打电话）。
- **异步通讯**：就像**发短信/邮件**。你编辑好一条信息发送出去后，不需要对方立刻回复。你可以立刻去做别的事情（比如发下一条信息给其他人，或者玩游戏）。对方会在方便的时候看到你的信息并回复，而你也会在方便的时候查看回复。发送和接收不是即时绑定的。

|特性维度|同步通讯 (Synchronous)|异步通讯 (Asynchronous)|
|---|---|---|
|**核心机制**|**调用方发起请求后，会主动等待并阻塞，直到收到响应结果。**|**调用方发起请求后，不会阻塞等待，可以立即返回去做别的事。响应结果通过回调、事件或消息等方式后续通知。**|
|**时间耦合性**|**强耦合**。发送和接收必须在同一时间点附近完成，要求接收方必须在线且 ready。|**弱耦合**。发送和接收可以发生在不同的时间，接收方可以暂时不在线。|
|**性能与资源**|**资源占用高**。在等待响应期间，调用方的线程或进程会被挂起，占用系统资源（如内存）却不能做其他工作，容易成为性能瓶颈。|**资源利用率高**。调用方不会被阻塞，有限的资源（如线程）可以处理大量请求，吞吐量更高。|
|**设计复杂性**|**相对简单、直观**。代码流程是线性的，符合顺序执行的思维习惯（调用 -> 等待 -> 得到结果 -> 继续）。|**相对复杂**。流程被割裂，需要通过回调函数、Promise、消息队列、监听器等机制来处理响应，逻辑更绕，调试也更困难。|
|**可靠性**|较低。如果接收方故障或网络中断，调用方会立即得到错误或超时，整个流程会中断。|**较高**。通常依赖**消息队列（Message Queue）** 作为中间件，消息会被持久化存储。即使接收方宕机，消息也不会丢失，待接收方恢复后可以继续处理。|
|**典型应用场景**|• **实时交互系统**：电话、视频会议、在线游戏。<br>• **需要立即结果的操作**：数据库的实时查询、ATM取款（必须等待扣款成功才能吐钞）。|• **高并发、高吞吐系统**：电商下单、秒杀、微博/微信朋友圈发布。<br>• **耗时操作**：发送邮件、处理视频转码、大数据分析。<br>• **跨系统集成**：微服务之间的通信，避免系统间的故障连锁反应。|
|**技术代表**|• **函数调用**（最基础的同步）<br>• **HTTP REST API**（同步请求）<br>• **RPC**（如 gRPC，通常是同步的）|• **消息队列**：RabbitMQ, Kafka, RocketMQ<br>• **异步编程模型**：JavaScript 中的 Promise/async-await，Java 中的 Future<br>• **事件驱动架构**|

### 异步调用
异步调用方式其实就是基于消息通知的方式，一般包含三个角色：
- 消息发送者：投递消息的人，就是原来的调用方
- 消息代理：管理、暂存、转发消息，你可以把它理解成微信服务器
- 消息接收者：接收和处理消息的人，就是原来的服务提供方

异步调用的优势：
- 耦合度低，扩展性强
- 异步调用，无需等待，性能好
- 故障隔离，下游服务故障不影响上游业务
- 缓存消息，流量削峰填谷
异步调用的问题：
- 不能立即得到调用的结果，时效性差
- 不确定下游业务执行是否成功
- 业务安全依赖于Broker的可靠性

## MQ
**MQ**（MessageQueue，消息队列），字面来看就是存放消息的队列，也就是异步调用中的Broker。

**消息发布接收流程**：
1. 发送消息
2. 生产者和Broker建立TCP连接。
3. 生产者和Broker建立通道。
4. 生产者通过通道消息发送给Broker，由Exchange将消息进行转发。
5. Exchange将消息转发到指定的Queue（队列）

**消息接收消息**：
1. 消费者和Broker建立TCP连接
2. 消费者和Broker建立通道
3. 消费者监听指定的Queue（队列）
4. 当有消息到达Queue时Broker默认将消息推送给消费者。
5. 消费者接收到消息。

```Go
// send.go
package main  
  
import (  
    "context"  
    amqp "github.com/rabbitmq/amqp091-go"  
    "log"    
    7"time")  
  
func failOnError(err error, msg string) {  
    // 检查每个amqp调用的返回值  
    if err != nil {  
       log.Fatalf("%s: %s", msg, err)  
    }  
}  
  
func main() {  
    // 连接到RabbitMQ服务器  
    conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")  
    failOnError(err, "Failed to connect to RabbitMQ")  
    defer conn.Close()  
  
    // 创建一个通道  
    ch, err := conn.Channel()  
    failOnError(err, "Failed to open a channel")  
    defer ch.Close()  
  
    // 声明一个队列供我们发送消息  
    // 只有在队列不存在时才会创建它  
    q, err := ch.QueueDeclare(  
       "hello", // name，要声明的队列的名称。如果队列不存在，服务器会根据其他参数创建它；如果已经存在，则检查参数是否匹配
       false,   // durable，表示队列是否持久化。如果为true，队列会在服务器重启后仍然存在。如果为false，队列则是在内存中的，重启后丢失  
       false,   // auto-delete，当最后一个消费者断开连接时，队列会自动被删除。如果为false，队列会一直存在，直到被显式删除  
       false,   // exclusive，如果为true，队列是独占的，只允许当前声明它的连接使用。其他连接无法访问该队列，且当该连接关闭时，队列会自动删除，常用于临时队列  
       false,   // no-wait，如果为true，服务器不会发送响应确认队列声明是否成功；如果为false，客户端会等待服务器的确认响应。
       nil,     // arguments，可选参数，用于传递额外的队列属性 
    )  
    failOnError(err, "Failed to declare a queue")  
  
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)  
    defer cancel()  
  
    body := "Hello World!"  
    err = ch.PublishWithContext(ctx,  
       "",     // exchange  
       q.Name, // routing key  
       false,  // mandatory  
       false,  // immediate  
       amqp.Publishing{  
          ContentType: "text/plain",  
          Body:        []byte(body),  
       })  
    failOnError(err, "Failed to publish a message")  
    log.Printf(" [x] Sent %s", body)  
}
```

#### 工作队列
工作队列（又名： _任务队列_ ）背后的主要思想是避免立即执行资源密集型任务，并且必须等待 it 完成。相反，我们将任务安排在稍后完成。我们将一个封装*任务*作为消息并将其发送到队列。在后台运行的工作进程将弹出任务并最终执行作业。当您运行许多工作进程时，任务将在它们之间共享。

利用`time.Sleep` 函数。我们将字符串中的点数作为其复杂度；每个点将占一秒钟的“工作”。例如，`Hello...` 将需要三秒钟。

我们将稍微修改前面示例中的 _send.go_ 代码， 允许从命令行发送任意消息。 这程序会将任务安排到我们的工作队列中，所以让我们将其命名 `new_task.go`：
```Go

       "hello", // name  
       false,   // durable  
       false,   // delete when unused  
       false,   // exclusive  
       false,   // no-wait

    )
```

##### 消息确认
执行任务可能需要几秒钟，您可能想知道如果消费者启动一个长任务并在完成之前终止会发生什么。使用我们当前的代码，一旦 RabbitMQ 向消费者传递消息，它会立即将其标记为删除。在这种情况下，如果终止工作程序，则它刚刚处理的消息将丢失。已分派给此特定工作程序但尚未处理的消息也会丢失。

但我们不想失去任何任务。如果一个 Worker 死亡，我们希望将任务交付给另一个 Worker。

为了确保消息永远不会丢失，RabbitMQ 支持 消息确认 。消费者发回一个 ==ack==（acknowledgement），以告诉 RabbitMQ 已接收、处理特定消息，并且 RabbitMQ 可以自由删除它。

如果消费者在没有发送 ack 的情况下死亡（其通道关闭、连接关闭或 TCP 连接丢失），RabbitMQ 将了解消息未完全处理，并将其重新排队。如果同时有其他消费者在线，它会迅速将其重新交付给另一个消费者。这样，即使工人偶尔会死亡，您也可以确保不会丢失任何信息。

超时（默认为 30 分钟）在使用者传递确认时强制执行。 这有助于检测从不确认交付的有缺陷（卡住）的消费者。 您可以增加此超时，如传递确认超时。

我们将通过为“auto-ack”参数设置`false` 来使用手动消息确认，然后在完成任务后使用 `d.Ack(false)` 发送来自工作程序的正确确认（这确认单个传递）。

- `d.Ack(false)`：只确认当前这一条消息
- `d.Ack(true)`：确认所有直到（包括）当前这条消息为止的、未确认的消息

```Go
msgs, err := ch.Consume(  
    q.Name, // queue  
    "",     // consumer  
    true,   // auto-ack  
    false,  // exclusive  
    false,  // no-local  
    false,  // no-wait  
    nil,    // args  
)  
failOnError(err, "Failed to register a consumer")  
  
var forever chan struct{}  
  
go func() {  
    for d := range msgs {  
       log.Printf("Received a message: %s", d.Body)  
       dotCount := bytes.Count(d.Body, []byte("."))  
       t := time.Duration(dotCount)  
       time.Sleep(t * time.Second)  
       log.Printf("Done")  
       d.Ack(false)  
    }  
}()
```

##### 消息持久性
当 RabbitMQ 退出或崩溃时，它会忘记队列和消息，除非你告诉它不要这样做。要确保消息不会丢失，需要做两件事：我们需要将队列和消息都标记为持久。

首先，我们需要确保队列在 RabbitMQ 节点重新启动后仍然存在。为此，**我们需要将其声明为持久**：
```Go
q, err := ch.QueueDeclare(  
    "task_queue", // name  
    true,   // durable，声明为持久化  
    false,   // delete when unused  
    false,   // exclusive  
    false,   // no-wait  
    nil,     // arguments  
)
```
==RabbitMQ 不允许您重新定义现有队列 使用不同的参数，并且会向任何程序返回错误 试图做到这一点。==
==这种`持久`选项更改需要同时应用于生产者代码和使用者代码。==

将消息标记为持久性：通过使用`amqp.Publishing`中`amqp.Persistent`选项
```Go
err = ch.PublishWithContext(ctx,  
	"", // exchange  
	q.Name, // routing key  
	false, // mandatory  
	false,  
	amqp.Publishing {  
		DeliveryMode: amqp.Persistent,  
		ContentType: "text/plain",  
		Body: []byte(body),  
})
```
将消息标记为持久性并不能完全保证消息不会丢失。虽然它告诉 RabbitMQ 将消息保存到磁盘，但 RabbitMQ 已经接受消息并且尚未保存它时，仍然有一个很短的时间窗口。此外，RabbitMQ不会对每个 message -- 它可能只是保存到缓存中，而不是真正写入 磁盘。持久性保证并不强，但绰绰有余 用于我们的简单任务队列。如果您需要更强的保证，那么您可以使用 [出版商确认](https://www.rabbitmq.com/docs/confirms) 。

##### 公平调度
您可能已经注意到，调度仍然没有完全按照我们的要求进行。例如，在有两个工作人员的情况下，当所有奇数消息都很重，偶数消息很轻时，一个工作人员将一直很忙，而另一个工作人员几乎不做任何工作。好吧，RabbitMQ 对此一无所知，仍然会均匀地调度消息。

发生这种情况是因为 RabbitMQ 只是在消息进入队列时分派消息。它不会查看消费者未确认的消息数量。它只是盲目地将每 n 条消息发送给第 n 个消费者。

为了打破这一点，我们可以将预取计数设置为 `1`。这告诉 RabbitMQ 一次不要向 Worker 提供多条消息。或者，换句话说，在处理并确认前一条消息之前，不要将新消息分派给工作程序。相反，它会将其分派给下一个不忙的工作人员。
```Go
err = ch.Qos(
	1, // prefetch count
	0, // prefetch size
	false, // global
)
failOnError(err, "Failed to set Qos")
```

#### Publish/Subscribe
- `publish/subscribe`：向多个消费者传递一条消息

为了说明这种模式，我们将构建一个简单的日志记录系统。它将由两个程序组成：
- 第一个将发出日志消息
- 第二个将接收和打印它们
在我们的日志记录系统中，接收程序的每个正在运行的副本都会收到消息。这样，我们将能够运行一个接收器并将日志定向到磁盘；同时，我们将能够运行另一个接收器并在屏幕上查看日志。
从本质上讲，已发布的日志消息将广播到所有接收者。

RabbitMQ 中消息传递模型的核心思想是==生产者从不直接向队列发送任何消息==。实际上，很多时候，生产者甚至不知道消息是否会被传递到任何队列。
相反，生产者只能向*exchange*发送消息。一次交换是一件非常简单的事情。一方面，它接收来自 生产者，另一方面则将他们推到队列中。交换机必须确切地知道如何处理收到的消息。是否应该被附加到特定队列？是否应该将其附加到许多队列中？ 或者它应该被丢弃。规则由 *exchange type*定义 。

有几种可用的交换类型：`direcct`、`topic`、`headers`、`fanout`。我们将重点关注于最后一个：`fanout`。我们将创建一个这种类型的交换机：
```Go
err = ch.ExchangeDeclare(
	"logs", // name
	"fanout", // type
	true, // durable
	false, // auto-deleted
	false, // internal
	false, // no-wait
	nil, // arguments
)
```
`fanout`：将它收到的所有消息广播到它知道的所有队列。

要列出服务器上的交换机，我们可以使用`rabbitmqctl`：
```Go
sudo rabbitmqctl list_exchanges
```
在此列表中，将有一些 `amq.*` 交换和默认（未命名）交换。这些是默认创建的，但您目前不太可能需要使用它们。
在前面的示例中，我们使用的是默认交换，该交换由空字符串 `""` 标识。

##### 临时队列
想要在生产者和消费者之间共享队列时，为队列命名是非常重要的。

首先，每当我们连接到 Rabbit 时，我们都需要一个新的空队列。为此，我们可以创建一个随机名称的队列，或者更好的是，让服务器为我们选择一个随机队列名称。

其次，一旦我们断开消费者的连接，队列就应该自动删除。

在 amqp 客户端中，当我们以空字符串形式提供队列名称时，我们会创建一个具有生成名称的非持久队列：
```Go
q, err := ch.QueueDeclare(
	"", // name
	false, // durable
	false, // delete when unused
	true, // exclusive
	false, // no-wait
	nil, // arguments
)
```
当方法返回时，队列实例包含 RabbitMQ 生成的随机队列名称。例如，它可能看起来像 `amq.gen-JzTY20BRgKO-HjmUJj0wLg` 。

当声明它的连接关闭时，队列将被删除，因为它被声明为独占（exclusive）。

#### Routing 路由
前面，我们构建了一个简单的日志记录系统。我们能够向许多接收器广播日志消息。
在这个部分中，我们将向其添加一个功能：我们将使进订阅消息的子集成为可能。例如，我们将能够仅将严重的错误消息定向到日志文件（节省磁盘空间），同时仍然能够在控制台上打印所有日志消息。

##### Bindings
绑定是交换机和队列之间的关系。这可以简单地理解为：队列对来自此交换机的消息感兴趣。

绑定可以采用额外的 `routing_key` 参数。为了避免与 `Channel.Publish` 参数混淆，我们将其称为 `binding key` 。这就是我们如何创建带有键的绑定：

##### Direct exchange
我们原来使用的是`fanout`交换机，这并没有给我们太多灵活性：它只能进行无意识的广播。

我们现在使用`direct`交换机，`direct`交换机背后的路由算法很简单：消息进入`binding key`和消息的`routing key`完全匹配的队列
![[images/Pasted image 20250907174206.png]]
在此设置中，我们可以看到`direct`交换机`X`绑定了两个队列。第一个队列绑定了绑定键`orange`，第二个队列有两个绑定，绑定键分别为`black`、`green`。

在这样的设置中，使用`routing_key`为`orange`的消息将到达队列Q1。`routing_key`为`black`或`green`的消息将进入Q2。所有其他消息都将被丢弃。

##### Multiple bindings
![[images/Pasted image 20250907181129.png]]
使用相同的绑定键绑定多个队列是完全合法的。在我们的示例中，我们可以在 `X` 和 `Q1` 之间添加绑定键 `black`。在这种情况下， `direct`交换的行为类似于`fanout` ，并将消息广播到所有匹配的队列。路由键为 `black` 的消息将传递到两者 `Q1` 和`Q2`。

#### Topic
在我们的日志系统中，我们不仅要订阅日志基于严重性，也要基于发出日志的源。

##### Topic exchange
发送到`topic`交换的邮件不能有任意`routing_key`：它必须是一个单词列表，用点分隔。这些单词可以是任何东西，但通常它们指定一些与消息有关的特征。一些有效的routing key示例：`stock.usd.nyse`、`nyse.vmw`、`quick.orange.rabbit`。routing key中的字数可以任意数量，最多限制为255字节。

binding key也必须采用相同的方式。`topic`交换的背后逻辑类似于`direct`交换：使用特定routing key发送的消息将传递到使用匹配binding key的所有队列。但是binding key有两个重要的特殊情况：
- `*`只能替换一个单词
- `#`可以替换零个或多个单词

![[images/Pasted image 20250908091026.png]]
在此示例中，我们将发送消息，这些消息都描述了动物。消息将使用routing key发送，routing key包括三个词（两个点）。routing key中的第一个单词将描述速度，第二个是颜色，第三个是物种：`<speed>.<colour>.<species>`
我们创建了三个绑定：Q1与binding key`*.orange.`，Q2与`*.*.rabbit`和`lazy.#`一起使用。
这些绑定可以概括为：
- Q1对所有橙色动物感兴趣
- Q2想知道关于兔子的一切消息，以及关于懒惰动物的一切

路由键设置为 `quick.orange.rabbit` 的邮件将传送到两个队列。消息 `lazy.orange.elephant` 也会转到它们。另一方面 `quick.orange.fox` 只会进入第一个队列，并且 `lazy.brown.fox` 仅到第二个。`lazy.pink.rabbit` 将 仅传递到第二个队列一次，即使它与两个绑定匹配。`quick.brown.fox` 与任何绑定都不匹配，因此它将被丢弃。

假如违反规定并发送一条包含一个或四个词的消息，例如`orange`或`quick.orange.new.rabbit`，这些消息将与任何绑定都不匹配并会丢失。

另一方面，`lazy.orange.new.rabbit`即使有四个词，也会匹配最后一个绑定，并将消息传递到第二个队列。

当队列与`#`（哈希）绑定键绑定时：它将接收所有消息，无论路由键如何 - 就像`fanout`交换中一样。

#### RPC（Remote procedure call）
在前面，我们学习了如何使用工作队列在多个工作线程之间分配耗时的任务。
但是，当我们需要在远程计算机上运行一个函数并等待其结果该怎么办？这中模式就成为**RPC**。

我们将使用RabbitMQ构建一个RPC系统：客户端与可扩展的RPC服务器。由于我们没有任何值得分发的耗时任务，所以我们将创建一个返回斐波那契数列的虚拟RPC服务。

- 确保哪个函数调用是本地的，哪个是远程的。
- 记录你的系统。明确组件之间的依赖关系。
- 处理错误情况。当RPC服务器长时间宕机时，客户端该如何进行反应。

##### Callback queue
RabbitMQ中的`request-reply`模式涉及服务器和客户端之间的直接交互。
- 客户端发送请求消息，服务器回复响应消息。

为了接收响应，我们需要在请求中发送一个*回调队列*名称。这样的队列通常是服务器命名的，但也可以有一个众所周知的名称（客户端命名）。
然后服务器使用该名称使用默认交换进行响应。


###### Message properties
AMQP 0-9-1 协议预定义了一组与消息相关的 14 个属性。大多数属性很少使用，但以下属性除外：
- `persistent`：将消息标记为持久性（值为 `true`）或瞬态性（`false`）。
- `ontent_type`：用于描述编码的 MIME 类型。例如，对于常用的 JSON 编码，最好将此属性设置为：`application/json`。
- `reply_to`：常用于命名回调队列。
- `correlation_id`：有助于将 RPC 响应与请求相关联。

###### Correlation Id
为每个 RPC 请求创建回调队列效率低下。更好的方法是为每个客户端创建一个回调队列。

这引发了一个新问题，在该队列中收到响应后，它不清楚响应属于哪个请求。这时便可以使用 `correlation_id` 属性。我们将其设置为唯一的每个请求的值。稍后，当我们在回调队列，我们将查看此属性，并在此基础上能够将响应与请求进行匹配。如果我们看到一个未知的 `correlation_id` 价值，我们可以安全地丢弃该消息：它不属于我们的请求。

##### 工作流程
![[images/Pasted image 20250908100826.png]]
- 当客户端启动时，它会创建一个独占回调队列
- 对于RPC请求，客户端发送具有两个属性的消息：`reply_to`，设置为回调队列；`correlation_id`，设置为每个请求的唯一值。
- 请求将发送到`rpc_queue`队列
- RPC工作程序（服务器）正在等待该队列上的请求。当请求出现时，它会执行该工作，并使用`reply_to`字段中的队列将包含结果的消息发送回客户端
- 客户端等待回调队列上的数据。出现消息时，它会检查`correlation_id`属性。如果它与请求中的值匹配，则会将响应返回给应用程序。

```Go
// rpc_server.go
package main  
  
import (  
    "context"  
    "log"    
    "strconv"    
    "time"
      
    amqp "github.com/rabbitmq/amqp091-go"  
)  
  
func failOnError(err error, msg string) {  
    if err != nil {  
       log.Panicf("%s: %s", msg, err)  
    }  
}  
  
func fib(n int) int {  
    if n == 0 {  
       return 0  
    } else if n == 1 {  
       return 1  
    } else {  
       return fib(n-1) + fib(n-2)  
    }  
}  
  
func main() {  
    conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")  
    failOnError(err, "Failed to connect to RabbitMQ")  
    defer conn.Close()  
  
    ch, err := conn.Channel()  
    failOnError(err, "Failed to open a channel")  
    defer ch.Close()  
  
    q, err := ch.QueueDeclare(  
       "rpc_queue", // name  
       false,       // durable  
       false,       // delete when unused  
       false,       // exclusive  
       false,       // no-wait  
       nil,         // arguments  
    )  
    failOnError(err, "Failed to declare a queue")  
  
    err = ch.Qos(  
       1,     // prefetch count  
       0,     // prefetch size  
       false, // global  
    )  
    failOnError(err, "Failed to set QoS")  
  
    msgs, err := ch.Consume(  
       q.Name, // queue  
       "",     // consumer  
       false,  // auto-ack  
       false,  // exclusive  
       false,  // no-local  
       false,  // no-wait  
       nil,    // args  
    )  
    failOnError(err, "Failed to register a consumer")  
  
    var forever chan struct{}  
  
    go func() {  
       ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)  
       defer cancel()  
       for d := range msgs {  
          n, err := strconv.Atoi(string(d.Body))  
          failOnError(err, "Failed to convert body to integer")  
  
          log.Printf(" [.] fib(%d)", n)  
          response := fib(n)  
  
          err = ch.PublishWithContext(ctx,  
             "",        // exchange  
             d.ReplyTo, // routing key  
             false,     // mandatory  
             false,     // immediate  
             amqp.Publishing{  
                ContentType:   "text/plain",  
                CorrelationId: d.CorrelationId,  
                Body:          []byte(strconv.Itoa(response)),  
             })  
          failOnError(err, "Failed to publish a message")  
  
          d.Ack(false)  
       }  
    }()  
  
    log.Printf(" [*] Awaiting RPC requests")  
    <-forever  
}
```
