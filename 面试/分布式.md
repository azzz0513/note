## cap理论
指的是一个分布式系统中，Consistency（一致性）、Availability（可用性）、Partition tolerance（分区容错性），三者不可兼得
![[images/Pasted image 20251130232425.png]]
#### 1. C - Consistency (一致性)
- **定义**：**强一致性**（Linearizability）。即使系统有多个节点，对于客户端来说，系统就像只有这一个节点一样。
- **具体表现**：一旦数据更新操作成功，**所有**节点在同一时间的数据必须完全一致。
- **场景**：你往节点 A 写入了 x=100，紧接着去节点 B 读取 x，必须立刻读到 100。如果 B 还没同步到，它应该阻塞或报错，决不能返回旧数据。
#### 2. A - Availability (可用性)
- **定义**：**服务必须一直可用**。
- **具体表现**：收到用户的请求，服务器必须给出非错误的响应（不会超时、不会报错）。
- **注意**：可用性**不保证**你读到的数据是最新的，但保证你一定能读到数据（哪怕是旧的）。
#### 3. P - Partition Tolerance (分区容错性)
- **定义**：**网络分区容忍**。
- **具体表现**：分布式系统中的节点分布在不同的网络中，如果网络发生故障（如光缆被挖断、路由器故障），导致节点之间无法通信（网络被切分成孤岛），系统依然要能继续工作。
- **现实**：在分布式系统中，网络故障是不可避免的，因此 **P 是必须具备的**。

#### 为什么只能三选二
假设有两个节点：**Node 1** 和 **Node 2**，它们之间通过网络同步数据。

**场景发生**：现在的网络断了（发生了 **P**，分区）。
此时，客户端向 Node 1 发送写请求 set a = 1。
- Node 1 更新了自己的数据。
- 但因为网络断了，Node 1 无法将数据同步给 Node 2。
**现在，客户端向 Node 2 读取 a：**
1. **选择 CP（放弃可用性 A）**：
    - 为了保证一致性（C），Node 2 知道自己可能数据过期了（联系不上 Node 1），所以它**必须拒绝回答**或**一直等待**直到网络恢复。
    - **结果**：请求失败/超时 -> 系统不可用。
2. **选择 AP（放弃一致性 C）**：
    - 为了保证可用性（A），Node 2 **必须立刻返回**它当前持有的数据（可能是旧值 a=0）。
    - **结果**：请求成功，但读到了脏数据 -> 数据不一致。


## 分布式场景
### 常见的限流算法
#### 固定窗口限流算法
固定窗口限流算法就是对一段固定时间窗口内的请求进行计数，如果请求数超过了阈值，则舍弃该请求；如果没有达到设定的阈值，则接受该请求，且计数加1。当时间窗口结束时，重置计数器为0。
![img](https://cdn.xiaolincoding.com//picgo/1714660128964-c3acea75-3a80-427b-aafa-8d75d2672532.png)

固定窗口限流优点是实现简单，但是会有“流量吐刺”的问题，假设窗口大小为1s，限流大小为100，然后恰好在某个窗口的第999ms来了100个请求，窗口前期没有请求，所以这100个请求都会通过。再恰好，下一个窗口的第1ms有来了100个请求，也全部通过了，那也就是在2ms之内通过了200个请求，而我们设定的阈值是100，通过的请求达到了阈值的两倍，这样可能会给系统造成巨大的负载压力。
![img](https://cdn.xiaolincoding.com//picgo/1714660725465-d851112c-47c6-4fa1-a383-8433555fbefb.png)

#### 滑动窗口限流算法
改进固定窗口缺陷的方法是采用滑动窗口限流算法，滑动窗口就是将限流窗口内部切分成一些更小的时间片，然后在时间轴上滑动，每次滑动，滑过一个小时间片，就形成一个新的限流窗口，即滑动窗口。然后在这个滑动窗口内执行固定窗口算法即可。

滑动窗口可以避免固定窗口出现的放过两倍请求的问题，因为一个短时间内出现的所有请求必然在一个滑动窗口内，所以一定会被滑动窗口限流。
![img](https://cdn.xiaolincoding.com//picgo/1714660846185-84b51249-3360-43d1-ab85-682094db87fc.png)

#### 漏桶限流算法
漏桶限流算法是模拟水流过一个有漏洞的桶进而限流的思路
水龙头的水先流入漏桶，再通过漏桶底部的孔流出。如果流入的水量太大，底部的孔来不及流出，就会导致水桶太满溢出去。

从系统的角度来看，我们不知道什么时候会有请求来，也不知道请求会以多大的速率来，这就给系统的安全性埋下了隐患。但是如果加了一层漏斗算法限流之后，就能够保证请求以恒定的速率流出。在系统看来，请求永远是以平滑的传输速率过来，从而起到了保护系统的作用。

使用漏桶限流算法，缺点有两个：
- 即使系统资源很空闲，多个请求同时到达时，漏桶也是慢慢地一个接一个地去处理请求，这其实并不符合人们的期望，因为这样就是在浪费计算资源。
- 不能解决流量突发的问题，假设漏斗速率是2个/秒，然后突然来了10个请求，受限于漏斗的容量，只有5个请求被接受，另外5个被拒绝。你可能会说，漏斗速率是2个/秒，然后瞬间接受了5个请求，这不就解决了流量突发的问题吗？不，这5个请求只是被接受了，但是没有马上被处理，处理的速度仍然是我们设定的2个/秒，所以没有解决流量突发的问题

#### 令牌桶限流算法
令牌桶是另一种桶限流算法，模拟一个特定大小的桶，然后向桶中以特定的速度放入令牌（token），请求到达后，必须从桶中取出一个令牌才能继续处理。如果桶中已经没有令牌了，那么当前请求就被限流。如果桶中的令牌放满了，令牌桶也会溢出。

放令牌的动作是持续不断进行的，如果桶中令牌数达到上限，则丢弃令牌，因此桶中可能一直持有大量的可用令牌。此时请求进来可以直接拿到令牌执行。比如设置 qps 为 100，那么限流器初始化完成 1 秒后，桶中就已经有 100 个令牌了，如果此前还没有请求过来，这时突然来了 100 个请求，该限流器可以抵挡瞬时的 100 个请求。由此可见，只有桶中没有令牌时，请求才会进行等待，最终表现的效果即为以一定的速率执行。令牌桶的示意图如下：
![img](https://cdn.xiaolincoding.com//picgo/1714661434275-d83c23d2-26c7-4415-af6a-5a71d9ea0e40.png)

令牌桶限流算法综合效果比较好，能在最大程度利用系统资源处理请求的基础上，实现限流的目标，建议通常场景中优先使用该算法。


## 微服务架构
### 服务拆分的原则是什么
服务拆分没有绝对的原则，通常遵循“高内聚，低耦合”的核心思想，并结合DDD（领域驱动设计）的方法论
#### 单一职责原则
每个微服务应该**只做一件事**，并且把这件事做好。一个服务应该仅针对一个特定的业务功能或业务领域负责。
- **示例**：订单服务只处理下单、改单逻辑，不应该包含支付逻辑或用户积分逻辑。

#### 领域驱动设计（DDD）与限界上下文（bounded Context）
这是目前最主流的拆分依据。根据业务领域划分边界，一个“限界上下文”通常对应一个微服务。
- **做法**：识别核心域（Core Domain）、支撑域和通用域。确保领域模型（Entity, Value Object）在同一个上下文内是统一的。
- **判定**：如果修改一个业务逻辑需要同时修改服务A和服务B，说明这两个服务耦合过重，拆分可能不合理。

#### 数据独立原则
**这是微服务的底线**。每个服务必须拥有自己独立的数据库（Schema）。
- **禁止**：严禁多个服务共享同一个数据库表，否则会导致数据库层面的紧耦合，无法独立演进。
- **通信**：如果服务A需要服务B的数据，必须通过API调用，而不是直接查服务B的表。

#### 服务粒度适中
- **过大**：变成了“分布式单体”，部署慢、代码冲突多。
- **过小**：导致“纳米服务”，服务间调用链路过长，网络延迟（Latency）极高，且运维成本爆炸。
- **建议**：初期可以稍大一些，随着业务发展逐步拆分（演进式架构）。

#### 康威定律
“系统架构是公司组织架构的反映”。
- **原则**：服务拆分应与团队结构相匹配。通常一个微服务由一个 2-Pizza Team（6-10人）全权负责（开发、测试、运维）。

### 高并发场景下的挑战及解决方案
#### 服务雪崩
**现象**：在高并发下，如果下游服务（如商品服务）响应变慢或宕机，上游服务（如订单服务）的线程会被阻塞，导致资源耗尽，最终整个系统像雪崩一样全部瘫痪。
- **解决方案**：
    - **熔断 (Circuit Breaking)**：使用 Hystrix 或 Sentinel。当检测到下游错误率达到阈值（如50%）时，直接切断调用，快速失败，防止主线程阻塞。
    - **降级 (Degradation)**：当非核心业务（如评论、推荐）不可用时，返回默认值或空数据，保全核心业务（如下单、支付）。
    - **超时设置 (Timeouts)**：严格设置所有 RPC 调用的超时时间，避免无限等待。

#### 流量激增导致系统过载
**现象**：瞬间流量（TPS）超过了系统的处理能力，导致数据库连接池被打满，CPU 100%，系统无响应。
- **解决方案**：
    - **限流 (Rate Limiting)**：在网关层（Nginx/Kong）或应用层（Sentinel/Redis）限制请求速率
        - 算法：令牌桶（Token Bucket）或漏桶（Leaky Bucket）算法。
    - **削峰填谷 (Message Queue)**：使用 **Kafka** 或 **RabbitMQ**。将同步的请求（如用户下单）转为异步消息发送到队列。后端服务按自己的处理能力从队列消费。
    - **多级缓存 (Multi-level Caching)**：
        - **本地缓存** (Guava/Caffeine)：抗住最热点的请求。
        - **分布式缓存** (Redis)：抗住大部分读流量。
        - **原则**：让流量尽可能在到达数据库之前被拦截。


#### 分布式数据一致性
**现象**：在微服务中，一个下单操作可能涉及：订单服务写库 -> 扣减库存服务 -> 扣减积分服务。由于不能使用本地事务，一旦某一步失败，会导致数据不一致（如钱扣了，货没发）。
- **解决方案**：
    - **放弃强一致性，追求最终一致性 (Eventual Consistency)**。
    - **事务消息**：使用 RocketMQ 的事务消息，确保“本地事务执行”与“消息发送”的原子性。
    - **Saga 模式**：将长事务拆分为多个本地事务，并提供“补偿接口”（Compensating Transaction）。如果中间失败，依次调用补偿接口回滚。
    - **Seata**：使用 AT 或 TCC 模式处理分布式事务（但在极高并发下，尽量避免使用 AT 模式，因为会有全局锁性能损耗，推荐基于 MQ 的最终一致性）。


#### 网络延迟与通信开销
**现象**：微服务将内部方法调用变成了 RPC 网络调用。在高并发下，网络拥塞、序列化/反序列化开销会导致响应时间（RT）显著增加。
- **解决方案**：
    - **I/O 模型优化**：使用 Netty 等 NIO 框架，或 Go 语言的 Goroutine 模型，实现非阻塞 I/O。
    - **通信协议优化**：内部调用使用 **gRPC (Protobuf)** 替代 RESTful (JSON)。Protobuf 体积更小，解析更快。
    - **批量请求 (Batching)**：将多个小的查询请求合并为一个批量请求，减少网络往返次数 (RTT)。


#### 故障排查困难
**现象**：一个请求经过了10个服务，中间报错了，运维很难瞬间定位是哪个服务的问题。
- **解决方案**：
    - **全链路追踪 (Distributed Tracing)**：部署 **SkyWalking**、**Jaeger** 或 **Zipkin**。为每个请求生成唯一的 TraceID，贯穿所有服务，可视化展示调用链路和耗时瓶颈。
