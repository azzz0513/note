### Linux中的线程和进程是怎么查看的
#### 1. 使用 ps 命令（静态快照）
ps 是最常用的查看进程状态的命令。
- **查看进程：**
    - ps aux：查看系统中所有的进程（BSD语法格式）。
    - ps -ef：查看系统中所有的进程（标准语法格式）。
- **查看线程：**
    - `ps -T -p <PID>`：查看指定进程的线程。**-T** 参数会开启线程查看，输出中会多出一列 **SPID**（或 TID），代表线程ID。
    - ps -eLf：查看系统中所有线程。输出中会多出一列 **LWP**（Lightweight Process），代表线程ID。NLWP列代表该进程有多少个线程。
    - 示例：ps -T -p 1234 （查看PID为1234的进程下的所有线程）

#### 2. 使用 top 命令（动态实时监控）
top 命令用于实时查看系统的资源使用情况。
- **查看进程：**
    - 直接输入 top，默认显示的就是各个**进程**的资源占用情况。
- **查看线程：**
    - **方法一（启动时指定）：** 输入 top -H，直接进入线程视图模式；或者 `top -H -p <PID>` 只监控某个进程下的线程。
    - **方法二（运行中切换）：** 在 top 界面运行期间，按下键盘上的大写字母 **H**，可以在“进程视图”和“线程视图”之间来回切换。切换到线程视图时，Tasks 统计行会变成 Threads。

#### 3. 使用 htop 命令（更直观的交互式工具）
htop 是 top 的增强版，支持鼠标点击，色彩丰富（如果没有安装，可通过 yum install htop 或 apt install htop 安装）。
- **查看进程：** 直接输入 htop。
- **查看线程：**
    - 在 htop 界面中，按下大写字母 **H** 可以显示/隐藏用户线程，按下大写字母 **K** 可以显示/隐藏内核线程。
    - 按下 **F5**（或小写字母 t）可以开启**树状视图**（Tree View），能够非常清晰地看到父进程、子进程以及挂载在进程下的线程结构（线程通常以绿色显示）。

#### 4. 使用 pstree 命令（树状图显示）
pstree 以树状图的方式展现进程之间的派生关系。
- **查看进程：**
    - pstree：显示进程树。
    - pstree -p：显示进程树，并同时显示每个进程的PID。
- **查看线程：**
    - `pstree -p <PID>`：查看指定进程的树状结构，默认会将线程包含在内。
    - `pstree -t -p <PID>`：明确展开该进程下的所有线程（在某些版本中，如果同名线程被折叠，加上 -t 可以展开）。在花括号 {} 中的通常是线程。

#### 5. 查看 /proc 虚拟文件系统（最底层的数据来源）
Linux 的所有进程和线程信息都以文件的形式存储在 /proc 目录下。
- **查看进程：**
    - 进程的信息存储在 `/proc/<PID>/` 目录下。
    - 你可以通过 `cat /proc/<PID>/status` 来查看某个进程的详细状态，里面有一行 Threads: 显示了该进程拥有的线程数量。
- **查看线程：**
    - 进程下的具体线程信息存储在 **`/proc/<PID>/task/`** 目录下。
    - 输入 `ls /proc/<PID>/task/`，你会看到一堆以数字命名的文件夹，这些数字就是该进程下各个**线程的TID（线程ID）**。

### Linux操作系统是基于什么数据结构来管理文件的
Linux 使用 inode、dentry、file、super_block 等内核数据结构来管理文件。用户通过文件描述符访问文件，文件描述符本质上是进程文件描述符表中的索引，最终指向内核中的 file 结构，再关联 inode 和磁盘数据块。

整体关系图：
```
用户程序
   ↓
文件描述符 fd（整数）
   ↓
进程文件描述符表 fd table
   ↓
struct file（打开文件对象）
   ↓
dentry（目录项）
   ↓
inode（文件元信息）
   ↓
data block（磁盘数据块）
```
#### inode
inode是什么？
inode（索引节点）：是文件的身份证
保存：文件的元信息
比如：
- 文件大小
- 文件权限（rwx）
- 创建时间
- 修改时间
- 用户 UID
- 组 GID
- 链接数
- 数据块位置（最重要）
==不保存文件名==

#### dentry（目录项）
dentry是文件名和inode的映射关系

比如：
```
a.txt → inode 123456
```
所以：文件名保存在目录里，而不是保存在inode中
所以，硬链接可以有多个文件名，因为多个dentry可以指向同一个inode

#### struct file（打开文件对象）
##### file
表示“这次打开行为”，不是文件本身
而是：
```C
open()
```
产生的那个对象
它保存：
- 当前读写位置 offset
- 打开方式（读/写）
- 引用计数
- 指向 inode 的指针

为什么需要file：
因为同一个文件，可以被多个进程打开，同一个进程也可以多次打开，每次都得独立维护offset

#### 文件描述符表（fd table）
##### fd（文件描述符）
如：
```C
int fd = open(...)
```
返回：3
这个3就是文件描述符
本质上就是：一个整数下标
用于索引当前进程的fd table

例如：

|fd|指向|
|---|---|
|0|stdin|
|1|stdout|
|2|stderr|
|3|struct file|
所以：
fd → file
而不是：
fd → inode

#### super_block
表示：整个文件系统的信息
比如：
- 文件系统类型（ext4/xfs）
- 总 inode 数量
- 总块数量
- 空闲块数量
- 挂载信息

是文件系统的总管家，不是单个文件的信息，而是整个磁盘分区的信息

### 删除一个文件，rm返回成功能代表整个文件真的被删除了吗
不能。
> `rm` 返回成功，**并不一定代表文件内容已经立刻从磁盘上彻底消失了**。

它真正表示的是：文件的目录项（dentry）被删除成功了
也就是：`文件名 → inode`这个映射被移除了，而不是马上就把磁盘数据块清空

inode和数据什么时候真正删除
必须满足两个条件：
1. 硬链接数（link count）为0：也就是没有任何文件名再指向这个inode
2. 打开引用数（open count）为0：没有线程还在使用这个文件

#### 删除了文件，为什么程序还能继续写日志
如：
```Bash
rm app.log
```
但是程序服务还能继续往这个文件里写：
因为，程序启动时已经：
```C
open("app.log")
```
拿到了：
```
fd → file → inode
```
即使`rm`，也只是删掉了名字，inode还在，所以程序继续写完全没问题

#### 为什么磁盘空间没有立刻释放
因为Linux要保证：正在使用的文件不能突然消失
否则：程序会直接异常
所以，采用延迟删除机制
```
先删名字  
再等引用归零  
最后删数据
```

#### rm成功意味着什么
意味着：路径访问失效：
```Bash
cat a.txt
```
```
No such file or directory
```
因为文件名没了

### 硬链接和软链接
硬链接本质上是多个文件名（目录项 dentry）指向同一个 inode，因此它们共享同一份文件数据；软链接本质上是一个独立的文件，拥有自己的 inode，文件内容保存的是目标文件的路径名。

硬链接：多个文件名指向同一个inode
比如：
原文件：a.txt
它的 inode：inode = 1001
执行：ln a.txt b.txt
结果：
a.txt → inode 1001  
b.txt → inode 1001
注意：两个文件名，共享同一个inode，这就是硬链接

软链接：一个独立的新文件
它有自己的inode，但它的内容是：目标文件的路径
比如：ln -s a.txt c.txt
结果：
c.txt 自己有 inode = 2001  
内容是："a.txt"
所以：软链接是“路径跳转”
不是共享 inode。

#### 删除原文件会怎么样
- 硬链接：
删除：rm a.txt
结果：b.txt 还能正常访问
因为：inode 1001
还被：b.txt
引用着。
只是：link count -1而已。
文件不会丢。

- 软链接
删除：rm a.txt
结果：c.txt 坏了（悬空链接）
因为：它保存的是：路径 a.txt
路径没了。自然失效。
这叫：dangling symlink（悬挂软链接）

#### 为什么硬链接不能跨文件系统？
因为： inode 只在当前文件系统唯一
比如：ext4 分区 A 的 inode：1001
到了另一个磁盘：
根本没有意义。
所以：硬链接不能跨文件系统
而软链接保存的是：路径字符串
所以：可以跨文件系统
甚至：可以链接目录
这也是区别。

#### 为什么硬链接通常不能链接目录
会导致：文件系统环
比如：
A → B  
B → A
这样：
find  
ls -R
就会无限递归，整个文件系统会乱掉。
所以，普通用户几乎不能对目录创建硬链接
这是设计限制。
