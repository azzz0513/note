## 核心作用
etcd是分布式系统的“大脑”和“信源”
从技术定义上说，etcd 是一个**高可用、强一致性的分布式键值（Key-Value）存储系统**。它也是用 Go 语言编写的，是 CNCF（云原生计算基金会）的顶级项目。
你可能会问：“Redis 也是 KV 存储，为什么要用 etcd？” 核心区别在于：**Redis 追求极速（高性能），而 etcd 追求绝对的稳和准（强一致性）。**

以下是etcd的四大核心作用：
**k8s的“大脑”**
这是 etcd 最著名的应用场景。可以这么说：**没有 etcd，就没有 Kubernetes。**
- **存什么？** K8s 集群中所有的状态数据（有哪些 Node、起了哪些 Pod、Service IP 是什么、ConfigMap 配置）全部都存在 etcd 里。
- **作用：** K8s 的 API Server 是无状态的，它只是个处理请求的网关，真正的数据落地全靠 etcd。
- **后果：** 如果 etcd 挂了，整个 K8s 集群就“脑死亡”了，虽然现有的容器可能还在跑，但你无法发布新应用，也无法扩缩容，甚至无法读取集群信息。

**服务注册与发现**
在微服务架构中，etcd 扮演着“通讯录”的角色。
- **注册（写）：** 当服务 A 启动时，它会向 etcd 写入自己的 IP 和端口（例如 /services/order-service/192.168.1.100），并设置一个**租约（Lease/TTL）**。
- **续约（心跳）：** 服务 A 每隔几秒会向 etcd 发送心跳（续租）。如果服务 A 宕机了，心跳停止，租约过期，etcd 会自动删除这条记录。
- **发现（读/Watch）：** 服务 B 想调用服务 A，它去 etcd 查一下 /services/order-service/ 下有哪些 IP，就能找到活着的实例。

**分布式配置中心**
应用在多台机器上运行时，修改配置是一件很麻烦的事。etcd 提供了强大的 **Watch（监听）机制**
- **场景：** 你有 100 台服务器运行着同一个业务，现在想把日志级别从 INFO 改为 DEBUG。 
- **做法：**
    1. 你只需要修改 etcd 中的一个 Key：config/log-level = DEBUG。
    2. 那 100 台服务器启动时都 **Watch（监听）** 了这个 Key。
    3. etcd 一旦检测到数据变化，会立刻**推送**通知给这 100 台服务器。
    4. 服务器收到通知，在内存中动态修改日志级别，**不需要重启服务**。

**分布式锁**
在分布式系统中，如何保证“同一时间只有一个节点在执行任务”（比如抢单、处理定时任务）？
etcd 基于 **Raft 算法** 保证了数据的强一致性，非常适合做分布式锁。
- **原理：**
    - 大家去 etcd 里抢着创建一个 Key（比如 lock/my-task）。
    - 利用 etcd 的 Revision（版本号）机制，谁生成的版本号最小，谁就抢到了锁。
    - 抢不到的排队监听（Watch）前一个版本号的删除事件。
- **优势：** 相比 Redis 分布式锁（Redlock），etcd 的锁在集群故障切换时更安全，不会因为主从异步复制导致“锁丢失”。
