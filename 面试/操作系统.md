# 用户态和内核态
## 用户态和内核态的区别
内核态和用户态是操作系统中的两种运行模式。它们的主要区别在于权限和可执行的操作：
- 内核态（Kernel Mode）：在内核态下，CPU可以执行所有的指令和访问所有的硬件资源。这种模式下的操作具有更高的权限，主要用于操作系统内核的运行。
- 用户态（User Mode）：在用户态下，CPU只能执行部分指令集，无法直接访问硬件资源。这种模式下的操作权限较低，主要用于运行用户程序。

内核态的底层操作主要包括：内存管理、进程管理、设备驱动程序控制、系统调用等。这些操作涉及到操作系统的核心功能，需要较高的权限来执行。

分为内核态和用户态的原因主要有以下几点：
- 安全性：通过对权限的划分，用户程序无法直接访问硬件资源，从而避免了恶意程序对系统资源的破坏。
- 稳定性：用户态程序出现问题时，不会影响到整个系统，避免了程序故障导致系统崩溃的风险。
- 隔离性：内核态和用户态的划分使得操作系统内核与用户程序之间有了明确的边界，有利于系统的模块化和维护。

内核态和用户态的划分有助于保证操作系统的安全性、稳定性和易维护性。

## 如何从用户态切换到内核态
用户态程序无法直接切换到内核态，必须通过特定的机制触发，由硬件和操作系统协同完成。主要有以下三种方式：
1. **系统调用 (System Call)**  
    这是最常见、也是用户程序**主动**发起的切换。当应用程序需要操作系统提供服务时（例如读写文件、创建进程、网络通信），就会调用一个系统调用接口（如 `read()`, `write()`）。这个调用会触发一个特殊的指令（如 `syscall`），使 CPU 从用户态切换到内核态，执行相应的内核代码来完成请求。
2. **异常 (Exception)**  
    这是由 CPU **被动**触发的切换。当用户程序在执行过程中出现错误或特殊情况时，CPU 会自动切换到内核态，由内核的异常处理程序进行处理。例如：
    - 除零错误
    - 访问了非法的内存地址（如空指针解引用）
    - 执行了特权指令（用户态程序尝试直接操作硬件）
3. **中断 (Interrupt)**  
    这也是**被动**触发的切换，但源头是外部硬件。当硬件设备完成一项操作后，会向 CPU 发送一个中断信号，通知 CPU 进行处理。例如：
    - 键盘输入完成
    - 网卡接收到新的数据包
    - 硬盘读写完成  
        CPU 收到中断信号后，会暂停当前正在执行的用户程序，切换到内核态去执行相应的中断处理程序。  
        操作系统中的内核态和用户态知多少


# 进程管理
## ==线程和进程的区别是什么==
![[images/Pasted image 20260401091502.png]]
- **本质区别**：进程是操作系统资源分配的基本单位，而线程是任务调度和执行的基本单位
- **在开销方面**：每个进程都有独立的代码和数据空间（程序上下文），程序之间的切换会有较大的开销；线程可以看做轻量级的进程，同一类线程共享代码和数据空间，每个线程都有自己独立的运行栈和程序计数器（PC），线程之间切换的开销小
- **稳定性方面**：进程中某个线程如果崩溃了，可能会导致整个进程都崩溃。而进程中的子进程崩溃，并不会影响其他进程。
- **内存分配方面**：系统在运行的时候会为每个进程分配不同的内存空间；而对线程而言，除了CPU外，系统不会为线程分配内存（线程所使用的资源来自其所属进程的资源），线程组之间只能共享资源
- **包含关系**：没有线程的进程可以看做是单线程的，如果一个进程内有多个线程，则执行过程不是一条线的，而是多条线

## 进程、线程、协程的区别是什么
- 首先，我们来谈谈**进程**。==进程是操作系统中进行资源分配和调度的基本单位，它拥有自己的独立内存空间和系统资源。每个进程都有独立的堆和栈，不与其他进程共享。==进程间通信需要通过特定的机制，如管道、消息队列、信号量等。由于进程拥有独立的内存空间，因此其稳定性和安全性相对较高，但同时上下文切换的开销也较大，因为需要保存和恢复整个进程的状态。
- 接下来是**线程**。线程是进程内的一个执行单元，也是CPU调度和分派的基本单位。与进程不同，线程共享进程的内存空间，包括堆和全局变量。线程之间通信更加高效，因为它们可以直接读写共享内存。线程的上下文切换开销较小，因为只需要保存和恢复线程的上下文，而不是整个进程的状态。然而，由于多个线程共享内存空间，因此存在数据竞争和线程安全的问题，需要通过同步和互斥机制来解决。
- 最后是**协程**。协程是一种用户态的轻量级线程，其调度完全由用户程序控制，而不需要内核的参与。==协程拥有自己的寄存器上下文和栈，但与其他协程共享堆内存。==协程的切换开销非常小，因为只需要保存和恢复协程的上下文，而无需进行内核级的上下文切换。这使得协程在处理大量并发任务时具有非常高的效率。然而，协程需要程序员显式地进行调度和管理，相对于线程和进程来说，其编程模型更为复杂。

|对比维度|进程 (Process)|线程 (Thread)|协程 (Coroutine)|
|:--|:--|:--|:--|
|基本定义|资源分配和隔离的基本单位|CPU调度的基本单位|用户态的轻量级并发单位|
|资源开销|大 (独立内存空间)|中 (共享进程内存)|极小 (共享线程内存)|
|切换开销|大 (涉及内存空间切换)|中 (内核调度)|极小 (用户态调度)|
|调度者|操作系统内核|操作系统内核|用户程序/运行时|
|隔离性|强 (进程间相互隔离)|弱 (线程间共享内存)|无 (协程间共享上下文)|
|通信方式|IPC (管道、Socket等)|共享内存+同步机制|共享变量、通道(Channel)|
关系可以类比为：
- **进程**：一个独立的**工厂**，拥有自己独立的厂房、设备和资源。
- **线程**：工厂里的**车间**，多个车间共享工厂的资源，可以并行生产。
- **协程**：车间里的**流水线**，多个流水线在同一个车间内协作，通过“主动让位”的方式高效切换任务。

### 如何选择合适的并发模型
#### step1：判断任务类型
##### 1. CPU 密集型任务 → 首选多进程
- **特征**：程序需要执行大量复杂的计算，CPU 持续高负荷运转。例如：图像处理、视频编解码、科学计算、复杂的加密解密、机器学习模型推理。
- **核心诉求**：需要**真正的并行**来利用多核 CPU 的优势，缩短计算时间。
- **为什么是多进程？**
    - **突破 GIL 限制**：在 Python 等语言中，全局解释器锁（GIL）会阻止多线程的并行计算。多进程为每个进程创建独立的解释器和内存空间，完美绕过 GIL，实现真正的多核并行。
    - **强隔离性**：一个进程崩溃不会影响到主程序或其他进程，稳定性高。

##### 2. I/O 密集型任务 → 首选协程，备选多线程
- **特征**：程序大部分时间都在等待外部操作完成，CPU 占用率很低。例如：网络请求（API调用、爬虫）、数据库读写、文件操作。
- **核心诉求**：需要高效地管理大量“等待”，在等待期间能迅速切换去处理其他任务。
- **为什么是协程？**
    - **极致的性能**：协程运行在用户态，创建和切换的开销极小（纳秒级），远小于线程（微秒级）和进程（毫秒级）。一个线程可以轻松承载数万个协程，非常适合高并发场景（如 Web 服务器、网关）。
    - **单线程高并发**：通过事件循环（Event Loop）和 `async/await` 语法，可以在单线程内高效地并发处理成千上万的 I/O 操作。
- **为什么备选多线程？**
    - **兼容性更好**：如果你的项目大量使用了不支持异步的第三方库（即“阻塞”库），强行使用协程会阻塞整个事件循环。此时，使用多线程是更务实的选择，因为线程在等待 I/O 时会被操作系统挂起，不会阻塞其他线程。

#### step2：评估资源与约束

|评估维度|考虑因素|决策影响|
|:--|:--|:--|
|系统资源|内存是否紧张？CPU 核心数多少？|内存紧张时，避免使用开销大的多进程。CPU 核心数决定了 CPU 密集型任务能并行到的上限。|
|团队能力|团队是否熟悉异步编程？|异步编程模型（协程）比多线程更复杂，需要团队有相应的技术储备，否则容易引入 Bug。|
|开发效率|是否需要快速迭代？|多线程模型虽然性能不如协程，但编程模型更传统，开发和调试可能更快。|
|生态兼容|现有库和框架的支持情况？|如果核心依赖的库不支持异步，那么协程方案可能寸步难行，多线程是更稳妥的选择。|
#### step3：组合与实战
在实际的复杂系统中，我们很少只使用一种模型，更多是**混合使用**，以发挥各自的优势。

一个典型的现代 Web 应用可能同时包含三种模型：
1. **主框架使用协程**：Web 框架（如 FastAPI, Sanic）本身是异步的，用协程高效处理海量的 HTTP 请求（I/O 密集型）。
2. **CPU 计算交给进程池**：当请求中需要进行复杂的图像处理或数据分析时，通过 `loop.run_in_executor()` 将任务提交给一个**进程池**（`ProcessPoolExecutor`）去执行，避免阻塞主事件循环。
3. **阻塞 I/O 交给线程池**：如果必须调用一个不支持异步的数据库驱动或文件库，则将其提交给**线程池**（`ThreadPoolExecutor`）执行。

## 为什么进程崩溃不会对其他进程产生很大影响
主要是因为：
- **进程隔离性**：每个进程都有自己独立的内存空间，当一个进程崩溃时，其内存空间会被操作系统回收，不会影响其他进程的内存空间。这种进程间的隔离性保证了一个进程崩溃不会直接影响其他进程的执行。
- **进程独立性**：每个进程都是独立运行的，它们之间不会共享资源，如文件、网络连接等。因此，一个进程的崩溃通常不会对其他进程的资源产生影响。

#### 内存隔离：独立的虚拟地址空间
这是最根本、最重要的隔离机制
- **私有空间**：操作系统并不会让进程直接访问物理内存，而是为每个进程分配一个独立的、私有的**虚拟地址空间**。在进程看来，它仿佛独占了整块连续的内存。
- **地址翻译**：当进程访问一个虚拟地址时，CPU中的内存管理单元（MMU）会在操作系统的帮助下，将其翻译成真实的物理内存地址。
- **隔离效果**：进程A的虚拟地址空间和进程B的完全独立。进程A的指针无论如何也无法指向进程B的内存区域。因此，如果一个进程发生内存错误（如访问非法地址、内存溢出），它只会“污染”自己的虚拟地址空间。操作系统会捕捉到这个错误，并精准地终结这个出问题的进程，回收其占用的物理内存，而这个过程对其他进程的内存空间**秋毫无犯**。

#### 资源隔离：独立的资源句柄
除了内存，操作系统分配给进程的其他资源也是相互独立的。
- **专属资源**：每个进程都拥有自己独立的资源句柄，如文件描述符、网络连接、设备句柄等。一个进程打开的文件，在它的进程表中有独立的记录，其他进程无法直接操作。
- **自动回收**：当一个进程崩溃时，操作系统会负责清理并回收该进程持有的所有资源，包括关闭它打开的文件、断开网络连接等，确保不会留下“烂摊子”影响其他进程。

#### 架构隔离：用户态与内核态分离
现代操作系统在架构上将应用程序与系统核心分离开来。
- **权限分级**：操作系统内核运行在权限最高的**内核态**，负责管理所有硬件和核心资源。而普通应用程序则运行在权限受限的**用户态**。
- **安全屏障**：用户态的程序无法直接访问硬件或修改内核数据。如果一个应用程序崩溃，它被限制在自己的用户态空间内，其“冲击波”无法穿透到内核态，因此不会干扰到操作系统内核或其他应用程序的稳定运行。

#### 为什么线程崩溃影响更大

|隔离级别|崩溃影响范围|原因|
|:--|:--|:--|
|进程|仅限自身|拥有独立的内存空间和系统资源，被“沙箱”完全隔离。|
|线程|整个进程|同一进程内的所有线程共享内存和大部分资源。一个线程崩溃（如未捕获异常）会导致整个进程随之崩溃。|

## 进程是分配资源的基本单位，那么这个资源指的是什么？
虚拟内存、文件句柄、信号量等资源。

- 把解压缩后的视频数据播放出来；

对于单进程的这种方式，存在以下问题：
- 播放出来的画面和声音会不连贯，因为当 CPU 能力不够强的时候，Read 的时候可能进程就等在这了，这样就会导致等半天才进行数据解压和播放；
- 各个函数之间不是并发执行，影响资源的使用效率；

那改进成多进程的方式：
![img](https://cdn.xiaolincoding.com//picgo/1720433762834-7790110f-4653-42fc-b06d-dea02856abac.png)

对于多进程的这种方式，依然会存在问题：
- 进程之间如何通信，共享数据？
- 维护进程的系统开销较大，如创建进程时，分配资源、建立 PCB；终止进程时，回收资源、撤销 PCB；进程切换时，保存当前进程的状态信息；

那到底如何解决呢？需要有一种新的实体，满足以下特性：
- 实体之间可以并发运行；
- 实体之间共享相同的地址空间；

这个新的实体，就是**线程(** _**Thread**_ **)**，线程之间可以并发运行且共享相同的地址空间。

## 多线程比单线程的优势、劣势？
- 多线程比单线程的**优势**：提高程序的运行效率，可以充分利用多核处理器的资源，同时处理多个任务，加快程序的执行速度。
- 多线程比单线程的**劣势**：存在多线程数据竞争访问的问题，需要通过锁机制来保证线程安全，增加了加锁的开销，并且还会有死锁的风险。多线程会消耗更多系统资源，如CPU和内存，因为每个线程都需要占用一定的内存和处理时间。

## 多线程是不是越多越好，太多会有什么问题？
多线程不一定越多越好，过多的线程可能会导致一些问题。
- 切换开销：线程的创建和切换会消耗系统资源，包括内存和CPU。如果创建太多线程，会占用大量的系统资源，导致系统负载过高，某个线程崩溃后，可能会导致进程崩溃。
- 死锁的问题：过多的线程可能会导致竞争条件和死锁。竞争条件指的是多个线程同时访问和修改共享资源，如果没有合适的同步机制，可能会导致数据不一致或错误的结果。而死锁则是指多个线程相互等待对方释放资源，导致程序无法继续执行。

## 进程切换和线程切换的区别？
1. 进程切换：进程切换涉及到更多的内容，包括整个进程的地址空间、全局变量、文件描述符等。因此，进程切换的开销通常比线程切换大。
2. 线程切换：线程切换只涉及到线程的堆栈、寄存器和程序计数器等，不涉及进程级别的资源，因此线程切换的开销较小。

## 线程切换为什么比进程切换快，节省了什么资源？
线程切换比进程切换快是因为线程共享同一进程的地址空间和资源，线程切换时只需切换堆栈和程序计数器等少量信息，而不需要切换地址空间，避免了进程切换时需要切换内存映射表等大量资源的开销，从而节省了时间和系统资源。

## 线程切换详细过程是怎么样的？上下文保存在哪里？
![image-20240725233216165](https://cdn.xiaolincoding.com//picgo/image-20240725233216165.png)

线程切换的详细过程可以分为以下几个步骤：
- 上下文保存：当操作系统决定切换到另一个线程时，它首先会保存当前线程的上下文信息。上下文信息包括寄存器状态、程序计数器、堆栈指针等，用于保存线程的执行状态。
- 切换到调度器：操作系统将执行权切换到调度器（Scheduler）。调度器负责选择下一个要执行的线程，并根据调度算法做出决策。
- 上下文恢复：调度器选择了下一个要执行的线程后，它会从该线程保存的上下文信息中恢复线程的执行状态。
- 切换到新线程：调度器将执行权切换到新线程，使其开始执行。

上下文信息的保存通常由操作系统负责管理，具体保存在哪里取决于操作系统的实现方式。一般情况下，上下文信息会保存在线程的控制块（Thread Control Block，TCB）中。

TCB是操作系统用于管理线程的数据结构，包含了线程的状态、寄存器的值、堆栈信息等。当发生线程切换时，操作系统会通过切换TCB来保存和恢复线程的上下文信息。
