## 消息队列
### ==什么是消息队列==
你可以把消息队列理解为一个**使用队列来通信**的组件。它的本质，就是个**转发器**，包含**发消息、存消息、消费消息**的过程。最简单的消息队列模型如下：
![](https://cdn.xiaolincoding.com//picgo/image-20250407141749839.png)
我们通常说的消息队列，简称**MQ（Message Queue）**，它其实就指**消息中间件**，当前业界比较流行的开源消息中间件包括：`RabbitMQ、RocketMQ、Kafka`。

### ==消息队列使用场景有哪些==
- **解耦**：可以在多个系统之间进行解耦，将原本通过网络之间的调用的方式改为使用MQ进行消息的异步通讯，只要该操作不是需要同步的，就可以改为使用MQ进行不同系统之间的联系，这样项目之间不会存在耦合，系统之间不会产生太大的影响，就算一个系统挂了，也只是消息挤压在MQ里面没人进行消费而已，不会对其他的系统产生影响。
- **异步**：加入一个操作设计到好几个步骤，这些步骤之间不需要同步完成，比如客户去创建了一个订单，还要去客户轨迹系统添加一条轨迹、去库存系统更新库存、去客户系统修改客户的状态等等。这样如果这个系统都直接进行调用，那么将会产生大量的时间，这样对于客户是无法接收的；并且像添加客户轨迹这种操作是不需要去同步操作的，如果使用MQ将客户创建订单时，将后面的轨迹、库存、状态等信息的更新全都放到MQ里面然后去异步操作，这样就可加快系统的访问速度，提供更好的客户体验。
- **削峰**：一个系统访问流量有高峰时期，也有低峰时期，比如说，中午整点有一个抢购活动等等。比如系统平时流量并不高，一秒钟只有100多个并发请求，系统处理没有任何压力，一切风平浪静，到了某个抢购活动时间，系统并发访问了剧增，比如达到了每秒5000个并发请求，而我们的系统每秒只能处理2000个请求，那么由于流量太大，我们的系统、数据库可能就会崩溃。这时如果使用MQ进行流量削峰，将用户的大量消息直接放到MQ里面，然后我们的系统去按自己的最大消费能力去消费这些消息，就可以保证系统的稳定，只是可能要跟进业务逻辑，给用户返回特定页面或者稍后通过其他方式通知其结果

### ==消息重复消费怎么解决==
生产端为了保证消息发送成功，可能会重复推送(直到收到成功ACK)，会产生重复消息。但是一个成熟的MQ Server框架一般会想办法解决，避免存储重复消息(比如：空间换时间，存储已处理过的message_id)，给生产端提供一个幂等性的发送消息接口。

但是消费端却无法根本解决这个问题，在高并发标准要求下，拉取消息+业务处理+提交消费位移需要做事务处理，另外消费端服务可能宕机，很可能会拉取到重复消息。

所以，只能业务端自己做控制，**对于已经消费成功的消息，本地数据库表或Redis缓存业务标识，每次处理前先进行校验，保证幂等。**

### ==消息丢失要怎么解决==
使用一个消息队列，其实就分为三大块：**生产者、中间件、消费者**，所以要保证消息就是保证三个环节都不能丢失数据。
![img](https://cdn.xiaolincoding.com//picgo/1719381898719-af6c00bf-8760-4639-bd21-e6d422ef7779.webp)
- **消息生产阶段**：生产者会不会丢消息，取决于生产者对于异常情况的处理是否合理。从消息被生产出来，然后提交给 MQ 的过程中，只要能正常收到 （ MQ 中间件） 的 ack 确认响应，就表示发送成功，所以只要处理好返回值和异常，如果返回异常则进行消息重发，那么这个阶段是不会出现消息丢失的。
- **消息存储阶段**：Kafka 在使用时是部署一个集群，生产者在发布消息时，队列中间件通常会写「多个节点」，也就是有多个副本，这样一来，即便其中一个节点挂了，也能保证集群的数据不丢失。
- **消息消费阶段**：消费者接收消息+消息处理之后，才回复 ack 的话，那么消息阶段的消息不会丢失。不能收到消息就回 ack，否则可能消息处理中途挂掉了，消息就丢失了。

### ==消息队列的可靠性、顺序性怎么保证==
消息可靠性可以通过下面这些方式来保证
- **消息持久化**：确保消息队列能够持久化消息是非常关键的。在系统崩溃、重启或者网络故障等情况下，未处理的消息不应丢失。例如，像 RabbitMQ 可以通过配置将消息持久化到磁盘，通过将队列和消息都设置为持久化的方式（设置`durable = true`），这样在服务器重启后，消息依然可以被重新读取和处理。
- **消息确认机制**：消费者在成功处理消息后，应该向消息队列发送确认（acknowledgment）。消息队列只有收到确认后，才会将消息从队列中移除。如果没有收到确认，消息队列可能会在一定时间后重新发送消息给其他消费者或者再次发送给同一个消费者。以 Kafka 为例，消费者通过`commitSync`或者`commitAsync`方法来提交偏移量（offset），从而确认消息的消费。
- **消息重试策略**：当消费者处理消息失败时，需要有合理的重试策略。可以设置重试次数和重试间隔时间。例如，在第一次处理失败后，等待一段时间（如 5 秒）后进行第二次重试，如果重试多次（如 3 次）后仍然失败，可以将消息发送到死信队列，以便后续人工排查或者采取其他特殊处理。

消息顺序性保证的方式如下：
- **有序消息处理场景识别**：首先需要明确业务场景中哪些消息是需要保证顺序的。例如，在金融交易系统中，对于同用户的转账操作顺序是不能打乱的。对于需要顺序处理的消息，要确保消息队列和消费者能够按照特定的顺序进行处理。
- **消息队列对顺序性的支持**：部分消息队列本身提供了顺序性保证的功能。比如 Kafka 可以通过将消息划分到同一个分区（Partition）来保证消息在分区内是有序的，消费者按照分区顺序读取消息就可以保证消息顺序。但这也可能会限制消息的并行处理程度，需要在顺序性和吞吐量之间进行权衡。
- **消费者顺序处理策略**：消费者在处理顺序消息时，应该避免并发处理可能导致顺序打乱的情况。例如，可以通过单线程或者使用线程池并对顺序消息进行串行化处理等方式，确保消息按照正确的顺序被消费。

### ==如何保证幂等写==
幂等性是指 **同一操作的多次执行对系统状态的影响与一次执行结果一致**。例如，支付接口若因网络重试被多次调用，最终应确保仅扣款一次。实现幂等写的核心方案：
- 唯一标识（幂等键）：客户端为每个请求生成全局唯一ID（如 UUID、业务主键），服务端校验该ID是否已处理，适用场景接口调用、消息消费等。
- 数据库事务 + 乐观锁：通过版本号或状态字段控制并发更新，确保多次更新等同于单次操作，适用场景数据库记录更新（如余额扣减、订单状态变更）。
- 数据库唯一约束：利用数据库唯一索引防止重复数据写入，适用场景数据插入场景（如订单创建）。
- 分布式锁：通过锁机制保证同一时刻仅有一个请求执行关键操作，适用场景高并发下的资源抢夺（如秒杀）。
- 消息去重：消息队列生产者为每条消息生成唯一的消息 ID，消费者在处理消息前，先检查该消息 ID 是否已经处理过，如果已经处理过则丢弃该消息。

### ==如何保证数据一致性，事务消息如何实现==
一条普通的MQ消息，从产生到被消费，大概流程如下：
![image-20250407142107477](https://cdn.xiaolincoding.com//picgo/image-20250407142107477.png)
1. 生产者产生消息，发送到MQ服务器
2. MQ收到消息后，将消息持久化到存储系统。
3. MQ服务器返回ACK到生产者。
4. MQ服务器把消息push给消费者
5. 消费者消费完消息，响应ACK
6. MQ服务器收到ACK，认为消息消费成功，即在存储中删除消息。

如何保证数据一致性呢？可以使用**事务消息**。一起来看下事务消息是如何实现的吧。
![image-20250407142122992](https://cdn.xiaolincoding.com//picgo/image-20250407142122992.png)
1. 生产者产生消息，发送一条**半事务消息**到MQ服务器
2. MQ收到消息后，将消息持久化到存储系统，这条消息的状态是**待发送**状态。
3. MQ服务器返回ACK确认到生产者，此时MQ不会触发消息推送事件
4. 生产者执行本地事务
5. 如果本地事务执行成功，即commit执行结果到MQ服务器；如果执行失败，发送rollback。
6. 如果是正常的commit，MQ服务器更新消息状态为**可发送**；如果是rollback，即删除消息。
7. 如果消息状态更新为可发送，则MQ服务器会push消息给消费者。消费者消费完就回ACK。
8. 如果MQ服务器长时间没有收到生产者的commit或者rollback，它会反查生产者，然后根据查询到的结果执行最终状态。

### ==如何处理消息队列的消息积压问题==
消息积压是因为生产者的生产速度，大于消费者的消费速度。遇到消息积压问题时，我们需要先排查，是不是有bug产生了。

如果不是bug，我们可以**优化一下消费的逻辑**，比如之前是一条一条消息消费处理的话，我们可以确认是不是可以优为**批量处理消息**。如果还是慢，我们可以考虑水平扩容，增加Topic的队列数，和消费组机器的数量，提升整体消费能力。

## Kafka
### ==特点==
- 高吞吐量、低延迟：kafka每秒可以处理几十万条消息，它的延迟最低只有几毫秒，每个topic可以分多个partition, consumer group 对partition进行consume操作。
- 可扩展性：kafka集群支持热扩展
- 持久性、可靠性：消息被持久化到本地磁盘，并且支持数据备份防止数据丢失
- 容错性：允许集群中节点失败（若副本数量为n,则允许n-1个节点失败）
- 高并发：支持数千个客户端同时读写
### ==Kafka与其他消息队列有什么区别==
- **Kafka (Log-based)**:
    - **设计初衷**：用于高吞吐量的日志处理和大数据流。
    - **存储模式**：**追加写（Append-only Log）**。消息被持久化到磁盘，且**不会**在被消费后立即删除，而是根据保留策略（时间或大小）过期删除。
    - **消费模式**：支持**消息回溯（Replay）**。消费者可以重置 Offset（偏移量），重新消费过去的消息。
    - **状态管理**：Broker 不维护消费者的状态，由消费者自己维护 Offset（记录读到哪里了）。
- **RabbitMQ / ActiveMQ (Queue-based)**:
    - **设计初衷**：用于复杂的业务路由和任务分发。
    - **存储模式**：基于内存（也可持久化）。设计目标是**“空队列”**，消息一旦被消费者确认（Ack），就会从队列中**物理删除**。
    - **消费模式**：不支持回溯。消息消费完就没了。
    - **状态管理**：Broker 维护消费者的状态（谁读了哪条消息），Broker 压力较大。

### ==kafka为什么这么快==
- **顺序写入优化**：Kafka将消息顺序写入磁盘，减少了磁盘的寻道时间。这种方式比随机写入更高效，因为磁盘读写头在顺序写入时只需移动一次。
- **批量处理技术**：Kafka支持批量发送消息，这意味着生产者在发送消息时可以等待直到有足够的数据积累到一定量，然后再发送。这种方法减少了网络开销和磁盘I/O操作的次数，从而提高了吞吐量。
- **零拷贝技术**：Kafka使用零拷贝技术，可以直接将数据从磁盘发送到网络套接字，避免了在用户空间和内核空间之间的多次数据拷贝。这大幅降低了CPU和内存的负载，提高了数据传输效率。
- **压缩技术**：Kafka支持对消息进行压缩，这不仅减少了网络传输的数据量，还提高了整体的吞吐量。

### ==kafka的模型==
#### 消费者模型
消息由生产者发送到kafka集群后，会被消费者消费。一般来说我们的消费模型有两种：推送模型(push)和拉取模型(pull)。

**推送模型（push）**
- 基于推送模型（push）的消息系统，有消息代理记录消费者的消费状态。
- 消息代理在将消息推送到消费者后，标记这条消息已经消费，但这种方式无法很好地保证消费被处理。
- 如果要保证消息被处理，消息代理发送完消息后，要设置状态为“已发送”，只要收到消费者的确认请求后才更新为“已消费”，这就需要代理中记录所有的消费状态，但显然这种方式不可取。

缺点：
- **push模式很难适应消费速率不同的消费者**
- 因为消息发送速率是由broker决定的，push模式的目标是尽可能以最快速度传递消息，但是这样很容易造成consumer来不及处理消息，典型的表现就是拒绝服务以及网络拥塞。

**拉取模型(pull)**
kafka采用拉取模型，由消费者自己记录消费状态，每个消费者互相独立地顺序拉取每个分区的消息。
![img](https://cdn.xiaolincoding.com//picgo/1717921512808-fb0372de-7ada-4bc4-bfb5-7e599d9335ac.png)
说明：
- 有两个消费者（不同消费者组）拉取同一个主题的消息，消费者A的消费进度是3，消费者B的消费进度是6。
- 消费者拉取的最大上限通过最高水位（watermark）控制，生产者最新写入的消息如果还没有达到备份数量，对消费者是不可见的。
- 这种由消费者控制偏移量的优点是：**消费者可以按照任意的顺序消费消息**。比如，消费者可以重置到旧的偏移量，重新处理之前已经消费过的消息；或者直接跳到最近的位置，从当前的时刻开始消费。

#### 消费者组
kafka 消费者是以consumer group消费者组的方式工作，由一个或者多个消费者组成一个组，共同消费一个topic。每个分区在同一时间只能由group中的一个消费者读取，但是多个group可以同时消费这个partition。
![img](https://cdn.xiaolincoding.com//picgo/1717921705776-04ba7b13-4350-4444-b332-9b3607096fe2.png)

上图中，有一个由三个消费者组成的group，有一个消费者读取主题中的两个分区，另外两个分别读取一个分区。某个消费者读取某个分区，也可以叫做某个消费者是某个分区的拥有者。

优点在于：
- 消费者可以通过水平扩展的方式同时读取大量的消息。
- 如果一个消费者失败了，那么其他的group成员会自动负载均衡读取之前失败的消费者读取的分区。


#### 消费方式
kafka 消费者采用 pull（拉）模式从 broker中读取数据。
pull 的优点：
- pull 模式可以根据 consumer 的消费能力以适当的速率消费消息

缺点：
- 如果 kafka 没有数据，消费者可能会陷入循环中，一直返回空数据。针对这一点，Kafka 的消费者在消费数据时会传入一个时长参数 timeout，如果当前没有数据可供消费，consumer 会等待一段时间之后再返回，这段时长即为 timeout。


### ==kafka如何保证顺序读取消息==
Kafka 可以保证在同一个分区内消息是有序的，生产者写入到同一分区的消息会按照写入顺序追加到分区日志文件中，消费者从分区中读取消息时也会按照这个顺序。这是 Kafka 天然具备的特性。

要在 Kafka 中保证顺序读取消息，需要结合生产者、消费者的配置以及合适的业务处理逻辑来实现。以下具体说明如何实现顺序读取消息：
- 生产者端确保消息顺序：为了保证消息写入同一分区从而确保顺序性，生产者需要将消息发送到指定分区。可以通过自定义分区器来实现，通过为消息指定相同的Key，保证相同Key的消息发送到同一分区。
- 消费者端保证顺序消费：消费者在消费消息时，需要单线程消费同一分区的消息，这样才能保证按顺序处理消息。如果使用多线程消费同一分区，就无法保证消息处理的顺序性。

Kafka 本身不能保证跨分区的消息顺序性，如果需要全局的消息顺序性，通常有以下两种方法：
- 只使用一个分区：将所有消息都写入到同一个分区，消费者也只从这个分区消费消息。但这种方式会导致 Kafka 的并行处理能力下降，因为 Kafka 的性能优势在于多分区并行处理。
- 业务层面保证：在业务代码中对消息进行编号或添加时间戳等标识，消费者在消费消息后，根据这些标

### ==kafka消息积压怎么办==
- 增加消费者实例可以提高消息的消费速度，从而缓解积压问题。你需要确保消费者组中的消费者数量不超过分区数量，因为一个分区同一时间只能被一个消费者消费。
- 增加 Kafka 主题的分区数量可以提高消息的并行处理能力。在创建新分区后，你需要重新平衡消费者组，让更多的消费者可以同时消费消息。

### ==如何保证消息不重复消费==
可以使用幂等性生产，同一个生产者实例发送的同一条消息（包含相同内容和唯一标识），即便网络波动、重试等原因多次发送，最终也只会被Kafka写入一次，以此来避免消息重复
启用幂等生产方式很简单，只需在生产端配置中设置enable.idempotence为true

### 为什么一个分区只能由一个消费者组的一个消费者消费
同一时刻，一条消息只能被组中的一个消费者实例消费

如果两个消费者负责同一个分区，那么就意味着两个消费者同时读取分区的消息，由于消费者自己可以控制读取消息的offset，就有可能C1才读到2，而C1读到1，C1还没处理完，C2已经读到3了，则会造成很多浪费，因为这就相当于多线程读取同一个消息，会造成消息处理的重复，且不能保证消息的顺序。


### 如果每次业务操作完都要发变更消息，对消息队列的压力会不会很大。应该如何解决
如果**每一次**细粒度的数据库操作（Create/Update/Delete）都触发一条 MQ 消息，在高并发场景下（如秒杀、大促、热点事件），会产生“写放大”效应。这不仅会给 MQ 带来巨大的吞吐压力，也会占用大量的网络带宽和 CPU 资源。

#### 发送策略优化：聚合与去重
不要产生变动就立刻发，而是“攒一攒”再发。
- **合并写（Write Merging / Debouncing）**：
    - **场景**：同一个 Key 在短时间内被修改了 100 次（例如商品的库存不断扣减，或者文章的点赞数疯狂上涨）。
    - **策略**：应用内部维护一个短时间的缓冲窗口（比如 500ms 或 1秒）。在这段时间内，无论该 Key 变动多少次，我们只记录“它变了”。窗口结束时，只发送**一条**消息：“ID为100的商品库存变了”。
    - **效果**：将 100 条 MQ 消息压缩为 1 条，压力直接降低 99%。
- **批量发送（Batching）**：
    - **策略**：生产者不要一条一条发，而是攒够 100 条或者每隔 200ms，打包成一个 List 一次性发送给 MQ。
    - **效果**：大幅减少网络 IO 次数（TCP 握手/挥手、协议头开销）。Kafka 和 RocketMQ 的 SDK 默认都支持开启 Batch 发送。

#### 消息体设计优化：轻量化
不要把整个数据对象都塞进 MQ，只发“信号”。
- **只发 ID（Cache Invalidation Pattern）**：
    - **重型消息**：{ "id": 1, "name": "Apple", "price": 10, "desc": "...", "timestamp": ... } （不仅大，而且消费者拿到后可能还得去校验）。
    - **轻型消息**：{ "target": "product", "id": 1, "action": "update" }。
    - **逻辑**：消费者收到消息后，动作仅仅是**删除 Redis 对应的 Key**（Cache Aside Pattern）。下一次用户请求时，发现 Redis 没数据，自然会去查 DB 并回填最新的。
    - **好处**：消息体极小，带宽占用低；且避免了并发更新时的时序问题（直接删 Key 比更新 Value 也就没有覆盖旧数据的风险）。
- **过滤无效字段**：
    - **策略**：在代码层面判断，只有**关键字段**（如价格、库存、状态）变更才发消息。如果只是更新了“最后修改时间”、“备注”等不影响业务逻辑的字段，完全可以不发消息，或者不刷新缓存。

#### 架构优化：基于Binlog的异步同步（Canal模式）
- **痛点**：如果在业务代码（Controller/Service）里发 MQ，业务接口的响应时间会变慢（因为多了一次网络调用），且业务服务器 CPU 升高。
- **解决方案**：业务代码**只管写 MySQL**，完全不操作 MQ。
    1. **MySQL**：负责落地数据，并生成 Binlog（这是数据库本来就要做的事，无额外负担）。
    2. **Canal/Debezium**：伪装成 MySQL Slave，实时解析 Binlog。
    3. **MQ**：Canal 将解析后的变动投递到 MQ。
- **为什么能减压？**
    - **剥离压力**：发送消息的压力从成百上千台业务服务器，转移到了几台 Canal 服务器上。
    - **Canal 内部聚合**：Canal 可以配置聚合策略，比如针对同一行记录的多次变更进行合并，仅输出最终状态。


#### 采样与降级
- **非核心数据采样**：对于点赞数、阅读量这种非精确数据，更新 100 次才发一次消息同步 Redis/DB。中间的 99 次只在本地内存（Local Cache）累加。
- **只读不写**：在极端峰值期间，暂停缓存同步。允许用户看到稍微滞后（几秒）的数据，等洪峰过去后，再触发一次全量预热或通过补偿机制追平。

### ==消息中间件如何做到高可用==
消息中间件如何保证高可用呢？单机是没有高可用可言的，高可用都是对集群来说的，一起看下kafka的高可用吧。

Kafka 的基础集群架构，由多个`broker`组成，每个`broker`都是一个节点。当你创建一个`topic`时，它可以划分为多个`partition`，而每个`partition`放一部分数据，分别存在于不同的 broker 上。也就是说，一个 topic 的数据，是分散放在多个机器上的，每个机器就放一部分数据。

有些伙伴可能有疑问，每个`partition`放一部分数据，如果对应的broker挂了，那这部分数据是不是就丢失了？那还谈什么高可用呢？
Kafka 0.8 之后，提供了复制品副本机制来保证高可用，即每个 partition 的数据都会同步到其它机器上，形成多个副本。然后所有的副本会选举一个 leader 出来，让leader去跟生产和消费者打交道，其他副本都是follower。写数据时，leader 负责把数据同步给所有的follower，读消息时， 直接读 leader 上的数据即可。如何保证高可用的？就是假设某个 broker 宕机，这个broker上的partition 在其他机器上都有副本的。如果挂的是leader的broker呢？其他follower会重新选一个leader出来。

### ==kafka是如何保证高可用性的==
Kafka的高可用性（High Availability，HA）主要依赖于多副本机制和故障转移能力。简单说就是Kafka允许数据在多个节点上都有备份，当某个节点挂掉的时候，系统能自动切换到有备份的节点继续服务
首先是副本机制，这是Kafka高可用的基石。kafka将数据分成多个分区，分布在不同的Broker上，每个分区可以配置多个副本。通常的生产环境配置为3个（1个Leader+2个Follower）。Leader负责所有的读写请求；而follower仅仅从Leader复制数据，不直接处理客户端请求。如果Leader所在Broker宕机，Kafka会从Follower中选举出一个新的Leader继续服务
Kafka还引入了ISR集合，ISR集合包含Leader本身以及所有能跟得上Leader数据同步进度的Follower。同步滞后过多或长时间未响应的Follower会被踢出ISR，放入OSR。当Leader宕机时，Controller只会从ISR列表中选举出新的Leader。保证新的Leader用于尽可能全的数据。
同时Kafka通过acks参数保证数据的一致性和生产者的可靠性。
Kafka集群中还有一个特殊的Broker角色叫Controller，它负责管理集群状态。KRaft模式下的故障检测是基于Raft共识算法，Quorum Controller内部进行心跳检测。
故障恢复流程：
1. Broker A 宕机。
2. Controller 检测到 Broker A 掉线。
3. Controller 遍历所有 Leader 位于 Broker A 上的分区。
4. 对于每个分区，Controller 从其 ISR 列表中选出一个新的 Leader。
5. Controller 通知其他 Broker 更新元数据（Metadata），让客户端请求指向新的 Leader。
Controller自身的高可用：
如果Controller挂了，依靠Raft 算法在 Controller Quorum 中选举新的 Leader Controller，速度更快，元数据同步更可靠。
Kafka还可以保证消费者的高可用，比如消费者组：组内消费者共同分担Topic的所有分区。重平衡：如果组内某个消费者挂了，Group Coordinator（协调者）会检测到，并触发重平衡，将挂掉的消费者负责的分区重新分配给组内其他存活的消费者。


### ==kafka的消息确认机制==
Kafka的消息确认机制有三种：0，1，-1：
- **ACK=0**：这是最不可靠的模式。生产者在发送消息后不会等待来自服务器的确认。这意味着消息可能会在发送之后丢失，而生产者将无法知道它是否成功到达服务器。
- **ACK=1**：这是默认模式，也是一种折衷方式。在这种模式下，生产者会在消息发送后等待来自分区领导者（leader）的确认，但不会等待所有副本（replicas）的确认。这意味着只要消息被写入分区领导者，生产者就会收到确认。如果分区领导者成功写入消息，但在同步到所有副本之前宕机，消息可能会丢失。
- **ACK=ALL**：这是最可靠的模式。在这种模式下，生产者会在消息发送后等待所有副本的确认。只有在所有副本都成功写入消息后，生产者才会收到确认。这确保了消息的可靠性，但会导致更长的延迟。

### ==kafka的副本机制和消息可靠性是如何是实现的==
### 一、 Kafka 的副本机制架构
Kafka 的每个 Topic 被切分为多个 Partition（分区），而每个 Partition 可以有多个副本（Replica）。
#### 1. Leader 与 Follower
- **Leader**：每个分区都有一个 Leader。**所有的读写请求全部由 Leader 处理**。
- **Follower**：其他副本是 Follower。它们**不处理客户端请求**（除了较新版本支持的 Follower Fetching，但主流场景下不服务），唯一的任务就是从 Leader 那里**被动拉取（Pull）**消息，并写入自己的本地日志。

#### 2. ISR (In-Sync Replicas) —— 核心概念
Kafka 并不是要求所有副本都同步完才算成功，也不是只同步 Leader 就结束。它维护了一个动态集合：**ISR**。
- **定义**：当前与 Leader 保持“同步”的副本集合（包含 Leader 自己）。
- **判定标准**：如果一个 Follower 能及时拉取 Leader 的消息，不落后太多（由 replica.lag.time.max.ms 控制），它就在 ISR 里。
- **作用**：只有 ISR 里的成员，才有资格在 Leader 挂掉时被选为新的 Leader。这保证了新的 Leader 拥有较全的数据。

#### 3. HW 与 LEO —— 数据一致性的标尺
为了防止 Leader 挂掉后，消费者读到了“未来会丢失”的数据，Kafka 引入了水位机制：
- **LEO (Log End Offset)**：日志末端位移。记录了每个副本当前写到了哪一条消息。
- **HW (High Watermark)**：高水位。**HW = ISR 中所有副本 LEO 的最小值**。
- **机制**：
    - ==消费者只能拉取到 **HW 之前**的消息。==
    - ==这意味着，只有当 ISR 中的所有副本都收到了这条消息，这条消息才对消费者可见。==
    - 这样保证了即使 Leader 挂了，新选出来的 Leader（来自 ISR）一定也有这条消息，消费者不会发现数据回滚。

### 二、 如何保证消息可靠性（全链路不丢数据）
仅仅有副本机制是不够的，还需要 Producer、Broker、Consumer 三端的配置配合，才能实现“Zero Data Loss”。
#### 1. Producer 端（发送端）
核心参数：**acks**
- **acks=0**：发完就走，不管服务端收没收到。性能最高，可靠性最低（**可能丢**）。
- **acks=1**：只要 Leader 写入成功就返回成功。如果 Leader 刚写完还没同步给 Follower 就挂了，**会丢数据**。
- **acks=all (或 -1)**：**推荐配置**。Leader 必须等待 **ISR 中所有的副本**都写入成功，才向 Producer 返回成功。这是最高可靠性。

辅助参数：
- retries：设置重试次数（建议设很大，如 MAX_INT），处理网络抖动。
- enable.idempotence=true：开启幂等性，防止重试导致消息重复。

#### 2. Broker 端（存储端）
仅开启 acks=all 还不够，如果 ISR 里只有 Leader 一个光杆司令，那 acks=all 就退化成了 acks=1。
核心参数：
- **replication.factor >= 3**：副本因子。建议至少 3 个副本。
- **min.insync.replicas > 1**：**最关键参数**。
    - 它规定了当 acks=all 时，ISR 中**至少**要有几个副本存活才能接受写请求。
    - 建议设置为 2（在 3 副本的情况下）。
    - 作用：如果 ISR 里只剩 Leader 了（数量 < 2），Broker 会直接拒绝写入，报错 NotEnoughReplicas，以此保护数据不丢失，而不是降级写入。  
- **unclean.leader.election.enable = false**：
    - 禁止“不洁”选举。如果 ISR 里的副本全挂了，是否允许从非 ISR（数据落后）的副本中选 Leader？
    - 设为 false（默认值），宁可服务不可用，也不选丢数据的副本上位。    

#### 3. Consumer 端（消费端）
消费者这端丢数据，通常是因为“自动提交”导致的。
- **策略**：**关闭自动提交，改为手动提交** (enable.auto.commit = false)。
- **流程**：
    1. 拉取消息。
    2. 执行业务逻辑（落库、计算等）。
    3. **业务成功后，手动提交 Offset**。
- 代价：如果业务处理完，提交 Offset 前挂了，重启后会重复消费（At Least Once），但绝对不会丢。


### 故障转移
#### 普通的broker故障
假设 Broker A 挂了，它上面原本是 Partition-0 和 Partition-1 的 **Leader**。现在的目标是：在其他活着的 Broker 上，为这两个分区选出新的 Leader。
##### 1. 故障检测 (Detection)
- **机制**：Broker 会与 Zookeeper 保持一个会话（Session），并定时发送心跳。
- **触发**：当 Broker A 宕机，心跳停止。Zookeeper 检测到 Session 超时，会把 /brokers/ids/A 这个临时节点删掉。
##### 2. 控制器感知 (Controller Notified)
- **机制**：集群中有一个特殊的 Broker 叫 **Controller**（控制器），它一直监听着 Zookeeper 的 /brokers/ids 路径。
- **触发**：一旦 Broker A 的节点消失，Controller 立刻收到 Zookeeper 的 **Watch 事件通知**：“Broker A 不见了！”
##### 3. 选举新 Leader (Election)
Controller 拿到故障 Broker A 上所有的 Leader 分区列表，开始为每一个分区选新 Leader：
- **规则**：Controller 会去读取该分区的 **ISR (In-Sync Replicas，同步副本列表)**。
- **策略**：
    - **优先**：从 ISR 列表中，选择**第一个存活**的副本作为新的 Leader。（因为 ISR 里的副本数据和旧 Leader 是同步的，不丢数据）。
    - **极端情况（Unclean Election）**：如果 ISR 里没有活着的节点了（比如全挂了），且配置了 unclean.leader.election.enable=true，则会从非 ISR（数据滞后）的副本中选一个。这会保住可用性，但**会丢失数据**。
##### 4. 广播变更 (Broadcast)
- Controller 选好新 Leader 后，会向集群中**所有存活的 Broker** 发送 LeaderAndIsrRequest 请求。
- **告诉新 Leader**：“你上位了，开始接收读写请求。”
- **告诉其他 Follower**：“老大换人了，以后去同步新 Leader 的数据。”
##### 5. 客户端元数据更新
- Producer 和 Consumer 在尝试连接旧 Leader 失败后，会向集群请求更新 **Metadata（元数据）**。
- 获取到新 Leader 的地址后，客户端自动重连到新 Broker 继续工作。

#### Controller Broker故障
##### 1. 竞选新 Controller
- **发现**：所有活着的 Broker 都在 Zookeeper 上监听 /controller 临时节点。当 Controller 挂了，ZK 删除了这个节点，所有存活 Broker 都会收到通知。
- **抢注**：所有 Broker 都会立刻尝试去 ZK 上创建 /controller 临时节点。
- **胜者为王**：**先到先得**。第一个创建成功的 Broker 自动成为新的 Controller。
###### 2. 新官上任 (Initialization)
- 新的 Controller 会从 ZK（或 KRaft Log）中拉取所有的元数据（分区状态、副本分配等），初始化内存状态。
##### 3. 处理故障
- 新 Controller 初始化完成后，它会发现：“咦，Broker A（旧 Controller）不在 /brokers/ids 列表里了”。
- 于是，它会触发**“情况一”**中的流程，去处理 Broker A 上原本承载的 Partition Leader 的转移工作。

### Controller Broker会处理业务数据吗
在经典的 Zookeeper 架构中，Controller 是从普通 Broker 中选举出来的，它**既要负责集群管理，也要负责正常的数据读写**，并没有特权不存数据。  
但是在最新的 KRaft 架构中，生产环境通常建议将 Controller 节点**独立部署**，此时 Controller 节点就**只负责管理**，不再承担业务数据的读写，以此来保证集群的高可用性。

### Controller如何监测其他节点存活
核心机制：RPC 心跳 (Heartbeat)
1. **主动汇报：**  
    所有的 Broker 节点，必须周期性地向 **Active Controller** 发送 RPC 请求。这个请求叫做 BrokerHeartbeatRequest。
    - 这不仅是报平安，Broker 还会通过这个请求汇报自己的状态（比如磁盘是否满了，元数据是否同步到了最新）。
2. **维护租约：**  
    Controller 内部维护了一个类似“考勤表”的机制。它记录了每个 Broker 上一次发送心跳的时间。
3. **判定死亡：**  
    如果 Controller 在规定的时间（broker.session.timeout.ms）内没有收到 Broker-1 的心跳请求：
    - Controller 直接判定 Broker-1 **“失联 (Fenced)”**。
    - Controller 将该 Broker 的状态标记为 Offline。
4. **处理后事：**  
    Controller 立即开始重新计算分区 Leader，并将新的元数据推送给其他活着的 Broker。
**总结：** 这是一个**主动**的过程。Broker 必须不断地说“我活着”，Controller 一旦听不到，就认为它挂了。

## ==分区策略==
**Kafka 的分区策略（Partition Strategy）** 是指 **Producer（生产者）** 在发送消息时，决定将这条消息发送到 Topic 中的哪一个 Partition（分区）的算法。
### 为什么要分区
1. **负载均衡**：将大量数据分散到不同的 Broker 上存储和处理。
2. **并发消费**：分区是 Kafka 并发的最小单位。Partition 越多，支持的并发消费者就越多。

### 四种常见的分区策略
Kafka 的生产者客户端（Client）默认提供了一个 DefaultPartitioner，它的行为逻辑如下：
#### 1. 指定分区策略 (Direct)
**逻辑**：在发送消息的 ProducerRecord 中显式指定 partition 字段的值（比如 0, 1, 2）。
- **行为**：Producer 不做任何计算，直接把消息发到你指定的那个分区。
- **应用场景**：极少使用，除非你有极其特殊的业务需求（例如：所有系统日志必须只发到 0 号分区）。

#### 2. 哈希策略 (Hash Strategy) —— **有 Key 时的默认策略**
**逻辑**：如果消息设置了 **Key**（例如 UserID、OrderID），但没有指定 Partition。
- **算法**：hash(key) % total_partitions
- **行为**：相同的 Key 永远会被路由到同一个分区。
- **应用场景**：**保证消息顺序性**。
    - 比如电商订单状态流转（下单 -> 支付 -> 发货），你必须确保同一个 OrderID 的消息按顺序处理。通过把 OrderID 设为 Key，它们就会全部进入同一个分区，而消费者在单个分区内是有序消费的。    
- **注意**：如果分区数量发生变化（扩容），Hash 结果会变，会导致之前的 Key 路由到新分区，从而打破顺序性。

#### 3. 粘性分区策略 (Sticky Partitioning) —— **无 Key 时的默认策略 (Kafka 2.4+)**
**逻辑**：如果消息 **没有 Key**，且 **没有指定 Partition**。
- **旧版本 (Round-Robin / 轮询)**：
    - 在 Kafka 2.4 之前，默认是简单的轮询。即消息 1 去 P0，消息 2 去 P1，消息 3 去 P2...
    - **缺点**：无法有效利用 Batching（批处理）。因为消息太分散，导致每个 Batch 可能只有一条消息，增加了网络 IO 请求次数，效率低。
- **新版本 (Sticky / 粘性)**：
    - 为了解决旧版轮询导致 Batch 效率低的问题，Kafka 引入了粘性策略
    - **行为**：Producer 会随机选择一个分区，并尽可能地**坚持（Stick）**使用这个分区，直到该分区的 Batch 满了（或者超时 linger.ms）。一旦 Batch 发送出去，Producer 再随机选一个新的分区，继续“粘”住它。
    - **优点**：这就好比坐大巴，旧版是每来一个人就发一辆空车；新版是等人坐满一辆车再发车，然后换下一辆。这极大地提高了**吞吐量**，减少了延迟。

#### 4. 自定义分区策略 (Custom Partitioner)
**逻辑**：实现 Kafka 提供的 Partitioner 接口，重写 partition() 方法。
- **行为**：完全由你的代码逻辑决定。
- **应用场景**：
    - **数据倾斜处理**：比如某些 Key 特别大，你想单独处理。   
    - **VIP 路由**：比如 Key 是 VIP 用户，强制路由到高性能的 Partition（比如 0-3），普通用户路由到 4-9。
    - **地域路由**：根据 IP 归属地路由到最近的机房对应的分区。

### ==怎么保证顺序性==
发送消息时必须设置key，并且使用Hash分区策略。这样同一个key的数据就定在同一个分区
- 如果在消费者端开启了多Goroutine消费数据怎么办
	- 那还是会乱序。如果必须严格顺序，消费者在单个分区内必须单线程处理；或者在内存中再做一次Hash分发到不同的channel

假如在消费者端开启了多线程：
```Go
package main

import (
	"fmt"
	"hash/fnv"
	"sync"
	"time"
)

// 1. 定义消息结构
type KafkaMsg struct {
	Key    string // 比如 OrderID
	Value  string // 比如 "订单创建", "订单支付"
	Offset int64
}

// 定义 Worker 的数量（类似于 Java 的线程池大小）
const WorkerNum = 5

func main() {
	// 2. 准备 Worker 的通道 (相当于 Java 的内存队列)
	// 每个 Worker 对应一个 buffered channel
	workerChans := make([]chan KafkaMsg, WorkerNum)
	for i := 0; i < WorkerNum; i++ {
		workerChans[i] = make(chan KafkaMsg, 100) // 缓冲区大小可调整
	}

	// 3. 启动 Worker Goroutines (消费者线程)
	var wg sync.WaitGroup
	for i := 0; i < WorkerNum; i++ {
		wg.Add(1)
		go startWorker(i, workerChans[i], &wg)
	}

	// 4. 模拟消费者拉取并分发消息 (Main Goroutine)
	// 假设我们从 Kafka 拉到了这几条消息，注意 Key 的重复和顺序
	messages := []KafkaMsg{
		{Key: "Order-100", Value: "1.创建订单", Offset: 1},
		{Key: "Order-200", Value: "1.创建订单", Offset: 2},
		{Key: "Order-100", Value: "2.支付成功", Offset: 3}, // Order-100 的第二步
		{Key: "Order-300", Value: "1.创建订单", Offset: 4},
		{Key: "Order-200", Value: "2.支付成功", Offset: 5},
		{Key: "Order-100", Value: "3.发货", Offset: 6},     // Order-100 的第三步
	}

	fmt.Println("--- 开始分发消息 ---")

	for _, msg := range messages {
		// 【关键点】：计算 Hash 并取模
		// 保证同一个 Key 总是算出同一个 workerIndex
		workerIndex := getWorkerIndex(msg.Key, WorkerNum)

		fmt.Printf("[Dispatcher] 将 %s (%s) 分发给 Worker-%d\n", msg.Key, msg.Value, workerIndex)

		// 发送到对应的 Channel
		workerChans[workerIndex] <- msg
	}

	// 模拟关闭：发送完所有数据后，关闭所有 channel
	for i := 0; i < WorkerNum; i++ {
		close(workerChans[i])
	}

	// 等待所有 Worker 处理完成
	wg.Wait()
	fmt.Println("--- 所有消息处理完毕 ---")
}

// startWorker 模拟具体的业务处理逻辑
func startWorker(id int, ch <-chan KafkaMsg, wg *sync.WaitGroup) {
	defer wg.Done()
	
	// 不断从 Channel 读取消息，直到 Channel 被关闭
	for msg := range ch {
		// 模拟处理耗时，故意加点随机性，看看会不会乱序
		time.Sleep(10 * time.Millisecond) 
		
		fmt.Printf("\t[Worker-%d] 处理完成: Key=%s, Content=%s, Offset=%d\n", 
			id, msg.Key, msg.Value, msg.Offset)
	}
}

// getWorkerIndex 根据 Key 算出固定的 Worker ID
func getWorkerIndex(key string, numWorkers int) int {
	h := fnv.New32a()
	h.Write([]byte(key))
	hash := h.Sum32()
	// 取模运算
	return int(hash % uint32(numWorkers))
}
```


### 什么是数据倾斜
某个分区的消息量远远大于其他分区，导致某个消费者处理不过来，发生积压，而其他消费者很闲
- 通常是因为Hash策略使用Key分布不均匀（例如90%的订单都来自同一个大客户ID）
	- 避免使用分布不均的key
	- 使用自定义分区策略，探测到热点key时，在key后面加随机点缀，将其打散到不同分区

### 增加分区数有什么影响
增加并发度，提升吞吐量
- 如果是基于KeyHash的策略，增加分区会导致Rehash，原来的Key可能会去往新分区，打破已有的顺序性

### 如何解决数据倾斜
#### 随机后缀法（最彻底但有代价）
这是解决单点热点Key的最常用的方法

原理：
如果 Key = "HotItem" 导致所有消息都去了 Partition 0。  
我们在发送消息前，给这个 Key 加上一个随机后缀（或者轮询后缀）
- HotItem -> HotItem-0 -> Partition 0
- HotItem -> HotItem-1 -> Partition 1
- HotItem -> HotItem-2 -> Partition 2

优点：
- 彻底解决了倾斜问题，数据被完美打散到多个分区
- 大幅提升吞吐量
缺点：
- 丧失了局部有序性。因为同一个原始Key的数据现在分散在了不同的分区，消费者可能先收到“修改订单”，后收到“创建订单”

#### 自定义分区器
如果你能提前知道哪些是“热点 Key”，可以使用这种“手术刀”式的精准治理方案。

原理：
编写各种逻辑，将热点Key和普通key分开处理
- 普通key：继续走默认的Hash策略
- 热点key：单独走一套逻辑，比如强制随机分发到一组专用的分区中，或者打散到所有分区

优点：
- 对普通key依然保留了顺序性
- 灵活可控

缺点：
- 维护成本高：需要维护一份“热点Key名单”。如果热点是突发的，静态名单会失效
- 热点Key依然会乱序

#### 消息预聚合
与其发送 1000 条 {"id": 1, "count": 1} 的消息，不如在内存中攒一攒，每隔 100ms 发送一条 {"id": 1, "count": 1000}。

适用场景：
- 统计类业务（如点赞数、浏览量）

优点：
- 极大减少了传输的消息数量（QPS）
- 从源头上减轻了Kafka和Consumer的压力

缺点：
- 增加延迟：数据不再是实时的，有聚合窗口的延迟
- 并不适用于所有业务：只能用于可聚合的数据，对于状态流转类业务无效
- 开发复杂度：需要在Producer端引入内存buffer或使用流计算引擎

#### 两阶段消费
如果业务既有热点数据，有要求数据最终有序，则需要在Consumer做文章
- 第一阶段（打散）：
	- Producer使用随机后缀法将热点key打散发送到Kafka的不同分区
	- Consumer GroupA快速拉取数据，不处理业务逻辑，只是把后缀去掉，根据原始Key进行二次Hash，转发到另一个新的Topic（Re-shuffle Topic）
- 第二阶段（有序处理）：
	- Consumer GroupB消费新的Topic。此时由于经过二次hash，相同的原始Key已经回到同一个分区，顺序性得以恢复，且由于数据已经被前面的步骤平摊处理了一部分，压力会小很多
	- 如果在 Kafka 层面只是为了解决单分区写入压力，第一阶段直接入库或处理即可。如果是为了解决 Flink/Spark 中的计算倾斜，通常用这种两阶段聚合。
