# IM
### 1) 架构与技术选型

#### 为什么用 `go-zero`，相比 gin/kratos 的优势是什么？
1. 微服务基础设施完整度：我们需要 gRPC、服务注册发现、治理能力，不只是 Web 框架。`gin` 很轻，但很多能力要自己拼。
2. 开发效率：`go-zero` 的代码生成和工程结构比较统一，团队多人协作时，接口定义、服务拆分和目录规范更一致。
3. 治理成本：IM 场景高并发、长连接，对限流熔断、超时控制、日志追踪依赖高。`go-zero` 在这块开箱能力更贴近我们的需求。
4. 团队现实：选型不只看“理论最好”，还要看团队经验和维护成本。我们团队已有 `go-zero` 实践，出问题能更快定位和处理。

- gin 真的不行吗？  
你答：`gin` 做单体或轻服务非常好，但我们是多服务协作+高并发治理场景，如果用 gin 要补很多中间层，前期快，后期治理成本会上来。
- go-zero 的短板是什么？  
你答：短板主要是框架约束相对强，灵活性和自定义空间会受一点影响；另外团队需要统一规范，不然会“只用了脚手架，没用好治理能力”。


#### 为什么要微服务拆分？按什么边界拆？
1. 业务职责边界
- IM 域负责实时通信主链路：消息收发、会话读写、长连接。
- 社交域负责关系数据：好友、群、成员、申请。
- 任务域负责异步处理：消费 Kafka，做消息转发、已读等后处理。  
    职责清晰后，开发和排障都更可控。
1. 数据一致性边界  
    强一致尽量收敛在单域内，比如关系数据在 social，消息与会话在 im。跨域通过 RPC/事件做最终一致，避免分布式事务把复杂度拉满
2. 流量与扩容边界  
    IM 的连接和消息是高频高峰值；社交关系是中低频查询；任务域是可水平扩展的消费模型。拆开后可以“哪里瓶颈扩哪里”，资源利用率更高。
最后再补一句取舍：微服务也带来链路变长和治理成本，我们通过 Etcd 服务发现、统一超时重试、幂等和监控告警来兜住复杂度。

- 你们是不是拆太细了？  
你答：没有按“功能点”细碎拆，而是按“业务域”拆。消息和会话在我们项目里仍在同一个 IM 域内，没有强行拆成两个独立进程，这样既有边界又不过度拆分。


#### `Etcd` 做服务发现时，如何处理实例上下线和临时网络抖动？
我们处理上下线和抖动，核心是“租约机制 + 优雅下线 + 调用端容错”：
1. 实例上线  
    服务启动后向 Etcd 注册带 lease 的临时节点（ephemeral），并持续 keepalive。只要续约正常，实例就会一直在可用列表里。
2. 实例下线
	- 优雅下线：先停止接收新流量，注销注册信息，再等待在途请求处理完后退出。
	- 异常下线：比如进程崩溃或机器故障，keepalive 中断，lease 超时自动过期，Etcd 会把节点移除。
3. 网络抖动处理
	- lease TTL 不设太激进，给短暂抖动留缓冲。
	- 客户端调用加超时、重试、失败转移，避免瞬时不可达扩大影响。
	- 监控注册数变化和错误率，防止频繁上下线（flapping）。
4. 防止“误摘除”后的雪崩  
    即使发现列表瞬时变化，客户端也会通过连接池和重试兜底，不把一次抖动放大成全链路故障。

- TTL 设多大合适？  
你答：看网络稳定性和故障发现时效的平衡。TTL 太短会误摘除，太长会故障发现慢。一般先按秒级保守配置，再根据线上抖动和恢复时间逐步调优。
- 如果 Etcd 本身抖动怎么办？  
你答：Etcd 要多节点部署，客户端要有重连和超时重试；同时服务侧不要把 Etcd 当强依赖的实时调用，拿到服务列表后本地有短暂缓存和容错，避免控制面抖动直接打崩数据面。

#### gRPC 为什么适合内部通信？你们有做连接池/超时/重试吗？
我选 gRPC 主要看三点：
1. 性能和连接模型更适合内部链路  
    内部调用频繁，gRPC 的 Protobuf 序列化更轻，HTTP/2 多路复用减少频繁建连开销，RT 和吞吐都更稳定。
2. 接口治理更强  
    通过 `.proto` 统一定义请求响应，跨服务协作时类型清晰、变更可控，能减少“字段约定不一致”的问题。
3. 和 go-zero 体系集成好  
    我们实际是 `zrpc.MustNewClient(...)` 这种方式接 Etcd 做服务发现，客户端连接复用由框架处理，不需要每次调用自己建连接。

关于你问的三点：
- 连接池/连接复用：  
    没有自己手写池子，主要依赖 gRPC 长连接 + go-zero 客户端管理来复用连接。
- 超时：  
    有做超时控制，避免下游慢导致请求堆积。
- 重试：  
    不是全链路自动重试，而是对“幂等、读类”接口做有限次数重试（比如你项目里查群成员就有最多 3 次重试），写操作谨慎重试，避免副作用放大。

- 为什么不全局开重试？  
你答：全局重试会把瞬时故障放大成重试风暴，特别是写请求会有重复执行风险。我们只对可重试场景做小次数、退避式重试，并结合超时和熔断一起用。

- 你们有没有自己做连接池参数调优？  
你答：优先使用框架默认和官方机制，重点放在超时、重试边界和监控告警；当出现连接瓶颈再按监控数据做针对性调优，而不是一开始就过度配置。

### 2) WebSocket 与实时通信

#### WebSocket 握手鉴权怎么做？
##### 1）客户端发起 WS 握手请求
- 客户端请求 `ws://<im-ws-host>/ws`
- 把 JWT token 放在请求头 `Sec-WebSocket-Protocol`（你代码里读的是小写 `sec-websocket-protocol`，HTTP header 实际大小写不敏感）
    - 这么做通常是为了兼容“浏览器 WS 不方便自定义 `Authorization`”的场景

##### 2）服务端升级连接（Upgrade）
在 `apps/im/ws/websocket/connection.go` 的 `NewConn`：

- 如果请求里带了 `Sec-WebSocket-Protocol`，服务端会把它原样回写到响应头（这是 WS 子协议协商要求）
- 然后调用 gorilla websocket 的 `Upgrade(w, r, responseHeader)` 把 HTTP 连接升级成 WebSocket 连接
- 升级成功后创建 `Conn`，并启动 `keepalive` 空闲检测协程

##### 3）握手后立刻执行鉴权
在 `apps/im/ws/websocket/server.go` 的 `ServerWs`：
- 创建好 `Conn` 后，马上执行：
- 调用你注入的认证器 `JwtAuth.Auth(w, r)`
- 如果返回 `false`：
    - 服务端会发一条提示消息（“不具备访问权限”）
    - 然后关闭连接 `conn.Close()`

##### 4）JwtAuth.Auth 里具体做了什么
在 `apps/im/ws/internal/handler/auth.go`：
1. 从 header 取 token
    - 先读 `sec-websocket-protocol`
    - 如果存在，就把它写到 `Authorization` header  
        目的：后面复用 go-zero 的 JWT 解析逻辑（它默认从 Authorization 取）
2. 解析并校验 JWT
    - 使用 `go-zero/rest/token` 的 `TokenParser.ParseToken`
    - 用配置里的 `JwtAuth.AccessSecret` 做验签
    - 解析失败或 `tok.Valid == false` 就鉴权失败
3. 提取用户身份
    - 把 `tok.Claims` 转成 `jwt.MapClaims`
    - 检查 `claims[ctxdata.Identity]` 是否存在（你们的 userId/identity 字段）
    - 没有就失败
4. 把 userId 写进 request context
    - `req.WithContext(context.WithValue(..., ctxdata.Identity, claims[ctxdata.Identity]))`
    - 后面如果需要从请求上下文取 uid，就能拿到（`ctxdata.GetUid`）

> 总结一下：token 从 `Sec-WebSocket-Protocol` 来 → 写到 `Authorization` → 验签校验 → 取 claims 里的 userId → 写入 context。

##### 5）鉴权通过后：绑定“用户 ↔ 连接”
仍在 `apps/im/ws/websocket/server.go`：
- `addConn(conn, r)` 会调用 `s.authentication.UserId(req)` 得到 uid
- 然后把连接加入映射：
    - `connToUser[conn] = uid`
    - `userToConn[uid] = conn`
并且有个关键逻辑：
- 如果 `userToConn[uid]` 已经有旧连接，会先 `Close()` 旧连接  
    所以你当前实现是 同一用户单连接（新设备会挤掉旧设备）

##### 6）连接建立后进入业务处理循环
鉴权通过后才会：
- `go s.handlerConn(conn)` 开始读写消息、心跳、ack 等逻辑

#### token 放哪，如何续期？
- 你的实现是把 token 放在请求头 `sec-websocket-protocol`：  
    `apps/im/ws/internal/handler/auth.go` 第 30 行取 `sec-websocket-protocol`，然后写入 `Authorization`：
    - `sec-websocket-protocol` → `Authorization`
- 然后用 go-zero 的 `TokenParser` 按 `JwtAuth.AccessSecret` 解析校验 JWT。

> 这种放法常用来兼容浏览器 WS 不方便带 `Authorization` 的情况。

#### 无感续期要怎么实现？
- AccessToken（短期）：给 API / WS 鉴权用，比如 30 分钟。
- RefreshToken（长期）：只用于换新 Access，比如 7 天。
##### 登录建立连接
1. 客户端登录，拿到 `accessToken + refreshToken`。
2. 用 `accessToken` 建 WS 连接（握手鉴权通过）。
3. 服务端给这个连接保存会话信息：`userId/deviceId/accessExp`。
##### 快过期时后台刷新（用户无感）
假设客户端发现 Access 还有 5 分钟过期：
1. 客户端悄悄调用 HTTP `POST /auth/refresh`（带 refreshToken）。
2. 服务端校验 refreshToken，签发新的 accessToken（必要时也轮换 refreshToken）。
3. 客户端拿到新 token 后，发一条 WS 控制消息（不是断线重连）：
    {
    "method": "auth.refresh",
    "accessToken": "new_access_xxx",
    "reqId": "123"
    }
4. WS 服务端校验这个新 accessToken（签名、过期、userId 是否和当前连接一致）。
5. 校验通过后，更新当前连接会话里的 `accessExp`，回 `auth.refresh.ack`。
6. 连接继续用，不中断聊天。

##### 为什么要加`auth.refresh`，只刷新HTTP不够吗？
不够。  
因为 WS 连接是长连接，服务端需要知道“这个连接的认证状态已经更新了”。  
否则服务端还是拿旧的过期时间，后续做过期巡检时可能会误踢。

###### 1) 连接会话状态
每个连接存：
- `userId`
- `deviceId`
- `accessExp`
- `authVersion`（可选，支持强制下线）

###### 2) WS 刷新路由
新增路由 `auth.refresh`：
- 入参：`newAccessToken`
- 校验：
    - token 有效
    - token.userId == 当前连接 userId
    - token 未撤销（可选：version 校验）
- 成功：更新 `accessExp`，返回 ack
- 失败：返回错误，必要时踢连接

###### 3) 过期巡检协程
每隔 N 秒扫描连接：
- 未过期：跳过
- 过期但在宽限期（例如 30s）：再等
- 过期且无刷新：关闭连接

> 有了这个巡检，你的鉴权生命周期才是闭环。

#### 心跳机制怎么设计（心跳间隔、超时判定）？误踢怎么避免？
项目中的心跳机制是使用的：
- 应用层心跳帧（Ping）：客户端发 `FramePing`，服务端收到后回一个 `FramePing`（相当于 pong）。
- 连接空闲超时（Idle Timeout）：服务端维护每条连接的“最后活跃时间”，超过 `maxConnectionIdle` 就主动断开。

#### 心跳帧怎么处理？
在 `apps/im/ws/websocket/server.go` 的 `handlerWrite` 里：
- 当收到 `FramePing`：
    - 服务端会立刻 `Send(&Message{FrameType: FramePing}, conn)` 回一帧 Ping（你项目里相当于 pong/ack）

同时在 `apps/im/ws/websocket/connection.go`：
- `ReadMessage()` 和 `WriteMessage()` 都会更新 `c.idle = time.Now()`  
    也就是说：只要有任何读写（包括心跳帧）就会刷新活跃时间。

> 你项目里“心跳间隔”不是服务端强制的，是客户端决定多久发一次 Ping。

#### 超时判定是怎么做的？
在 `apps/im/ws/websocket/connection.go` 的 `keepalive()`：
- 启动一个定时器
- 定时检查 `time.Since(c.idle)`
- 如果超过 `maxConnectionIdle`：调用 `c.s.CloseConn(c)` 断开连接

而你在启动 WS 服务时（`apps/im/ws/im.go`）配置了：
- `websocket.WithMaxConnectionIdle(24*7*time.Hour)`

所以你当前配置下的“超时判定”是：
- 连接空闲超过 7 天才会被服务端踢下线（非常宽松）

另外默认值（`apps/im/ws/websocket/default.go`）其实是 `math.MaxInt64`，等于“几乎不踢”。

#### 怎么避免误踢？
你这套方案里，误踢主要来自“空闲超时设得太短 + 网络抖动 + 客户端心跳不稳定”。避免手段通常是：
- 阈值要大于心跳间隔的多倍：比如心跳 30s，空闲超时至少设 90s~~180s（3~~6 倍），留出抖动空间
- 宽限与重试：第一次超时先标记“疑似离线”，再给 1~2 个心跳周期宽限（或等一次读写）
- 用读写刷新 idle：你项目已经做了（任何消息/心跳都算活跃），这会显著降低误踢
- 客户端心跳加抖动与退避：避免大量客户端同一时刻发心跳造成尖峰
- 弱网识别：移动网络切换时，优先让客户端快速重连，而不是频繁踢/连

>结合你项目当前 `maxConnectionIdle=7天`，实际上几乎不会“误踢”，更多是网络断开后靠读写报错/重连恢复。面试时可以补一句：我们配置很宽松，更多依赖客户端重连策略；如果要更“实时”识别断链，会把 idle 超时调小并加宽限策略。

#### 心跳机制有什么可以改进的
##### 现有的主要问题
###### 1）`maxConnectionIdle` 设成 7 天，失去“在线状态”的意义
你现在用空闲超时来判活，但阈值是 `24*7*time.Hour`。结果是：
- 客户端即使已经断网/进程挂了，只要底层连接没立刻触发读写错误，服务端可能很久才清理。
- “在线列表/在线状态”会滞后，影响消息投递路由、已读同步、好友在线展示等。

###### 2）心跳间隔不由服务端约束，客户端不发也没关系
服务端只是被动收到 `FramePing` 就回包，但没有：
- 规定客户端多久必须 ping 一次
- 超过 N 秒没 ping 就判定离线

所以“心跳”更像可选功能。

###### 3）没有利用 TCP/WS 原生 Ping/Pong（更标准）
你现在的 `FramePing` 是应用层自定义帧，能用，但在很多网络设备/代理场景下：
- TCP 层可能半开（客户端已死但连接还在）
- 不做底层 ping/pong + read deadline，容易清理不及时

###### 4）误踢控制策略缺失（如果你把超时调小就会暴露）
一旦把超时从 7 天调到 1~3 分钟，弱网用户可能频繁被踢，因为当前没有：
- 宽限期（grace window）
- 连续失败次数阈值
- 抖动策略（jitter）

#### 改进
###### 改进A：把“在线判活”从 7 天调到分钟级（并且有宽限）
推荐配置（面试也好说）：
- 心跳间隔：客户端每 20~30s 发一次
- 超时判定：服务端 60~90s 没收到任何读写（含 ping）判定离线
- 避免误踢：要求连续 2~3 次心跳周期都缺失才下线（例如 90s+宽限）

这样在线状态更准确，又能抗抖动。

###### 改进B：使用 gorilla/websocket 的 Ping/Pong + ReadDeadline（更标准）
典型做法是：
- 服务端定时 `WriteControl(PingMessage)`
- 设置 `SetReadDeadline(now + timeout)`
- 在 `PongHandler` 里延长 deadline 效果：
- 能更快发现半开连接
- 更符合 WebSocket 生态

###### 改进C：心跳加“抖动”与分散（减少尖峰）
客户端心跳不要整点齐发：
- 在 20~30s 基础上加随机抖动（±3s）
- 大规模在线时能明显降低心跳尖峰

###### 改进D：心跳与业务解耦，在线状态落 Redis（多实例必备）
如果 WS 多实例部署：
- 每次收到 ping 更新 `Redis online:{uid}:{deviceId}` TTL（比如 90s）
- TTL 过期即离线（天然抗实例宕机） 这是“在线状态”最常见工程解法。

#### 断线重连后如何保证消息不丢？如何补拉离线消息？
##### 1）项目里有没有设计断线重连？
- 服务端：没有“主动重连”这种说法（服务端只接受连接）。断了就清连接映射，客户端再来重新握手鉴权即可。
- 客户端侧（在仓库里能看到的）：你们有一个 WS 客户端实现 `apps/im/ws/websocket/client.go`，`Send` 失败会 `Dial` 一次重新连再发（代码里还写了 `TODO: 重连发送`）。  
    但这更像是服务间 WS 客户端（比如 `task/mq` 推送到 `im-ws`），真正“用户端 App 的重连策略”不在这个仓库里。

面试可以这样说：
> “服务端支持重连（重连就是重新握手鉴权并重新绑定 user->conn），客户端需要做重连策略（指数退避、抖动、上限、网络切换重连）。”

##### 2）断线重连后怎么保证消息不丢？
你项目的核心思路是 “消息先可靠落地，再尽力实时推”：
###### 2.1 发送链路（你代码里是这个顺序）
1. 用户通过 WS 发消息 `conversation.chat`
2. WS 服务并不直接推给对端，而是先把消息写入 Kafka：`MsgChatTransferClient.Push(...)`
3. `task/mq` 消费 Kafka 后：
    - 先落库：`addChatLog()` 写 `ChatLogModel`（Mongo），并且按 `MsgId` 做了幂等（`FindByMsgId`）
    - 更新会话最后一条消息：`ConversationModel.UpdateMsg(...)`
    - 再推送：调用 `WsClient.Send(Method:"push")` 推给 ws 服务做在线推送

所以哪怕断线/对方不在线：
- 消息已经在 Kafka 和 Mongo 里（不会因为 WS 断了就丢）
- WS 推送失败最多是“没实时到达”，不影响后续补拉

###### 2.2 在线推送是“尽力而为”
`push.single()` 里如果 `GetConn(receiverId)` 为空就直接返回，说明：
- 对方不在线就不推
- 依赖后续“离线补拉”拿到消息

这也是 IM 常见设计：实时通道不承诺送达，送达靠存储和补偿。

###### 离线消息怎么补拉
你项目有拉历史消息的 RPC/API：
- `Im.GetChatLog`：按 `conversationId` + 时间范围分页拉取（`startSendTime/endSendTime/count`），或者按 `msgId` 查单条  
    代码在 `apps/im/rpc/internal/logic/getchatloglogic.go`、`apps/im/api/internal/logic/getchatloglogic.go`

因此“补拉离线消息”的推荐说法是：
> 断线重连后，客户端先重建会话列表（GetConversations），再对每个会话用 GetChatLog 按时间/游标分页补拉；最后恢复实时 WS 收发。

（你仓库里 `GetConversations` 也有，但“游标=lastAckSeq”这种更严格的离线补拉机制在你当前实现里还没完全做成标准形态，你可以用“按时间窗口/分页补拉”来讲，符合现状。）

##### 离线消息补拉存在的问题
你现在有 `GetChatLog(conversationId, startSendTime, endSendTime, count)`，能拉历史消息。  
问题是：缺“统一游标”，重连后补拉容易靠时间窗口，可能重复/漏拉/乱序处理复杂。

把离线补拉从“按时间查”升级为“按游标查”：
- 服务端为每个会话消息维护 `seq`（单调递增）
- 客户端维护 `lastAckSeq`
- 重连后请求：`list where seq > lastAckSeq limit N`
- 拉完更新 `lastAckSeq`

这样就很稳定：不会漏、重复易去重、顺序清晰。

###### 具体完善
###### 1) 消息模型加 `seq`
在消息落库结构（`ChatLog`）里增加：
- `seq`（int64）
- 并在同会话内唯一（`conversationId + seq`）

分配方式：
- 简单做法：`Conversation` 里维护 `nextSeq`，发消息时原子+1后写入消息。

###### 2) 补拉接口改为游标接口
新增接口（保留旧接口兼容）：
- `PullOfflineMsgReq`:
    - `conversationId`
    - `afterSeq`（客户端已确认的最后序号）
    - `limit`
- `PullOfflineMsgResp`:
    - `list`（按 seq 升序）
    - `maxSeq`
    - `hasMore`

查询条件：
- `conversationId = ? AND seq > afterSeq`
- `ORDER BY seq ASC LIMIT ?`

###### 3) 客户端重连补拉流程
重连成功后：
1. 先拉会话列表（拿每会话未读信息/最新 seq）
2. 对每个会话执行离线补拉：`afterSeq = localLastAckSeq`
3. 收到消息按 seq 连续应用
4. 更新本地 `lastAckSeq = max(lastAckSeq, msg.seq)`
5. 拉到 `hasMore=false` 再切实时流

###### 4) 幂等与顺序保证
- 幂等：客户端按 `msgId` 或 `seq` 去重（`seq <= lastAckSeq` 直接丢）
- 顺序：服务端返回升序，客户端按 seq 应用
- 补洞：若收到实时消息 seq=105 但本地到102，先触发补拉 `[103,104]`

###### 多端建议使用"两层游标"
- 设备游标 `deviceLastAckSeq`：表示该设备已经收到并落地到哪条。
- 用户已读游标 `userReadSeq`：表示这个用户真正读到哪条（跨设备共享）。

这样做的结果：
- A 设备离线，B 在线也不影响 A 之后补拉。
- B 已读到 200，可以同步把 A 的会话未读也更新到 200（看你产品策略）。

#### 多端同步如何做（同一用户多设备在线）？消息顺序如何保证？
##### 1) 连接层：先支持“同一用户多连接”
你当前是单连接（新端挤掉旧端），先改这个。
- 映射结构从  
    `userId -> Conn`  
    改成  
    `userId -> map[deviceId]*Conn`
- 握手时带 `deviceId`、`clientType`（ios/android/web/pc）
- 服务端保存 `conn -> {userId, deviceId}` 元信息
- 下线清理按 device 维度，不影响同用户其他端

##### 2) 消息层：建立会话内全局序号 `seq`
这是顺序和补拉的核心。
- 每条消息写入时分配 `conversationSeq`（会话内单调递增）
- 消息体必须带：`msgId + conversationId + seq + sendTime`
- Kafka 分区键尽量用 `conversationId`（减少乱序）
- 存储加唯一约束：`(conversationId, seq)`、`msgId`（幂等）

##### 3) 同步层：多端同步策略
定义清楚“哪些要多端同步”：
- 聊天消息：同用户所有在线设备都推（可配置是否排除发送端）
- 已读回执：同步到该用户所有设备（统一未读）
- 会话变更（置顶/免打扰等）：广播到该用户所有设备

实现方式：
1. 消息落库成功后，查 `userId` 在线设备列表
2. 对每个设备投递同一条消息（同一个 `seq`）
3. 每设备独立 ACK（可做投递统计）
4. 离线设备后续按游标补拉

##### 4) 离线补拉：按游标，不按时间
- 每设备维护 `deviceLastAckSeq[conversationId]`
- 重连后调用：`PullOffline(conversationId, afterSeq=deviceLastAckSeq, limit)`
- 返回按 `seq ASC`，客户端应用后推进游标
- `seq <= lastAckSeq` 直接丢弃（去重）

##### 5) 顺序保证怎么做（重点）
你要明确回答：只保证会话内有序，不保证全局有序。
- 服务端保证：会话内 `seq` 单调递增
- 传输层保证：同会话尽量同分区
- 客户端保证：
    - 收到 `seq == last+1`：直接展示
    - 收到 `seq <= last`：重复丢弃
    - 收到 `seq > last+1`：短暂缓冲 + 触发补拉缺口

#### 现在是单用户单连接，要改成同一用户多设备在线，要怎么改？
- 把映射从`userId -> Conn`  改成 `userId -> {deviceId -> Conn}`（或 `[]Conn`，但建议带 deviceId）
-  建议实现
	- 握手时除了 token，再传 `deviceId`（移动端/PC/Web）和 `clientType`。
	- 服务端维护：
		- `connToSession[conn] = {userId, deviceId, clientType}`
		- `userToConns[userId][deviceId] = conn`
	- 路由发送策略改为可配置：
		- 单端投递：只投当前设备
		- 多端同步：投该用户所有在线设备
		- 排除发送端：群聊常见（自己发的不再推回自己，或只回执）
	- 下线清理要精确到 device 级别，不是整用户级别。
	- Redis 可加一层在线索引（`online:{userId}:{deviceId}`）用于跨节点查询。

### 3) 消息可靠性与一致性
- Kafka 里如何选 topic/partition key？为什么这么选？
	- 先确定Topic边界（按事件类型，不按接口）：
		- `msg_chat_transfer`（聊天消息）
		- `msg_read_transfer`（已读回执）
		- 这样消费逻辑解耦，权限、重试、监控也更清晰
	- 再确定Partition key（按顺序需求）：
		- IM的关键是“会话内有序”，所以聊天消息key选`conversionId`
		- 原因：
			- 同key会路由到同分区，天然更容易保证顺序
			- 比按sessionId/receiverId更贴近会话语义
			- 已读回执也建议用 `conversationId`，确保状态推进与消息顺序一致。
	- 最后看负载均衡与热点：
		- 分区数要大于或等于预期消费并发
		- 如果某些超级群导致热点，会考虑“群会话分片key+客户端seq重排”做折中
		- 先保证顺序，再处理热点
- 如何保证消息“至少一次/至多一次/恰好一次”语义？
	- 至少一次（At-Least-Once）怎么做？
		- 可能重复，但尽量不丢
			- Producer 发送失败重试（带超时、退避）
			- Consumer 先处理业务，成功后再提交 offset
			- 消费失败不提交 offset，让消息可重投
			- 业务侧做幂等：`msgId` 唯一约束/去重表/Redis setnx
	- 至多一次（At-Most-Once）怎么做？
		- 不重复，但可能丢
			- Consumer 先提交 offset，再处理业务
			- 一旦处理过程异常，消息不会再来（可能丢）
			- 适合日志、埋点等可容忍丢失场景，不适合核心 IM 消息
	- 恰好一次（Exactly-Once）怎么做？
		- 既不丢也不重
			- Kafka 事务（幂等 producer + transactional.id）
			- 消费-处理-生产在同一事务边界（Kafka Streams 这类更容易）
			- 落库侧还要幂等/唯一约束（因为 DB 不在 Kafka 事务里）
- 消息入库、投递、ACK 三者顺序怎么设计？
	1. 客户端发消息到 WS（`conversation.chat`）
	2. WS 服务先写 Kafka（`msgChatTransfer`）
	3. `task/mq` 消费后：
	    - 先做幂等检查（按 `MsgId`）
	    - 先入库 `ChatLog`
	    - 更新会话最后一条消息/会话信息
	    - 再通过 WS 客户端发 `push` 做在线投递
	4. ACK 现状：
	    - 你有 WS 层 `OnlyAck/RigorAck`，这是“连接层 ACK”（帧级确认）
	    - 另有 `msgReadTransfer` 用于“已读回执”
	    - 但“投递成功 ACK（receiver delivered）”还不是完整的持久化闭环
	- 优点：
		- 先入库再投递，实时推送失败也不丢消息（高可靠性）
		- Kafka解耦写入和推送，抗峰值更稳
		- MsgId幂等避免重复消费导致重复入库
	- 优化：
		- 补齐 ACK 分层语义
		    - `send_ack`：服务端已接收并入队
		    - `deliver_ack`：目标设备已收到
		    - `read_ack`：用户已读  
		        现在你主要有连接 ACK + read ACK，建议补“deliver ACK”。
		- 投递状态持久化  
		    建一个消息状态表（或扩展字段）：`msgId, deviceId, sent/delivered/read, updateTime`，方便重试与对账。
		- 失败重投机制  
		    push 失败不要只返回 nil，建议入“重投队列 + 退避重试 + DLQ”，重试超限转离线补拉。
		- 离线补拉游标化  
		    用 `conversationSeq/lastAckSeq` 替代纯时间窗口，保证不漏不重、顺序更稳。
		- 多端 ACK 策略  
		    区分“任一端送达即送达”还是“全部端送达才算送达”，按产品需求定义，避免状态混乱。
- 幂等如何做（messageId、去重表、Redis setnx）？
	- 消息入库幂等（主链路）
- 乱序消息怎么处理？客户端还是服务端解决？

### 4) 存储设计（MySQL + Redis）
- 会话表、消息表、未读计数表怎么建模？索引怎么设计？
- 热点会话/热点群聊如何扛住（分表、缓存、批量写）？
- Redis 在系统里承担哪些角色（缓存、在线状态、分布式锁、限流）？
- 缓存一致性如何处理？穿透/击穿/雪崩怎么防？

### 5) 高可用与治理（APISIX）
- 限流策略怎么配（按用户/IP/接口）？令牌桶参数如何定？
- 熔断和降级触发条件是什么？降级后用户体验怎么保证？
- 网关层做了哪些安全策略（鉴权、黑名单、重放防护）？

### 6) 故障与运维

## 你必须吃透的“重点知识”
- 消息生命周期：发送 -> 入队 -> 持久化 -> 投递 -> ACK -> 已读/未读更新（每一步失败怎么补偿）。
- 可靠性语义：明确你们系统是“至少一次”，并说明重复消息如何被幂等消化。
- 顺序性边界：是“全局有序”还是“会话内有序”；为什么不能全局有序。
- 离线消息策略：存多久、怎么拉取、怎么清理、客户端断点续拉机制。
- 连接稳定性：心跳、重连退避、会话恢复、服务端连接迁移。
- 高并发治理：限流阈值如何评估，熔断指标如何设定，降级路径是什么。
- 容量规划：峰值连接数、消息吞吐、Kafka 分区数、MySQL 写入上限估算。

## 一、IM 项目面试 30 问标准答案（口语化）

### 1) 你这个 IM 项目整体是怎么设计的？

**答：**我用 `go-zero` 做微服务拆分，核心有消息服务、会话服务、关系服务和推送链路。外部流量先过 APISIX，内部通过 gRPC 通信，服务注册发现走 Etcd。消息链路上用 Kafka 做削峰和解耦，MySQL 做持久化，Redis 做热点缓存和在线状态。

### 2) 为什么选 go-zero？

**答：**它对微服务比较开箱即用，像服务治理、gRPC、日志链路这些都比较完整。对我来说能更快把精力放在业务链路，不用自己拼太多基础设施。

### 3) 服务拆分的边界怎么定的？

**答：**按“职责和数据一致性边界”拆。消息收发和落库是一个边界，会话列表是一个边界，好友关系是一个边界。这样每个服务只关心自己模型，避免大泥球。

### 4) Etcd 在你们里面起什么作用？

**答：**主要做服务注册发现。服务实例启动时注册，异常退出会自动摘除。调用端拿到可用节点后做负载均衡，减少硬编码地址带来的维护成本。

### 5) 内部为什么用 gRPC，不用 HTTP？

**答：**内部高频调用更看重性能和类型约束，gRPC 在序列化效率和接口规范上更合适，延迟和吞吐都更稳，跨服务协作也更清晰。

### 6) WebSocket 握手鉴权怎么做？

**答：**连接建立时带 token，网关或接入层先验签，解析出 userId 写入连接上下文。无效 token 直接拒绝，过期 token 走刷新或重登逻辑。

### 7) 心跳机制怎么设计的？

**答：**客户端定时 ping，服务端维护最近活跃时间。连续超时就判定离线并清理连接。间隔和超时时间会按网络环境调，尽量减少误踢。

### 8) 断线重连后怎么保证消息不丢？

**答：**客户端重连会带上上次确认到的消息游标，服务端按游标补拉离线消息。在线推送失败也会落离线池，确保最终可达。

### 9) 多端同步是怎么做的？

**答：**一个用户可以绑定多个连接（手机/PC）。消息写入后会按用户维度分发到所有在线端，已读状态再做端间同步。

### 10) Kafka 在链路里具体解决什么问题？

**答：**把“前台写入压力”和“后台处理耗时”拆开。发送请求先快速入队，后面异步做持久化和分发，峰值时能明显减少抖动和超时。

### 11) Kafka 的 partition key 怎么选？

**答：**通常按会话 ID 或接收方 ID 选，目标是同一会话尽量进同一分区，便于保证会话内顺序，同时兼顾分区均衡。

### 12) 你们保证的是哪种消息语义？

**答：**工程上是“至少一次”。所以我会做幂等，允许重复投递但不允许重复入库或重复业务生效。

### 13) 幂等怎么做？

**答：**每条消息有全局 messageId，消费侧先做去重判断，比如 Redis setnx 或落库唯一索引。命中重复就直接跳过后续处理。

### 14) 顺序性怎么保证？

**答：**不追求全局有序，成本太高。我们保证“单会话内有序”，通过 partition key + 客户端 seq 校验来控制乱序影响。

### 15) 消息状态（已发送/已送达/已读）怎么设计？

**答：**状态分层处理：发送成功是服务端接收，送达是对端设备收到，已读是用户行为回执。这三个状态分开存，便于追踪。

### 16) MySQL 表结构你怎么考虑的？

**答：**消息表重点是会话维度查询，所以 `(conversation_id, seq/time)` 做核心索引；会话表存最后一条消息和未读计数，列表查询走覆盖索引。

### 17) Redis 在项目里怎么用？

**答：**主要三块：在线状态缓存、热点会话缓存、以及限流/幂等辅助。高频读走 Redis，最终一致性回落 MySQL。

### 18) 缓存一致性怎么处理？

**答：**优先“先写库再删缓存”或延迟双删，降低脏读概率。极端并发下允许短暂不一致，通过过期时间和回源兜底。

### 19) APISIX 你做了哪些治理？

**答：**做了统一鉴权、限流、熔断和降级。高峰时先保护系统，再保证核心接口可用，避免级联故障。

### 20) 限流策略怎么定？

**答：**按用户和接口维度做令牌桶，阈值来自压测结果和容量评估。先保登录、收发消息等核心链路，边缘能力优先降级。

### 21) 熔断降级在 IM 里怎么体现？

**答：**当下游超时或错误率升高时，先熔断非核心调用，比如某些扩展能力；核心收发链路走简化流程，保证主路径可用。

### 22) 离线消息怎么存和取？

**答：**消息先持久化，用户上线时按游标分页拉取，拉完再更新游标。拉取失败可重试，保证可恢复。

### 23) 如何防止重复消费/重复推送？

**答：**消费端幂等 + 推送回执校验。只要 messageId 已处理就不重复生效，推送层也会检查状态避免重复通知。

### 24) 你们怎么做压测？

**答：**分层压：连接层、发送链路、消费链路、数据库。观察 QPS、RT、错误率、Kafka lag、DB 慢查询，逐个定位瓶颈。

### 25) 系统里最容易成为瓶颈的是哪？

**答：**一般是消息落库和热点会话读写。我会用批量写、索引优化、缓存前置和异步化去分摊压力。

### 26) 线上你遇到过什么问题？

**答：**遇到过高峰时消费堆积，导致离线消息延迟。处理方式是扩 consumer 并发、优化慢 SQL、临时提高分区消费能力，最后把 lag 拉平。

### 27) 你如何保证高可用？

**答：**多实例部署、网关限流熔断、消息链路异步解耦、关键状态持久化。即使某个节点故障，流量也能切到健康节点。

### 28) 监控你重点看哪些指标？

**答：**连接数、消息吞吐、端到端延迟、失败率、Kafka lag、MySQL 慢查询和 Redis 命中率。告警阈值按历史峰值和 SLO 定。

### 29) 安全方面做了什么？

**答：**鉴权、签名校验、接口限流、防重放（nonce+timestamp）、敏感操作审计。核心是防止伪造请求和恶意刷接口。

### 30) 如果让你再做一版，你会优化什么？

**答：**我会优先做三件事：消息链路全链路追踪、热点群聊专项优化、以及更细粒度的降级开关，让故障面更可控。

## 你必须重点吃透的 8 个点（最容易被深挖）

- 1）分布式锁正确姿势：`SET key value NX EX` + Lua 校验 value 再删除，避免误删他人锁。
- 2）幂等设计：前端请求幂等键 + 后端唯一索引 + 业务层去重，三层兜底。
- 3）缓存一致性：常见推荐“先写 DB，再删缓存”；并理解延迟双删适用场景。
- 4）JWT 生命周期：访问 token 短期、刷新 token 长期，配 Redis 黑名单/会话版本号。
- 5）扫码登录安全：二维码状态机（待扫码/已确认/已过期），一次性消费，短 TTL。
- 6）事务边界：哪些步骤必须同事务，哪些要异步补偿。
- 7）索引与唯一约束：把“不能重复打卡”落实到数据库层，不只靠代码判断。
- 8）性能数据：准备 2-3 个量化指标（如并发、响应时间、错误率、优化前后对比）。

追问：被转发了怎么办？  
示例回答：  
链接携带一次性随机码，不直接暴露用户身份。随机码短 TTL、单次消费，绑定发起端上下文（如设备或会话），消费后立即删除。即使被转发，过期或状态不匹配也无法登录。

---

## 9) 打卡为什么需要分布式锁？

追问：不用锁会怎样？  
示例回答：  
高并发下同一用户短时间重复请求会造成并发写冲突，出现重复记录或状态覆盖。锁用于串行化同一业务键的写操作，保证同一时刻只有一个请求修改核心数据。

---

## 10) 锁的 key 怎么设计？

追问：锁粒度太大怎么办？  
示例回答：  
我们按业务唯一维度设计，比如 `checkin:lock:userId:date:sceneId`。这样只串行“同用户同场景同时间窗”的请求，不会阻塞无关用户，兼顾一致性和吞吐。

---

## 11) 锁如何避免误删？

追问：A 超时后 B 拿到锁，A 释放会误删吗？  
示例回答：  
加锁时写随机 value，释放时用 Lua 脚本校验 value 一致才删除。这样即使锁过期后被别人拿到，旧请求也删不掉新锁，避免误删。

---

## 12) 锁超时时间怎么定？

追问：太短或太长有什么问题？  
示例回答：  
按业务耗时上限加缓冲，比如核心逻辑 P99 + 安全边际。太短会提前过期导致并发问题，太长会增加等待。我们通过监控锁等待时间和冲突率持续调优 TTL。

---

## 13) 只靠分布式锁够吗？

追问：锁挂了怎么办？  
示例回答：  
不够。我们是三层防线：分布式锁、业务幂等、数据库唯一约束。即使锁异常，DB 唯一索引仍能兜底，确保最终不出现重复数据。

---

## 14) 幂等怎么做的？

追问：重复请求如何识别？  
示例回答：  
请求带幂等键（或按用户+日期+场景生成业务幂等键），先查 Redis/DB 是否已处理。已处理直接返回首次结果，未处理才执行业务并记录状态，避免重复写。

---

## 15) 重复打卡你们怎么定义？

追问：规则在哪层实现？  
示例回答：  
业务层有规则判断（时间窗、场景、用户状态），数据库层有唯一约束强兜底。规则可配置，约束保证数据底线，避免只靠代码判断带来的竞态漏洞。

---

## 16) 事务怎么设计？

追问：哪些放同一事务？  
示例回答：  
强一致步骤（如打卡记录写入+统计主表更新）放一个事务。缓存更新不放事务内，采用提交后删缓存/异步刷新，减少长事务和锁表风险。

---

## 17) 你们缓存策略是什么？

追问：为什么这样选？  
示例回答：  
主要用 Cache Aside。读：先查缓存，miss 再查 DB 回填。写：先写 DB，再删缓存。这个模式实现简单、可控，适合读多写少的签到查询场景。

---

## 18) 为什么写后删缓存，不是更新缓存？

追问：删缓存会有短暂 miss。  
示例回答：  
更新缓存需要保证构建逻辑完全一致且处理并发覆盖，复杂度高。删缓存更稳，下一次读自然回源重建。对热点数据可配合异步预热降低 miss 抖动。

---

## 19) 怎么处理缓存击穿？

追问：热点 key 过期瞬间怎么办？  
示例回答：  
对热点 key 加互斥重建（singleflight/锁），同一时刻只允许一个请求回源；并使用逻辑过期或提前刷新机制，避免大量请求同时打到 DB。

---

## 20) 怎么处理缓存穿透？

追问：查不存在的数据怎么办？  
示例回答：  
参数合法性校验 + 布隆过滤器（可选）+ 空值缓存（短 TTL）。这样恶意或无效请求不会每次都打到数据库。

---

## 21) 怎么处理缓存雪崩？

追问：大量 key 同时过期怎么办？  
示例回答：  
TTL 加随机抖动，避免同一时刻集中失效；关键接口做限流降级；热点数据做预热。多层手段结合，防止瞬时流量击穿 DB。

---

## 22) MySQL 索引怎么设计？

追问：最关键索引是哪几个？  
示例回答：  
围绕高频查询条件建联合索引，比如 `user_id + checkin_date`，统计场景按时间和组织维度建索引。并加唯一索引约束同用户同场景同时间窗唯一打卡。

---

## 23) 统计接口慢怎么优化？

追问：你做过哪些优化？  
示例回答：  
先用慢 SQL 日志定位，再做索引调整、减少回表、避免大范围扫描。高频统计结果做 Redis 缓存或按天预聚合，接口响应从秒级降到百毫秒级。

---

## 24) 你们做过压测吗？

追问：结果如何？  
示例回答：  
做过高峰场景压测，重点看成功率、P95/P99、DB/Redis 负载、锁冲突率。通过锁粒度优化和缓存命中提升，峰值下错误率明显下降、延迟更稳定。  
（这里最好补你真实数字）

---

## 25) 限流做了吗？

追问：恶意刷接口怎么办？  
示例回答：  
对登录和打卡入口做了基于用户/IP 的限流，超阈值返回友好错误。关键是保护下游 DB 和 Redis，避免被突发流量拖垮。

---

## 26) 日志怎么设计？

追问：如何定位线上问题？  
示例回答：  
按请求维度打结构化日志，包含 traceId、userId、接口耗时、错误码、关键业务参数。出现问题可按 traceId 串联网关到业务层快速定位。

---

## 27) 线上遇到过什么故障？

追问：你怎么处理的？  
示例回答：  
遇到过高峰期重复打卡投诉。排查发现并发竞争导致重复写入窗口。处理上先临时限流，再上线“锁+幂等+唯一索引”组合方案，回放验证后问题消失。

---

## 28) 你怎么保证可维护性？

追问：代码层面做了什么？  
示例回答：  
按 MVC 分层，handler 只做参数和响应，核心业务在 logic，DAO 负责数据访问。公共错误码、日志、鉴权中间件统一抽象，减少重复代码和耦合。

---

## 29) 测试怎么做？

追问：如何验证并发正确性？  
示例回答：  
单元测试覆盖规则判断，集成测试覆盖打卡主链路。并发场景用压测和并发测试脚本验证“无重复记录、状态一致、失败可重试”，同时监控锁冲突和异常率。

---

## 30) 如果重做一次，你会怎么升级？

追问：下一步优化方向？  
示例回答：  
我会补三点：  
1）引入消息队列做异步统计与削峰；  
2）完善可观测性（指标+告警+链路追踪）；  
3）把配置和规则中心化，支持多租户和更灵活考勤策略。  
目标是进一步提升扩展性和高峰稳定性。

---

## 额外建议：面试时最加分的表达

- 尽量说真实指标：如“P95 从 320ms 降到 140ms，重复打卡率降到 0”。
- 每个点尽量带一次故障/优化案例。
- 避免只说“用了 Redis/JWT”，要说“为什么用、怎么用、踩过什么坑”。

## 1) 认证：JWT 为什么配 Redis？

二层回答：  
JWT 只解决“身份可验证”，不解决“状态可撤销”。我们在 Redis 里维护 `session_version` 或 token 黑名单：

- 用户主动退出：服务端写失效标记；
- 修改密码/封禁：提升版本号，旧 token 全失效；
- 网关鉴权时做“验签 + Redis 状态校验”。  
    这样兼顾性能和可控下线能力。

---

## 2) token 续期怎么防滥用？

二层回答：  
刷新 token 只允许单用途或短窗口复用，且绑定设备指纹（或会话 ID）。每次刷新都会轮换 refresh token（rotate），旧 refresh token 立即失效。  
如果检测到旧 refresh token 被再次使用，判定可能泄漏，直接失效整条会话链。

---

## 3) 扫码登录如何防重放？

二层回答：  
核心是“一次性 + 状态机 + 时效”：

- Redis 存二维码会话状态：待扫码 -> 待确认 -> 已确认/已过期；
- 会话 ID 随机且短 TTL；
- 确认成功后 Lua 原子改状态并消费，后续重复提交直接拒绝。  
    即使抓包重放，也会因状态已消费失败。

---

## 4) 分布式锁为何不用 Redlock？

二层回答：  
我们场景是单 Redis 主从/哨兵，优先“工程可控和简单稳定”。  
在这个前提下，用 `SET NX EX + 随机 value + Lua compare-and-del` 足够。  
如果未来升级到跨机房强一致诉求，再评估 Redlock 或者 DB 悲观锁/队列串行化。

---

## 5) 锁过期导致并发穿透怎么办？

二层回答：  
三层兜底：  
1）锁 TTL 按 P99 业务耗时 + buffer 设置；  
2）关键路径控制执行时长，避免超长事务；  
3）数据库唯一索引最终兜底。  
即使锁偶发提前过期，也不会写出重复数据。

---

## 6) 为什么还要数据库唯一索引？

二层回答：  
因为“任何中间件都可能抖动”，最终一致性要落在数据库约束。  
应用层锁和幂等是降低冲突概率，唯一索引是最后防线。  
我们会把唯一索引冲突转换成业务成功/重复提示，避免给用户报 500。

---

## 7) 幂等键具体怎么落地？

二层回答：  
优先业务幂等键（用户ID+日期+场景）而不是随机请求 ID。  
随机 ID 只保证“同一次请求重试”，业务幂等键保证“同一业务动作唯一”。  
存储上 Redis 做短期去重，MySQL 唯一键做长期兜底。

---

## 8) 缓存一致性为什么“先写库再删缓存”？

二层回答：  
这是实践里最稳妥的折中：

- 先删缓存再写库会有并发脏读窗口；
- 写库后删缓存，最多出现短暂 cache miss，不会长期脏数据。  
    对极端并发更新，可加“延迟双删”或 binlog 异步修正。

---

## 9) 延迟双删什么时候需要？

二层回答：  
当你遇到“写库后，恰好有旧值回填缓存”的竞态风险时使用。  
流程：写库 -> 删缓存 -> sleep 短时间 -> 再删一次。  
我们只在高冲突热点键上用，不全量用，避免不必要复杂度。

---

## 10) 缓存击穿怎么证明有效？

二层回答：  
看三个指标：

- 回源 QPS 是否被限制到接近 1 条重建链；
- DB 瞬时连接数/慢 SQL 是否下降；
- 接口 P95 抖动是否收敛。  
    有数据对比才算“处理了击穿”，不是只说“加了锁”。

---

## 11) 缓存穿透除了空值缓存还有什么？

二层回答：  
先做参数合法性校验，过滤明显无效请求；再加布隆过滤器挡不存在主键；最后用空值缓存拦截重复穿透。  
空值 TTL 设短，避免新数据创建后长期读不到。

---

## 12) 雪崩怎么做容量保护？

二层回答：  
不是只靠 TTL 抖动，还要有“降级开关”。  
当 Redis/DB 指标异常时，非核心统计接口直接返回兜底结果或稍后重试，优先保证打卡主链路可用。

---

## 13) 事务边界怎么划？

二层回答：  
“影响业务真相”的数据放事务内（打卡记录、状态主表），  
“可重建/可补偿”的动作放事务外（缓存刷新、通知、统计明细异步化）。  
原则是：短事务、少锁行、失败可补偿。

---

## 14) 统计接口慢你怎么排查？

二层回答：  
流程固定：  
慢日志定位 SQL -> `EXPLAIN` 看 type/rows/回表 -> 调整索引与查询条件 -> 压测回归。  
如果本质是大范围聚合，做离线/准实时预聚合，不硬扛在线实时算。

---

## 15) 索引设计你如何避免“看起来有索引但没用上”？

二层回答：  
遵循最左匹配，避免对索引列做函数计算，避免隐式类型转换。  
同时检查排序字段是否和索引一致，尽量让查询走覆盖索引减少回表。

---

## 16) 高并发下你先优化了什么？

二层回答：  
优先做“低风险高收益”：  
1）热点接口缓存命中率；  
2）SQL 索引和慢查询；  
3）锁粒度和冲突率。  
先把系统从“不可用”拉回“稳定可用”，再做精细优化。

---

## 17) 限流怎么避免误伤正常用户？

二层回答：  
多维度限流：用户维度、IP 维度、接口维度分开配额；对登录和打卡采用不同阈值。  
命中限流后返回明确错误码和重试时间，且对内部白名单保留通道。

---

## 18) 线上故障你怎么止血？

二层回答：  
先止血再定位：

- 开限流/降级，保护 DB；
- 关闭高风险非核心功能；
- 快速回滚最近变更；
- 同步排查日志和指标。  
    恢复后再做根因分析和防复发改造。

---

## 19) 可观测性你怎么做闭环？

二层回答：  
三件套：指标（QPS/延迟/错误率）+ 日志（结构化带 traceId）+ 告警（阈值和趋势）。  
目标不是“看见问题”，而是“5 分钟内定位到模块，15 分钟内给出止血动作”。

---

## 20) 你怎么证明“可靠性提升了”？

二层回答：  
用变更前后对比：

- 重复打卡率、接口错误率、P95/P99、锁冲突率；
- 高峰时段工单量。  
    有对比数据和时间窗口，面试官才会相信“不是主观感觉”。

---

## 21) 并发测试怎么设计才靠谱？

二层回答：  
不只压 QPS，还要压“同一用户同一时段重复请求”这种冲突型流量。  
验证点：最终记录条数、状态一致性、唯一索引冲突处理是否符合预期。

---

## 22) 如果 Redis 宕机怎么办？

二层回答：  
降级策略：

- 认证：短时间允许仅验签（风险可控窗口）或强制重新登录；
- 打卡：依赖 DB 唯一索引兜底，必要时收紧流量。  
    同时快速切换哨兵/集群并恢复缓存预热。

---

## 23) 如果 MySQL 压力过高怎么办？

二层回答：  
短期：限流、缓存加大 TTL、降级统计接口。  
中期：读写分离、热点表索引优化、批处理异步化。  
长期：按业务维度分库分表或引入 OLAP 做统计分流。

---

## 24) MVC 在你们项目怎么落地？

二层回答：  
Controller 只做参数校验和响应映射；Logic 承载业务规则和事务；DAO 只做数据访问。  
这样测试和维护都更清晰，避免“handler 写满业务代码”的失控状态。

---

## 25) 你做过哪些安全控制？

二层回答：  
密码加盐哈希、JWT 密钥轮换策略、敏感接口鉴权与限流、关键日志脱敏、SQL 参数化防注入。  
扫码/链接登录走一次性凭证和短时效，降低被盗链风险。

---

## 26) 如何处理时区和打卡跨天问题？

二层回答：  
统一服务端时区（通常 UTC 存储，本地展示转换），业务日界线按租户或组织配置。  
跨天判断只基于服务端时间和规则，不信任客户端时间。

---

## 27) 为什么不用消息队列做所有异步？

二层回答：  
核心写路径要先保证“业务真相落库”，不能全部异步。  
队列更适合统计、通知、报表等可延迟模块。  
先同步保证正确，再异步做扩展。

---

## 28) 你怎么做灰度发布？

二层回答：  
按接口或用户分组灰度，先小流量观察错误率和延迟。  
关键指标稳定后逐步放量，出现异常可一键回滚到旧版本。

---

## 29) 你在团队里怎么协作推进？

二层回答：  
我会先把问题拆成“风险、收益、工期”三维度，给出可落地方案。  
推进时先做最小闭环验证，再逐步扩展，降低一次性大改失败风险。

---

## 30) 如果让你再优化一个版本，你先做什么？

二层回答：  
第一优先级是可观测性和压测体系完善，因为它决定优化方向是否正确；  
第二是把统计链路异步化，进一步提升高峰稳定性；  
第三是规则配置化，提高业务变化时的交付速度。

## 你必须准备清楚的“10个技术点”
- 性能基线：优化前后 RT（平均/P95）、QPS、数据量规模、并发量。
- 瓶颈拆解：DB 查询慢、N+1、排序开销、跨服务调用慢，分别如何验证。

- 降级策略：某个数据源失败时是空字段返回还是整体失败。

## 回答模板

### 1) “7s 到 0.8s 你怎么做到的？”
“先做 profiling，把耗时拆成 DB 查询、聚合排序、跨服务调用三段。DB 侧通过慢 SQL 治理和索引重建降低扫描与回表；服务侧把串行查询改成 `mr` 并行聚合；结果侧用 Redis 会话索引缓存减少重复排序计算。最终核心接口平均 RT 从 7s+ 到 0.8s。”

### 2) “为什么首查和滑动差这么多？”
“首查要全量构建排序索引，滑动刷新复用会话缓存，只拉当前页明细，所以首查更重。我们通过会话级索引+锁控重建，把首查压到 3s，滑动降到 600ms。”

### 3) “`mr` 比 `WaitGroup` 强在哪？”
“`WaitGroup` 只负责等待，不处理错误聚合和协程治理；`mr` 可以统一收敛错误、控制并发处理流程，代码可维护性和故障可控性更好。”

## 最后建议

把这 4 个数字准备成“可复述”：

- 优化前后 RT：`7s+ -> 0.8s`
- 首查/滑动：`14s+ -> 3s`，`4s+ -> 600ms`
- 数据规模：例如用户数、单次候选集规模
- 命中率/稳定性：缓存命中率、错误率或超时率变化

### 1) 你在实习里最核心的工作是什么？
答： 主要负责医疗管理平台中“数据授权 + 报告处理”核心域的后端建设，覆盖授权申请、变更、撤销、协议确认、结果回传、详情查询，以及报告查询、模板管理、PDF 生成等链路，并重点优化了用户态势查询的性能与稳定性。

### 2) 你说开发/改造了 35+ 接口，如何体现不是 CRUD？
答： 接口背后是完整业务闭环和状态机，不是单表增删改查。比如授权链路涉及多角色、多状态流转、协议确认、回传与审计查询；态势查询涉及多源数据聚合、排序索引缓存、分页会话和并发治理。

### 3) 核心查询接口为什么最开始会到 7s+？
答： 瓶颈主要在三块：
1. 多表慢 SQL（扫描行数大、索引命中差）；
2. 多源数据串行查询导致累计等待；
3. 每次请求重复做全量排序与聚合，缺少结果复用。

### 4) 你具体做了哪些优化把 7s+ 降到 0.8s？
答： 三层优化：
- DB 层：慢 SQL 治理、联合索引优化、条件裁剪；
- 服务层：把串行查询改为 `mr` 并行聚合；
- 缓存层：引入 Redis 会话级排序索引，减少重复计算。  
    最终关键接口平均耗时由 7s+ 降至 0.8s。

### 5) 为什么用 `mr`，不用传统 `WaitGroup`？
答： `WaitGroup` 只能等待，错误处理和收敛要自己维护。`mr` 在并发执行、错误传播、流程收敛上更统一，代码可维护性更高，失败场景也更好控。

### 6) 你态势查询为什么“首查 3s、滑动 600ms”？
答： 首查要做候选用户全量排序索引构建，计算量大；滑动刷新复用会话缓存，只取当前页明细，计算量小。所以首查和后续刷新耗时不同是链路设计决定的。

### 7) 态势查询的核心流程是什么？
答：
1. 获取候选用户并按关键词筛选；
2. 并行查询睡眠/体征/问卷等“排序时间”；
3. 计算每个用户最近活跃时间并排序；
4. 将排序后 ID 列表缓存到 Redis；
5. 按页截取 ID，再并行拉取详情并返回。

### 8) Redis 会话索引缓存如何设计？
答： 以 `userId + sessionId` 作为 key，value 存 keyword hash + 有序 userId 列表，并设置 TTL。关键词变化时 hash 不匹配直接重建；强制刷新时通过 `lastSessionId` 淘汰旧缓存并生成新会话。

#### 1. 为什么要带 `userId`（管理员）
- 缓存的是「某个后台用户看到的、有权限的候选用户排序结果」。
- 不同管理员权限范围、可见用户列表不同，排序的候选集也可能不同。
- 若 Key 里不带操作者，两个管理员可能共用一个 key，轻则串数据，重则越权看到别人的列表语义（取决于候选集是否按人过滤）。
所以：`userId` 做租户/操作者隔离。

---

#### 2. 为什么要带 `sessionId`（会话）
- 同一个人多次打开列表或强制刷新时，你希望的是新的一轮浏览：新的排序基准、新的分页上下文。
- 若只有 `userId` 一个维度，同一管理员永远共用一个 key：
    - A 窗口在翻页，B 窗口刷新/换条件，会互相覆盖同一份缓存；
    - 或者旧排序一直被复用，无法表达「新的一次会话」。
`sessionId` 表示「这一次打开/这一次浏览生命周期」，让多 Tab、多次进入、强制刷新可以各用各的索引，互不踩。

---

#### 3. 为什么关键词不放进 Key（而用 value 里的 `keyword_hash`）
- Key 若写成 `userId:session:keyword`，关键词一长、编码一多，key 会很长，且前端可能传法不一（空格、大小写），还要自己做规范化再拼 key。
- 当前做法是：Key 仍只表示「谁 + 哪次会话」；当前搜索词放进 value 的 `keyword_hash`，读的时候比对，不一致就当 miss 重建。
这样既控制 key 形态简单，又保证同 session 下换关键词不会误用旧列表。

### 优化
#### 1. Redis 单 key 存整段有序 ID，体积大
思路： 少存、拆存、压缩。
- 分页/分段存储：不要一个 key 存全量 `ids`，改成多段（如每段 500～1000 个 id）`situation:index:{user}:{session}:seg:{n}`，或用 Redis List 分段 `RPUSH`，按页算段号读取。
- 只存 id，不存冗余：已是 id 列表，避免再塞大 JSON；必要时对 id 做短编码（若业务允许）。
- 换 ZSET：`member=user_id`，`score=排序键`（需能维护 score），分页用 `ZREVRANGE`，避免超大 string。
- 限流会话：同一用户同时活跃 session 数上限 + TTL，防止无限开新 key。

---

#### 2. 首查 / 冷启动仍重
思路： 少算、异步、预热。
- 异步预热：首屏先快速返回「加载中」或第一页降级数据，后台任务构建索引完成后推送/轮询（要产品配合）。
- 预计算 `last_active_at`：离线或准实时写入汇总表，首查只做 ORDER BY + limit，会话索引只存小范围或甚至不需要全量列表。
- 缩小候选集：管理端先选机构/标签再进列表，减少 `userIDs` 规模。
- 构建阶段限流：单用户/单 IP 构建 QPS 限制，避免恶意刷冷启动。

---

#### 3. TTL 内顺序「冻结」，新鲜度差
思路： 缩短 TTL、显式刷新、版本化。
- 缩短 TTL（如 5 分钟）或按业务分档（高峰短、低峰长）。
- 强制刷新你们已有 `last_session_id`，可引导产品：刷新即新 session。
- 软失效：详情接口仍拉实时数据，仅排序用缓存；若产品要求排序也实时，只能缩短 TTL 或走预计算。
- 版本号：数据大变更时 bump `data_version`，key 带 version，自然淘汰旧索引。

---

#### 4. 分布式锁边角（误删、TTL、竞态）
思路： 规范锁实现 + 可观测。
- Lua 原子释放：`if get==token then del`，避免 GET/DEL 之间的竞态。
- 看门狗/续期（谨慎）：构建时间可能超过 TTL 时用 Redisson 类续期；你们有 45s 构建上限，锁 TTL 略大于构建上限即可。
- 监控：锁等待时长、构建失败率、`build_lock` 竞争次数。
- 单飞封装：统一用 `singleflight` 或封装好的 Redis lock 库，减少手写差异。

---

#### 5. 强依赖 Redis
思路： 降级路径 + 高可用。
- Redis 哨兵/集群 + 连接超时、重试策略。
- 降级：Redis 不可用时走 DB 直查慢路径（限流 + 超时），返回明确「服务繁忙」而非拖死。
- 本地短缓存（可选）：进程内 LRU 只缓存极小热点，注意多实例一致性。

---

#### 6. `session_id` 语义依赖前端
思路： 约定 + 兜底。
- 文档化：首屏不带、后续必带；刷新传 `last_session_id`。
- 服务端兜底：异常长时间未带 session 可视为新会话，避免脏读。
- 可选 HttpOnly Cookie 存 session（若同源），减少前端乱传（要评估安全域）。

---

#### 7. 多 session 冷启动叠加打 DB
思路： 全局削峰 + 合并构建。
- 同 keyword + 同候选集 的构建可考虑 全局单飞 key（除 session 外再加一层「候选集 fingerprint」），多 session 共享同一份排序结果（注意是否允许「快照共享」）。
- 全局限流 + 队列：构建任务进队列，worker 串行/限并发执行。
- 读扩展：MySQL 只读副本 承担列表构建读流量。

#### 1. Redis 里一条缓存太大（整份有序 ID 列表）

##### 你们在解决什么问题？
Redis 里 value 是一大串用户 ID。用户越多，这一串越长，占内存、网络传输慢，单次 `GET` 也可能慢。

##### 方案 A：分段存（最容易理解）
不要一个 key 存「全部 ID」，改成：
- 第 1～1000 个 ID 放 `...:part:0`
- 第 1001～2000 放 `...:part:1`
- …

翻第 5 页时，只要算出「这一页落在哪一段」，只读那一段，不用每次把整个大字符串拉下来。

类比： 一本书太厚，不要每次整本复印，按章节拆成多本小册子，需要哪章拿哪章。

##### 方案 B：用 Redis 有序集合 ZSET
Redis 提供 ZSET：每个用户一个 `member`，排序用一个 `score`（比如「最近活跃时间」转成的数字）。  
翻页时用 `ZREVRANGE` 按排名取一段，不需要自己存一整条 JSON 数组。

难点： 用户一有新数据，就要更新对应 score（写会变多），这是和「大 JSON 一次算完」的取舍。

类比： 不再是手里拿一张排好序的纸条，而是把名字贴在成绩榜上，按名次直接念前 10 名。

---

#### 2. 第一次打开列表还是很慢（冷启动 / 首查）

##### 你们在解决什么问题？
第一次没有缓存，必须：查多源数据 → 算每个人的「最近活跃」→ 排序，这一步再优化也不可能变成 0。

##### 方案 A：预计算「最近活跃时间」（治本方向）
平时用定时任务或数据变更时，把每个用户的 `last_active_at` 写进一张表（或字段）。  
打开列表时只做：按这个字段排序 + 取前 N 个，比临时从睡眠/体征/问卷三张里现算轻很多。

代价： 要维护「什么时候更新这个字段」，有一点延迟（例如 1 分钟级）。

类比： 不再每次考试现场算总分，而是平时小测成绩已经记在成绩单上，期末只按成绩单排名。

##### 方案 B：异步构建（体验换时间）
第一次请求先快速返回（例如「正在加载」或先给空/骨架屏），后台慢慢把索引建好；建好后前端再请求第二下拿列表。

代价： 交互变复杂，要产品接受。

类比： 饭店先给你号，菜在后厨做，好了叫号，而不是让你在柜台干等所有菜一次性全好。

##### 方案 C：缩小「要排序的人数」
进列表前先选：科室 / 机构 / 标签，候选用户从 5 万变成 500，排序自然快。

类比： 全年级排名改成「只在本班排名」。

---

#### 3. TTL 10 分钟内顺序不变（不够「实时」）

##### 你们在解决什么问题？
缓存是为了快，但 10 分钟内排序顺序可能和「此刻真实最新」不一致。

##### 方案 A：缩短 TTL
5 分钟、2 分钟……更实时，但重建更频繁，首查压力略升。

##### 方案 B：分清「排序」和「详情」
很多产品可以接受：名单顺序稍旧，但点进去的数据要新。  
你们可以是：排序用缓存，详情仍实时查库（你们详情本来就是按页再查的，这一点可以强调）。

若产品要求顺序也必须实时，就只能短 TTL 或预计算更频繁。

##### 方案 C：用户点「刷新」
你们已有 新 session / 删旧缓存，等于用户主动要一份新快照。

类比： 排行榜每小时更新一次；用户点「刷新」立刻重算一版。

---

#### 4. 分布式锁容易出事故

##### 你们在解决什么问题？
- 锁永远不释放 → 别人永远进不来（死锁）
- 删错锁 → 把别人的锁删掉，两个请求同时建索引（更乱）

##### 你们已经做的
- 过期时间 TTL：人没了，锁也会自动过期。
- 锁里写随机 token，删前比对：减少误删。

##### 还能加强的：Lua 原子删
现在逻辑是：先 `GET` 看是不是我的 token，再 `DEL`。  
中间若刚好锁过期被别人抢走，理论上仍有极小窗口出问题。  
用 一段 Lua 脚本在 Redis 里「判断 + 删除」一步完成，就没有这个缝隙。

类比： 取钱时「验指纹 + 出钱」要一次完成，不能验完指纹后隔十秒再出钱（中间可能被换人）。

---

#### 5. Redis 挂了怎么办
##### 你们在解决什么问题？
列表强依赖 Redis，Redis 不可用就全挂或全慢。

##### 方案 A：Redis 高可用
主从、哨兵、集群，减少单机故障。

##### 方案 B：降级路径
Redis 不行时：直接走数据库算列表（慢），同时 限流（例如每人每分钟只能触发一次），返回「系统繁忙」也比把 DB 打崩好。

类比： 电梯坏了走楼梯，但楼梯口要限流，避免所有人挤塌楼梯。

---

#### 6. session_id 依赖前端

##### 你们在解决什么问题？
服务端靠客户端传的 `session_id` 区分「这一轮浏览」。传错了就串页、串缓存。

##### 方案：约定 + 文档 + 简单兜底
- 和产品/前端写清楚：第一次不带，之后每页必带；刷新带 `last_session_id`。
- 服务端对明显异常（例如过期很久的 session）可以当新会话处理，避免无限错下去。

类比： 取餐号由顾客拿着，店员要反复核对号，号丢了就重新取号。

---

#### 7. 很多人同时「第一次打开」打爆数据库
##### 你们在解决什么问题？
锁只能保证同一个 Redis key 只有一个在构建；  
不同用户、不同 session 会对应不同 key，会很多个构建同时进行，DB 压力还是会叠上去。

##### 方案 A：队列 / 限流
把「建索引」变成任务进队列，后台 worker 控制并发（例如同时最多 5 个在算），多出来的请求排队或稍后再试。

类比： 医院挂号窗口只有 3 个，后面的人拿号排队，而不是所有人同时挤进诊室。

##### 方案 B：「同一份候选集」共享构建结果（进阶）
如果两个 session 关键词相同、候选用户集合相同，理论上排序结果可以一样。  
可以用 候选集的指纹（例如 hash）再做一个「全局缓存」，多个 session 共用同一份有序列表（要接受：同一时刻大家看到同一份快照）。

类比： 同一场电影多张票，放映内容是一份拷贝，不必每个座位单独拍一部电影。

#### 方案 A：MySQL 汇总表（`user_id` + `last_active_at` + 维度字段）
做法：  
每当睡眠/体征/问卷等上传或更新时（或在消费变更消息后），更新该用户在这张表里的 `last_active_at`（取各源最新时间的 max，或业务规则）。

建列表 / 建索引缓存时：  
只查这张表（+ 权限过滤），`ORDER BY last_active_at` 排序，再分页；需要时仍可把「当前页的 user_id」再去拉详情。

特点：
- 持久化，重启不丢；好审计、好排查。
- 和事务/业务容易对齐（同一套 DB）。
- 读路径简单，适合你们后面仍想用「有序 ID 列表 + Redis」时：可以先从这张表算出有序列表再塞 Redis，或直接 DB keyset 分页。

---

#### 方案 B：只维护 Redis ZSET（`ZADD user_id score`）
做法：  
数据变更时同步或异步 `ZADD`：`member = user_id`，`score = 最近活跃时间戳`（毫秒等）。  
列表分页用 `ZREVRANGE BY SCORE` 一类按排名取。

特点：
- 读极快，分页天然。
- 易丢：Redis 故障、flush、过期策略要小心；通常要能重建（从 DB 汇总表或全量重算）。
- 多租户/多管理员：若每人可见用户集不同，往往要 每个管理员一个 ZSET，或一个全局 ZSET + 再过滤（过滤成本高）。

---

#### 方案 C（常见生产组合）：DB 汇总表为真源 + Redis ZSET 为加速
做法：
1. 写入路径：业务写明细表 → 更新汇总表（事务或最终一致）。
2. 同时发消息或直接 更新 ZSET（可异步，短暂不一致）。
3. 读路径：优先 ZSET 取排名；miss 或 Redis 挂了 → 回源汇总表。

特点：  
既有可靠底账，又有Redis 速度；不一致时用 DB 纠偏或定时对账。

#### 很多人同时第一次打开或刷新，会让数据库压力激增，有什么方案
##### 你提的方案：分布式锁 + 带缓冲 channel

###### Channel 能做什么、不能做什么
- 带缓冲 channel + 固定数量 worker：在一个 Go 进程里限制「同时最多 N 个 goroutine」在做重任务——有效。
- 多实例部署时：每个实例各有一套 channel，全局并发 = N × 实例数，不能从整体上限制「全集群只有 5 个在构建」。
- 所以：channel 适合做单机内的 worker 池，不适合单独当「分布式限流」。

###### 分布式锁若用来「全局串行」
- 若一把大锁：全系统同一时刻只允许 1 个构建 → DB 压力是小了，延迟会炸（所有人排队）。
- 你们态势里已有「按 indexKey 的构建锁」，解决的是同一个缓存 key 不要并发重建，不是全站总共允许多少个构建。
结论：  
「锁 + channel」可以这样理解：channel 限单机并发；跨机器还要别的东西（见下文）。

---

##### 2. 更合适的几类方案（从简到强）
###### A. 单机：worker 池 + 有界 channel（你已有思路的正规版）
- 进程内：`jobs chan` + 固定 `N` 个 worker，所有「建索引」任务先丢进 channel。
- 作用： 保护本机的 DB 连接池、CPU，避免一次来 1 万个 goroutine。
- 部署多副本时： 每台都这样，总并发仍 = N×副本数，需配合 B 或 C。

###### B. 分布式：Redis 信号量 / 租约（全局「最多 M 个构建」）
- 用 Redis 实现计数信号量：`INCR` + 过期，或 Redisson 的 semaphore，保证全集群同时最多 M 个「重构建」在执行。
- 拿不到许可的请求：等待 / 快速失败 / 返回「请稍后重试」。
- 比一把全局互斥锁好： 是 M 路并发，不是 1 路。

###### C. 真正的消息队列（推荐「削峰」时用）
- Redis Stream / List、Kafka、RabbitMQ、NSQ 等：任务入队，独立 consumer 固定并发消费。
- 优点： 峰值写入队列，DB 按固定速率消化；可持久化、可重试、可观测。
- 缺点： 多组件、列表可能要异步（先 202 + 轮询）或稍延迟，产品要接受。

###### D. 网关 / 接口限流（最快落地）
- 对「态势列表首查 / 触发构建」做 QPS + 并发 限制（按用户、按 IP、按租户）。
- 超限：429 或友好文案，保护 DB 不被打穿。
- 常和 B、C 一起用：先挡流量，再控内部并发。

###### E. 读路径优化（治本）
- 预计算 last_active + 索引好的表 或 ZSET，让「第一次打开」不再是全量多源重算——比纯限流更能从根上减压。
- 限流是兜底，优化数据源是主菜。

#### 结合singleflight怎么使用
`singleflight` 用来解决：同一进程内，很多请求同时要算同一个 key 的结果时，只真正执行一次，其余等这一次的结果（和「合并重复请求」很像）。

---

##### 1. 它是什么、不是什么

| |`singleflight`|你们现在的 Redis 分布式锁|
|---|---|---|
|范围|单机、单进程内去重|多实例之间互斥|
|典型场景|同一台机器上 100 个 goroutine 同时 miss 同一个缓存|多台机器同时 miss 同一个 Redis key|

结论： 多实例部署时，`singleflight` 不能替代 Redis 锁；常用来和锁搭配：锁解决跨机，singleflight 解决同机内重复计算。

---

##### 2. 怎么用（标准写法）
包：`golang.org/x/sync/singleflight`
```GO
import "golang.org/x/sync/singleflight"

var sf singleflight.Group

func loadOrderedIDs(key string) ([]string, error) {

v, err, shared := sf.Do(key, func() (interface{}, error) {

// 这里只会对同一个 key 并发执行一次

ids, err := buildFromDB(key) // 你的重逻辑

return ids, err

})

if err != nil {

return nil, err

}

return v.([]string), nil

// shared == true 表示当前调用复用了别人正在算的那次结果

}
```
- `Do(key, fn)`：相同 `key` 并发进来，只有一个会跑 `fn`，其它阻塞到 `fn` 结束，拿到同一个 `(v, err)`。
- `DoChan`：需要带 `context` 取消时用。
- `Forget(key)`：极少数场景要丢掉进行中的合并（慎用）。

---

##### 3. 套在你们「态势索引构建」上怎么理解
- Redis key 已经是 `situation:index:{userId}:{sessionId}` 这类。
- 单机内：同一瞬间多个请求（同 key）都 miss，没有 singleflight 时，可能多个 goroutine 都进到「构建」逻辑（你们用锁挡住了部分，但锁之前可能仍有重复）。
- 在 `getOrBuildIndexWithLock` 里、或 `buildAndSaveIndex` 外包一层：
```Go
singleflightKey = indexKey // 或 indexKey + ":" + keywordHash

sf.Do(singleflightKey, func() { ... 构建并写 Redis ... })
```

这样同一进程对同一 indexKey 只会跑一个构建函数，其它请求共享结果。
- 多实例：实例 A 和实例 B 各有一套 `singleflight.Group`，互不共享，所以仍需要 Redis Setnx 锁（或只允许一个实例构建）。

推荐组合：  
Redis 锁（跨实例） + singleflight（单实例内），双检缓存仍然保留。

---

##### 4. 和「带缓冲 channel 的 worker 池」区别
- singleflight： 按 key 合并，不同 key 互不影响，可以并行多个 key。
- 全局 worker 池： 限制总共多少个任务在跑，不按 key 合并。

可以同时用：先 singleflight 合并同 key，再用 semaphore/channel 限制全局 DB 压力。

### 9) 为什么还要分布式锁？
答： 防止缓存 miss 时多个请求同时重建索引（缓存击穿），造成 DB 突刺。锁控制只允许一个构建者，其他请求短轮询等缓存，显著降低并发重建成本。

### 10) 如何避免“误删别人的锁”？
答： 锁值使用唯一 token（如 UUID），释放锁前先比对当前 value 与 token，一致才删除，避免锁过期后新持有者被误删。

### 11) 为什么要“超时隔离 Context”？
答： 索引构建是重任务，若直接绑定请求 context，客户端断开可能导致 `context canceled`，构建中断。使用独立超时 context 可以让关键构建在可控时限内完成，提高稳定性。

### 12) 并行聚合时某一路失败怎么办？
答： 按场景分级：
- 关键路径失败则返回明确错误并记录日志；
- 非关键字段可降级为空值，保证主流程可用。  
    同时打点监控错误率，便于后续修复。

### 13) 你怎么证明优化有效，不是“体感变快”？
答： 用统一口径对比优化前后 RT（平均/P95）、慢 SQL 数量、缓存命中率和超时率，保持相同环境与同级数据规模。结论是核心接口平均 RT 由 7s+ 到 0.8s，态势首查与滑动也有稳定下降。

### 14) SQL 优化你通常怎么做？
答： 先看慢日志和 `EXPLAIN`，定位全表扫描、回表、索引失效；再做索引重建与查询改写（条件下推、字段裁剪、减少不必要排序），最后回归验证执行计划和时延。

### 15) 为什么不直接每页实时查库排序？
答： 每页实时排序会重复计算全量用户“最近活跃时间”，成本高、抖动大。先构建有序 ID 索引再分页取明细，能把昂贵计算前置并复用，整体更稳。

### 16) 强制刷新是怎么实现的？
答： 客户端传 `lastSessionId`，服务端淘汰旧会话缓存并生成新 `sessionId`，本次请求走新索引构建或命中新缓存，避免“删同 key 又写回”的竞态问题。

### 17) 你在安全上做了什么？
答： 详情组装阶段对敏感信息做脱敏（如身份证中间位隐藏），并在查询链路严格按用户权限返回范围，保证“可用”与“合规”同时满足。

### 18) 授权链路的难点是什么？
答： 难点在状态一致性和规则复杂度：申请、变更、撤销、协议确认、回传等环节跨角色协同。重点是保证状态流转可追踪、接口语义一致、异常场景可回溯。

### 19) 你在团队协作里的角色是什么？
答： 在性能优化和态势查询链路中承担主导角色，负责方案拆解、接口改造与效果验证；在授权和报告模块中负责核心接口实现并协同联调上线。

### 9. Redis 会话索引为什么要带 keyword hash？
追问点： 只用 sessionId 不行吗？  
高分要点：
1. Key 里通常不带 keyword  
    会话 key 一般是「谁在查 + 哪一次会话」，方便滑动分页一直复用同一份排序结果。
2. 但 keyword 会变  
    用户改搜索词时，如果客户端仍带同一个 `sessionId`，Redis 里还是旧 key，里面却是按上一个关键词筛出来的候选用户 + 排好序的 ID。  
    若不校验，就会把旧关键词下的顺序/集合当成新关键词的结果，属于逻辑错误。
3. keyword hash 的作用  
    在 value 里存 `keyword_hash`，读缓存时对比当前关键词的 hash：
    - 一致：说明这份排序索引就是当前关键词下的，可以复用。
    - 不一致：视为 miss，重新按新关键词筛候选、再聚合排序、再写回。

#### 为什么 `keyword_hash` 也算「双检」的一部分？
因为 value 里带了 `keyword_hash`。  
第一道检查不仅是「有 key」，还要 hash 一致 才算命中。

否则会出现：同一会话换了关键词，Redis key 没变，但旧排序列表对应旧关键词——  
不做 hash 校验就会错误复用。  
所以「检」不仅是「有没有缓存」，更是「这份缓存是不是当前查询语义下的缓存」。

### 10. 分布式锁怎么避免死锁或误删？
追问点： 锁过期后怎么办？  
高分要点：
- 怎么避免死锁：
	- 给锁加过期时间（TTL）  
    用 `SET key value NX EX ttl` 或你们这种 `SetnxEx`，保证锁最多持有 ttl 秒，进程崩溃也会自动过期，别人还能继续抢锁。
	- 控制临界区必须在 ttl 内完成  
	    若业务执行可能超过 ttl，要评估加长 ttl、拆任务、或换更合适的锁方案，否则会出现「锁过期但业务还在跑」的并发问题（见下面误删相关）。
	- 不要嵌套多把 Redis 锁（除非有严格顺序和超时设计）  
	    嵌套锁最容易在异常路径上漏释放。
- 怎么避免误删：典型误删场景：A 拿到锁 → A 执行慢 → 锁 TTL 过期 → B 抢到锁 → A 执行完去 `DEL`，把 B 的锁删了。
	- 常见正确做法：
		- 锁 value 用全局唯一 token（如 UUID），谁抢到锁就写谁的 token。
		- 释放前先比对：只有 `GET(key) == 自己的 token` 才 `DEL`。  
		    你们代码里就是这种思路：释放前读当前值，相等再删。
		- 更严谨：用 Lua 脚本保证「GET + DEL」原子性  
		    避免「GET 还是自己的，DEL 前锁刚好过期被 B 抢走，结果还是删了 B 的锁」这种竞态。Lua 一次执行：`if get(key)==token then del(key) end`。

#### 当没抢到锁时
- 双检缓存：进函数先再读一次缓存。
- 短轮询：最多等约 2 秒（`situationLockWaitSec`），每 50ms（`situationLockPollMs`）看缓存是否已被构建者写好。
- 超时后再 `Setnx` 一次；仍抢不到则 兜底直接构建（允许极端情况下少量并发重建，避免一直空结果）。

#### 双检缓存
当没有先查缓存时，会出现大量请求同时发现缓存里没有排序好的 ID 列表（缓存 miss）。
- 每个请求都去抢分布式锁、甚至都去查库做「全量候选用户 × 多源时间」的聚合排序。
- 即使用 `Setnx` 只让一个线程真正构建，抢锁、轮询、Redis 往返也会非常多。
- 更糟的是：若实现不好，可能出现重复构建或短时间 DB 被打爆（缓存击穿）。
所以要在「重活」之前，尽可能用一次极便宜的读把大部分流量挡掉。

在 `getOrBuildIndexWithLock` 最开头：
- 读 Redis：`getIndexFromCache(indexKey, keywordHash)`
- 若命中且 `ids` 非空：直接 return，后面 锁、`Setnx`、多源并行查库、排序 全都不走。
这就是双检里的第一检（在竞争锁之前）：
- 成本：一次 Redis GET + 反序列化 + 比对 `keyword_hash`
- 收益：索引一旦存在，所有后续请求（翻页、同会话滑动）几乎都是这条快路径，锁基本不参与

第二道检查：
经典单机「双检加锁」是：
1. 查缓存 → miss
2. 加本地互斥锁
3. 再查一次缓存（可能别的 goroutine 已经写好了）
4. 还 miss 才真正加载

你们是分布式：锁在 Redis 上，等待者不是阻塞在 `mutex` 上，而是：
- 没抢到 `Setnx` → 进入 for 循环短轮询（`Sleep 50ms`，最多约 2 秒）
- 每一轮都调用 `getIndexFromCache`
- 一旦构建者写完缓存 → 等待者下一轮就读到 → 直接 return

所以对「没抢到锁的请求」来说：轮询里每一次读缓存，就是分布式意义上的「第二道（乃至第 N 道）检查」——  「也许在我等锁/等构建的这段时间里，别人已经写好缓存了，我就不该再自己建。（用户重复刷新，导致同一时间多个请求同时打入）」

当抢到锁后：
- 再尝试读一次缓存，防止：极短时间窗口内别的实例/别的逻辑已写入。
- 用一次 GET 换「极端并发下少一次重算」

### 11. 为什么要“短轮询等待”？
追问点： 为啥不是直接失败或直接重建？  
高分要点：
- 直接重建会放大并发压力。
- 等待构建者写缓存可减少重复计算。
- 超时后再抢锁/兜底，平衡时延与资源。

### 12. 独立超时 Context 的意义是什么？
追问点： 请求断开还继续算，不浪费吗？  
- 单次请求看可能白算，但索引是共享资产，写完缓存能服务后续滑动和等待中的请求；如果跟客户端生命周期强绑定，容易频繁中断、重复构建，DB 压力反而更大。我们用独立超时把成本封顶，在可复用场景下这是性能和稳定性的权衡。
高分要点：
- 构建索引是可复用结果，服务后续请求。
- 若随请求取消，容易频繁失败重算。
- 用独立超时限制成本，避免无限执行。

#### 请求断了，索引构建仍可能值得做完
态势索引构建是 重活（多源 DB 查询 + 排序 + 写 Redis），结果写入后 同一 `session` 的后续滑动请求 都能用。
若全程用 `r.Context()`（请求 context）：
- 客户端断开或超时 → `ctx` 取消 → 下游 `QueryContext` 可能中断 → 构建失败。
- 下一个用户再滑，又要重新构建，体验差、DB 压力也大。
用 独立超时 的意思是：
- 取消由服务端控制：最多跑 `situationBuildCtxSec`（你们 45 秒），而不是被浏览器随便掐断。
- 只要在这 45 秒内算完并写入 Redis，后续请求仍受益，不浪费已付出的计算。
代价（面试可主动说）：用户已经走了，服务端仍可能跑满超时时间，占一点 CPU/连接；所以必须 严格超时，不能无限跑。

#### 并行查询时避免误伤
`buildAndSaveIndex` 里用 `mr.Finish` 并行查睡眠 / 体征 / 问卷时间。
若多路共享一个会在某处 `cancel()` 的 ctx，容易出现：
- 一路先返回、`defer cancel()` 触发；
- 其它路还在跑，ctx 已取消 → `context canceled`，明明能查完的数据被掐断。
用 `WithoutCancel` 包一层，是让 并行任务用的 ctx 不会因为父级 `cancel` 的连锁反应被提前取消（在你们注释里写得很直白）。

### 14. 分页为什么不用数据库 offset？
追问点： offset 不是更简单吗？

高分要点：
- 大 offset 性能抖动明显。
- 先有序 ID 列表再分页，避免重复全量排序。
- 更适合你的多源聚合排序场景。

### 19. 如果线上突增流量，你的方案抗得住吗？
追问点： 最怕什么风险？  
- 常态和中等突增一般能扛，主要靠缓存复用和锁控重建把 DB 突刺压住；极端突增仍可能受连接池、下游 DB、Redis 瓶颈限制，需要限流、扩容和降级预案配合。
高分要点：
- 先靠缓存复用和锁控重建抗峰值。
- 风险在缓存集中失效和下游抖动。
- 预案：错峰过期、降级、限流、扩容。

顶不住的典型点：
- 大量新会话 / 新关键词：miss 多，构建次数上去，DB 与 Redis 写增多。
- 候选用户集很大：单次构建的 `userIDs` 多，并行查询数据量与连接占用都涨。
- 多实例部署：锁能防单 key 重复构建，但不同 user/session 仍会横向叠加压 DB。
- Redis 抖动：索引读不到或慢，会放大 miss 与锁竞争。
- DB 连接池打满：并行 `mr` + 多请求仍可能把池子占光。

预案：
1. 限流：网关/接口级 QPS、按用户或 IP，保护核心 DB。
2. 扩容：应用水平扩；DB 只读扩展或读写分离（视架构）。
3. 降级：构建超时或负载高时返回简化列表、提示稍后重试、或只返回缓存旧数据（需产品接受）。
4. 预热 / 错峰：大活动前预热热点；缓存 TTL 避免同一秒大面积同时过期。
5. 观测：RT、错误率、DB QPS、连接池、Redis 延迟、锁等待、索引构建次数。

突增时我们主要靠会话索引和双检，让大部分流量走缓存；miss 时用分布式锁把重建收敛到少量实例执行，避免击穿。风险在于新会话、大候选集、多实例叠加仍会增加 DB 压力，所以线上要配限流、容量规划和监控；极端情况需要降级或扩容。方案目标是把尖刺削平，不是单机无限扛。

### 1) 你说 7s 到 0.8s，这个数据真实吗？

“这个数据是同环境、同业务入口下的对比口径，优化前后都按同一统计方式看平均耗时。除了平均值，我还关注了高频场景的首查与滑动刷新，分别从 14s+/4s+ 降到 3s/600ms，不是单点偶然值。”

- 环境：预发还是生产、时间段、是否高峰期。

示例：

更多的是这个是属于历史遗留问题，在bug测试单中，测试人员对我的这个接口优化进行的评估

### 6) 分布式锁很容易出事故，你怎么保证安全？
高压追问： 误删锁、锁过期竞争怎么处理？  
反击回答：  
“我们采用唯一 token 持锁，释放前做 value 比对，避免误删；加过期时间防死锁；未抢到锁短轮询等缓存，超时再兜底重建，确保不会长期空返回。”

####  防死锁（锁永远能释放）
- 锁必须带 TTL：你们用 `SetnxEx(..., 60s)`，进程宕机没删 key，最多 60 秒也会自动过期，不会永久占死。
- 临界区时长 < TTL：构建有 45s 级超时，避免业务跑太久导致锁过期后仍写数据（可顺带提你们会注意这点）。
- 避免多锁嵌套：减少互相等待导致的死锁概率。

---

#### 2. 防误删（不删别人的锁）
- 锁 value 用唯一 token（你们用 UUID），释放前 `GET` 比对，只有仍是自己的 value 才 `DEL`。
- 注释里写明的场景：锁过期后可能被别的请求抢走，若盲目 `DEL` 会删掉别人的锁。
- 更严做法（可加分）：用 Lua 脚本 做「if get==token then del」原子释放，避免 GET 和 DEL 之间锁被换人的竞态。

---

#### 3. 防并发事故（击穿、风暴）
- 双检缓存：进函数先读缓存，减少无意义抢锁。
- 没抢到锁：短轮询等构建者写缓存，而不是立刻再全量打 DB。
- 超时后再抢锁 / 兜底：避免一直空结果；同时承认极端情况允许少量并发重建是可用性权衡（和你们代码一致）。

### 7) 你这个 `Context` 设计会不会造成资源浪费？
高压追问： 客户端断了还继续跑，不浪费吗？  
反击回答：  
“索引构建结果可复用，后续请求会直接受益，所以不是无效计算。我们用独立超时 `Context` 严格限制执行时长，平衡资源成本和可用性，避免频繁中断导致反复重算。”

#### 1. 先承认：有可能「白算」
用户断线或网关超时后，服务端若仍按 独立 `Context` 把索引构建跑完，从这一次 HTTP 请求看，确实可能多占一点 CPU、DB 连接、时间。

---

#### 2. 为什么说 Often 不浪费：结果可复用
你们构建的是 Redis 里的会话级排序索引，写成功后：
- 同一会话后续滑动/翻页直接用缓存；
- 其它正在等的请求也可能读到这份结果。

若绑死请求 `Context`，客户端一断就取消，容易 反复 miss、反复全量构建，DB 和整体延迟反而可能更差。  
所以是：用「有上限」的额外计算，换「少重复重活」。

---

#### 3. 浪费有没有边界：有
- `WithTimeout(Background, 45s)`：最多跑这么久，不是无限跑。
- 和 60s 锁 TTL、双检缓存 一起，把「多算」关在可控范围里。

---

#### 4. 什么时候算真浪费、要不要改
- 若监控发现：大量构建完成后没有任何后续读（会话立刻废弃），可以再考虑 更短超时 或 与请求 ctx 组合（例如仅构建阶段用独立 ctx，但接受更复杂语义）。
- 这是用指标迭代，不是一句话否定当前设计。

### 8) 你怎么证明 SQL 真被优化了，不是心理作用？
高压追问： 给我证据。  
反击回答：  
“证据来自执行计划和耗时对比：优化前后 `EXPLAIN` 中扫描行数、索引命中、回表情况都有改善；再叠加接口 RT 对比和慢 SQL 数量下降，能形成闭环。”

#### 1. 执行计划（`EXPLAIN` / `EXPLAIN ANALYZE`）
- 优化前后各跑一次，对比：
    - type：是否从 `ALL` 变成 `range`/`ref` 等；
    - rows：扫描行数是否明显下降；
    - Extra：是否还 `Using filesort` / `Using temporary`；
    - 有 MySQL 8.0.18+ 可用 `EXPLAIN ANALYZE` 看实际耗时。
一句话：「计划从全表扫几万行变成索引命中几千行，这不是心理作用。」

---

#### 2. 慢查询日志 / 平台监控
- 优化前：该 SQL 是否进慢日志、平均耗时多少。
- 优化后：条数减少或平均耗时下降。
- 若有 APM / DB 监控（如 `performance_schema`、云厂商 SQL 洞察），用同一条 SQL 模板看 P95 变化。

---

#### 3. 接口分层耗时（可选但很强）
- 在代码或 tracing 里打 DB 段耗时（整条 handler 里只算 DAO 调用）。
- 若 DB 段从 X ms 降到 Y ms，而总 RT 也降，说明不是「全靠缓存盖住」。

---

#### 4. 控制变量
- 同库、同数据量、同级别参数（或说明预发/生产差异）；
- 对比时注明：是否同一查询条件，避免「这次条件简单所以快」。
